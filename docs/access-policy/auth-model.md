# 인증 모델

> 본 문서는 BuildMap 7단계 문서다. 7단계는 실제 RLS SQL 작성 단계가 아니라, SQL 작성 전 권한/공개/접근 정책을 자연어로 고정하는 단계다. `docs/decisions/phase6-5-db-schema-corrections.md`를 최우선 기준으로 삼는다.

## 1. 문서 목적

이 문서는 BuildMap의 인증 원천과 제품 프로필 구조를 정리한다. 핵심은 인증과 제품 역할 데이터를 분리하는 것이다.

권장 개념 구조는 다음이다.

```text
auth.users
→ app user profile 또는 user_profiles 후보
→ builder_profiles
→ scout_profiles 후보
```

실제 테이블명, SQL, Supabase Auth 연동 방식은 아직 확정하지 않는다.

## 2. auth.users를 인증 원천으로 보는 원칙

`auth.users`는 인증의 원천이다. 이메일, 인증 ID, 로그인 제공자, 인증 상태 등은 인증 계층의 책임이다.

BuildMap의 제품 데이터는 인증 테이블에 직접 섞지 않는다. Builder 역할, 공개 표시명, 프로젝트 소유, 피드백 작성 이력 등은 제품 데이터에 가깝다.

### 포함할 수 있는 정보

- 인증 식별자
- 이메일 또는 로그인 제공자 정보
- 인증 생성 시점
- 인증 상태

### 공개 가능 정보

없음. 인증 식별자와 이메일은 공개 프로젝트 페이지에 노출하지 않는다.

### 누가 생성하는가

인증 시스템이 생성한다.

### 누가 수정할 수 있는가

사용자의 인증 설정 또는 인증 시스템 정책에 따른다. BuildMap 제품 화면에서 직접 다루는 범위는 후순위다.

### 관련 화면

- 로그인
- 회원가입
- 계정 설정 후보

### 1차 포함 여부

1차 인증 원천으로 고려한다.

## 3. app user profile 또는 user_profiles 후보

앱 사용자 프로필 후보는 인증 원천과 제품 역할 사이의 공통 사용자 정보다.

### 역할

- 앱에서 보여줄 기본 표시 정보 관리
- Builder 또는 Scout 역할로 확장되기 전 공통 사용자 상태 관리
- 약관 동의, 계정 상태, 표시명 등 제품 데이터 관리 후보

### 포함할 수 있는 정보

- 표시명
- 프로필 이미지 후보
- 기본 소개 후보
- 계정 상태 후보
- 약관 동의 후보
- 생성 시점과 수정 시점 후보

### 공개 가능 정보

- 사용자가 공개로 설정한 표시명
- 공개로 설정한 프로필 이미지 후보

### 비공개 정보

- 이메일
- 인증 ID
- 내부 계정 상태
- 약관 동의 상세

### 누가 생성하는가

로그인 사용자 생성 시 자동 생성 후보 또는 최초 프로필 설정 시 생성 후보.

### 누가 수정할 수 있는가

해당 로그인 사용자.

### 관련 화면

- 계정 설정 후보
- Builder 프로필 설정
- Feedback 작성 시 표시 정보

### 1차 포함 여부

1차에서 분리 후보로 둔다. 실제 테이블명은 migration 직전 결정한다.

## 4. builder_profiles

Builder Profile은 프로젝트를 성장시키는 사람의 역할 데이터다. Builder는 프로젝트를 만든 사람이 아니라 프로젝트를 성장시키는 사람이다.

### 역할

- 프로젝트 소유와 작성 권한의 기준
- 공개 프로젝트 페이지에서 Builder 정보를 보여주는 원천 후보
- 프로젝트 판단 흐름의 주체 표시

### 포함할 수 있는 정보

- 공개 이름
- Builder 역할 태그
- 관심 분야
- 소개
- 소유 프로젝트 목록과의 관계
- 공개 여부 설정 후보

### 공개 가능 정보

- 공개 이름
- Builder가 공개하기로 한 소개
- 공개 역할 태그
- 공개 프로젝트 목록

### 비공개 정보

- 인증 ID
- 이메일
- 내부 계정 상태
- 비공개 프로젝트 목록

### 누가 생성하는가

로그인 사용자가 Builder 역할을 선택할 때 생성한다.

### 누가 수정할 수 있는가

해당 Builder 본인.

### 관련 화면

- Builder 가입 및 역할 선택
- Builder 대시보드
- 공개 프로젝트 페이지

### 1차 포함 여부

1차 필수다.

## 5. scout_profiles 후보

Scout Profile은 초기 핵심 필수 데이터가 아니다. Scout는 초기에는 채용 담당자가 아니라 프로젝트 발견자, 피드백 제공자, 테스터, 협업자, 멘토, 팀원 탐색자다.

### 역할

- 프로젝트 탐색 목적과 관심 분야 표현 후보
- 저장/팔로우, 테스터 신청, 협업 제안이 확장될 때 활용 후보

### 포함할 수 있는 정보

- 관심 분야
- 탐색 목적
- 저장한 프로젝트 후보
- 피드백 작성 이력 후보
- 테스터 신청 이력 후보

### 공개 가능 정보

1차에서는 최소화한다. Scout 전용 공개 프로필은 후순위다.

### 비공개 정보

- 개인 연락처
- 내부 탐색 이력
- 채용/투자 목적 정보 후보

### 누가 생성하는가

Scout 성격의 사용자가 별도 탐색 프로필을 만들 때 생성 후보.

### 누가 수정할 수 있는가

해당 사용자.

### 관련 화면

- Scout 프로젝트 탐색 화면
- 피드백 작성 화면
- 테스터 신청 후보 화면

### 1차 포함 여부

1차 선택 데이터다. 없어도 로그인 사용자 기반 피드백 작성과 프로젝트 탐색은 가능해야 한다.

## 6. 로그인 사용자와 비로그인 방문자의 차이

| 구분 | 읽기 | 쓰기 |
|---|---|---|
| 비로그인 방문자 | 전체 공개 프로젝트의 공개 정보, 링크 공개 접근 조건을 만족한 공개 정보 | 1차에서는 없음 |
| 로그인 사용자 | 공개 정보 + 본인이 접근 가능한 내부 정보 | Feedback 작성, Builder 프로젝트 생성 후보 |
| Builder | 자신의 프로젝트 내부 기록 접근 | Project, Rough Note, AI Draft, Change Card 작성/수정/승인 후보 |

비로그인 쓰기 권한은 1차에서 제외한다.

## 7. 1차에서 필요한 인증 전제

- 로그인 사용자 식별이 가능해야 한다.
- Builder Profile과 인증 사용자를 연결할 수 있어야 한다.
- Feedback 작성자는 로그인 사용자여야 한다.
- Project Owner를 식별할 수 있어야 한다.
- 공개 페이지 접근은 로그인 여부와 무관하게 가능할 수 있으나, 노출 데이터는 공개 정책을 따른다.

## 8. 1차에서 보류할 인증 기능

- 조직 계정
- 팀 권한
- 공동 편집 권한
- 관리자 상세 권한
- 채용 담당자 인증
- 투자자 인증
- 외부 OAuth 기반 프로젝트 연동
- 비로그인 피드백 작성
