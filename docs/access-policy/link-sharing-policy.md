# 링크 공개 정책

> 본 문서는 BuildMap 7단계 문서다. 7단계는 실제 RLS SQL 작성 단계가 아니라, SQL 작성 전 권한/공개/접근 정책을 자연어로 고정하는 단계다. `docs/decisions/phase6-5-db-schema-corrections.md`를 최우선 기준으로 삼는다.

## 1. 문서 목적

이 문서는 링크 공개의 목적, 전체 공개와의 차이, public_slug와 share_token의 역할을 정리한다.

## 2. 링크 공개의 목적

링크 공개는 프로젝트를 전체 탐색에 공개하지 않더라도 특정 사람에게 공개 가능한 판단 흐름을 공유하기 위한 방식이다.

예:

- 멘토에게 프로젝트 판단 흐름 공유
- 협업 후보에게 제한 공개
- 테스터에게 공개 페이지 전달
- 학원 프로젝트 리뷰용 링크 공유

## 3. 전체 공개와 링크 공개의 차이

| 구분 | 링크 공개 | 전체 공개 |
|---|---|---|
| 접근 방식 | 유효한 링크 접근 식별자 필요 | 공개 URL 또는 탐색 접근 가능 |
| 탐색 노출 | 기본적으로 제한 | 공개 탐색 노출 가능 |
| 보안 성격 | 추측 어려운 식별자 필요 | 보안 토큰으로 간주하지 않음 |
| 주요 후보 | share_token | public_slug |

## 4. public_slug와 share_token의 목적 차이

### public_slug

- 전체 공개용 읽기 쉬운 경로 후보다.
- 보안 토큰이 아니다.
- 추측 가능할 수 있으므로 링크 공개 접근 권한 판단에 사용하지 않는다.

### share_token

- 링크 공개 접근을 위한 추측 어려운 식별자 후보다.
- 링크 공개 상태에서만 유효하게 해석한다.
- 노출되면 접근 가능하므로 재발급/폐기 정책이 필요하다.

## 5. share_token 생성 후보

- Project Owner가 링크 공개를 선택할 때 생성 후보.
- 링크 공개를 다시 켤 때 재사용할지 재발급할지는 추가 검토 필요.
- 토큰 값 자체는 공개 페이지 내용과 분리한다.

## 6. share_token 재발급 후보

- Project Owner가 공유 링크를 폐기하고 새 링크를 만들고 싶을 때 재발급 가능해야 한다.
- 재발급 시 기존 share_token은 무효화될 수 있다.

## 7. share_token 폐기 후보

- Project가 비공개로 전환되면 기존 share_token 접근은 차단되어야 한다.
- share_token 폐기 후 기존 링크는 접근 불가가 되어야 한다.

## 8. Project 전체 공개 전환 시 share_token의 역할

- Project가 전체 공개로 전환되면 public_slug 또는 공개 경로가 주요 접근 방식이 될 수 있다.
- share_token은 계속 유지할 수도 있지만, 접근 권한 판단에서 필수 조건은 아닐 수 있다.
- 정확한 정책은 7단계 이후 RLS 설계 직전에 추가 검토한다.

## 9. 링크 공개에서 노출 가능한 정보

링크 공개에서도 노출 가능한 정보는 공개 페이지 정책과 동일하게 제한한다.

- Project 공개 정보
- 공개 가능한 Problem/Hypothesis
- 승인됨 + 공개됨 + 민감 정보 없음 Change Card
- 공개 Feedback Request
- 공개 선택된 Feedback
- Builder 공개 정보

## 10. 링크 공개에서 노출하지 않는 정보

- Rough Note
- AI Draft
- 내부 전용 Change Card
- 공개 가능이지만 공개됨이 아닌 Change Card
- 민감 정보 포함 Change Card
- 내부 검토 Feedback
- 이메일과 인증 ID

## 11. 반드시 지켜야 할 원칙

- public_slug는 보안 토큰이 아니다.
- share_token은 링크 공개 접근을 위한 추측 어려운 식별자 후보다.
- 링크 공개는 전체 공개보다 좁은 공개 범위다.
- 비공개 전환 시 링크 접근은 차단되어야 한다.
- 추측 가능한 URL을 보안 수단으로 사용하지 않는다.
