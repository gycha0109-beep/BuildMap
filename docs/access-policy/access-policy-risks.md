# 권한/공개/RLS 정책 위험과 대응

> 본 문서는 BuildMap 7단계 문서다. 7단계는 실제 RLS SQL 작성 단계가 아니라, SQL 작성 전 권한/공개/접근 정책을 자연어로 고정하는 단계다. `docs/decisions/phase6-5-db-schema-corrections.md`를 최우선 기준으로 삼는다.

## 1. 문서 목적

이 문서는 7단계 권한/공개/RLS 정책 설계에서 발생할 수 있는 위험과 1차 대응 원칙을 정리한다.

## 1. 공개 가능 상태를 공개됨으로 오해해 외부 노출하는 위험

### 위험 설명

공개 가능 상태를 공개됨으로 오해해 외부 노출하는 위험

### 왜 문제가 되는가

공개 가능은 외부 공개가 아니라 공개 후보 상태다.

### 예방 원칙

공개 상태 이름과 정책을 분리하고 공개 Timeline에는 공개됨만 허용한다.

### 1차 정책 대응

공개 Timeline 조건에 공개됨을 명시한다.

### 후순위 검토 사항

공개 전 확인 UI와 관리자 검토 후보를 검토한다.

## 2. 민감 정보 포함 Change Card가 공개 Timeline에 노출되는 위험

### 위험 설명

민감 정보 포함 Change Card가 공개 Timeline에 노출되는 위험

### 왜 문제가 되는가

개인정보나 내부 전략이 외부에 노출될 수 있다.

### 예방 원칙

민감도 플래그를 공개 상태와 분리한다.

### 1차 정책 대응

공개 Timeline 조건에 민감도 일반을 포함한다.

### 후순위 검토 사항

민감 정보 자동 감지와 공개 전 경고를 검토한다.

## 3. Project가 비공개인데 공개 Change Card가 노출되는 위험

### 위험 설명

Project가 비공개인데 공개 Change Card가 노출되는 위험

### 왜 문제가 되는가

Project 공개 정책보다 Change Card 공개 정책이 우선되면 내부 프로젝트가 노출된다.

### 예방 원칙

Project 공개 상태와 Change Card 공개 상태를 함께 본다.

### 1차 정책 대응

Project 비공개이면 모든 공개 표현을 차단한다.

### 후순위 검토 사항

파생 뷰 테스트 케이스를 추가 검토한다.

## 4. share_token이 폐기되지 않아 링크 공개가 계속 유지되는 위험

### 위험 설명

share_token이 폐기되지 않아 링크 공개가 계속 유지되는 위험

### 왜 문제가 되는가

비공개 전환 후에도 외부 접근이 남을 수 있다.

### 예방 원칙

비공개 전환 시 링크 접근 차단 원칙을 둔다.

### 1차 정책 대응

share_token 재발급/폐기 정책을 문서화한다.

### 후순위 검토 사항

토큰 회전과 접근 로그를 검토한다.

## 5. public_slug를 보안 토큰처럼 사용하는 위험

### 위험 설명

public_slug를 보안 토큰처럼 사용하는 위험

### 왜 문제가 되는가

읽기 쉬운 URL은 추측 가능할 수 있다.

### 예방 원칙

public_slug는 보안 토큰이 아니라고 명시한다.

### 1차 정책 대응

링크 공개에는 share_token 성격 식별자를 사용한다.

### 후순위 검토 사항

slug 변경/점유 정책을 검토한다.

## 6. Rough Note 또는 AI Draft가 외부에 노출되는 위험

### 위험 설명

Rough Note 또는 AI Draft가 외부에 노출되는 위험

### 왜 문제가 되는가

거친 원문과 AI 초안은 미검토 내부 기록이다.

### 예방 원칙

기본 비공개 원칙을 둔다.

### 1차 정책 대응

공개 페이지와 Timeline 파생 대상에서 제외한다.

### 후순위 검토 사항

내부 기록 접근 테스트를 검토한다.

## 7. Feedback 내용이 일반 댓글처럼 공개되는 위험

### 위험 설명

Feedback 내용이 일반 댓글처럼 공개되는 위험

### 왜 문제가 되는가

BuildMap의 Feedback은 판단 근거이지 공개 댓글창이 아니다.

### 예방 원칙

Feedback Request는 공개 가능하지만 Feedback 내용은 기본 내부 검토로 둔다.

### 1차 정책 대응

Builder가 공개 선택한 Feedback만 노출한다.

### 후순위 검토 사항

작성자 공개 동의 정책을 검토한다.

## 8. Feedback 작성자의 개인정보가 과도하게 노출되는 위험

### 위험 설명

Feedback 작성자의 개인정보가 과도하게 노출되는 위험

### 왜 문제가 되는가

이메일, 인증 ID 등이 공개될 수 있다.

### 예방 원칙

공개 표시 정보를 제한한다.

### 1차 정책 대응

공개 Feedback에는 표시명 후보만 노출한다.

### 후순위 검토 사항

익명/닉네임 정책을 검토한다.

## 9. Problem/Hypothesis 이력이 내부 기록과 공개 기록에서 불일치하는 위험

### 위험 설명

Problem/Hypothesis 이력이 내부 기록과 공개 기록에서 불일치하는 위험

### 왜 문제가 되는가

이력 원천이 분산되면 판단 흐름이 흔들린다.

### 예방 원칙

이력 원천은 Change Card로 둔다.

### 1차 정책 대응

현재값은 Problem/Hypothesis, 과거 흐름은 공개 조건을 통과한 Change Card에서 보여준다.

### 후순위 검토 사항

별도 이력 테이블 도입 시 원천 규칙을 재검토한다.

## 10. Change Card 작성자와 승인자 권한이 섞이는 위험

### 위험 설명

Change Card 작성자와 승인자 권한이 섞이는 위험

### 왜 문제가 되는가

작성자가 곧바로 공식 기록으로 확정할 수 있는지 모호해진다.

### 예방 원칙

작성자와 승인자 개념을 분리한다.

### 1차 정책 대응

1차에서는 Project Owner 중심 승인으로 둔다.

### 후순위 검토 사항

팀 승인 워크플로우를 후순위로 검토한다.

## 11. Scout/채용 기능이 권한 모델을 과도하게 복잡하게 만드는 위험

### 위험 설명

Scout/채용 기능이 권한 모델을 과도하게 복잡하게 만드는 위험

### 왜 문제가 되는가

초기 제품이 채용 플랫폼처럼 변질될 수 있다.

### 예방 원칙

Scout는 피드백/테스터/협업 중심으로 제한한다.

### 1차 정책 대응

채용/헤드헌팅 권한을 보류한다.

### 후순위 검토 사항

Scout Profile 확장 조건을 별도 검토한다.

## 12. 파생 데이터 접근 정책이 원천 데이터 정책과 어긋나는 위험

### 위험 설명

파생 데이터 접근 정책이 원천 데이터 정책과 어긋나는 위험

### 왜 문제가 되는가

Public Page나 Timeline이 원천보다 더 많이 보여줄 수 있다.

### 예방 원칙

파생 표현은 원천 데이터 공개 조건을 초과하지 않는다.

### 1차 정책 대응

파생 뷰 조건을 원천 정책과 동일하게 문서화한다.

### 후순위 검토 사항

캐시/스냅샷 도입 시 무효화 정책을 검토한다.

## 13. 비로그인 사용자가 쓰기 권한을 갖는 위험

### 위험 설명

비로그인 사용자가 쓰기 권한을 갖는 위험

### 왜 문제가 되는가

악성 피드백과 스팸 위험이 커진다.

### 예방 원칙

비로그인 쓰기 권한은 1차 제외한다.

### 1차 정책 대응

Feedback 생성은 로그인 사용자 기반으로 제한한다.

### 후순위 검토 사항

제한적 비로그인 피드백은 후순위 검토한다.

## 14. RLS SQL 작성 시 정책 문서와 다르게 구현되는 위험

### 위험 설명

RLS SQL 작성 시 정책 문서와 다르게 구현되는 위험

### 왜 문제가 되는가

문서와 구현이 불일치하면 보안 사고가 발생할 수 있다.

### 예방 원칙

정책 ID와 조건을 SQL 작성 전에 검토한다.

### 1차 정책 대응

RLS readiness checklist를 통과한 뒤 SQL을 작성한다.

### 후순위 검토 사항

테스트 케이스와 정책 매트릭스 검증을 검토한다.
