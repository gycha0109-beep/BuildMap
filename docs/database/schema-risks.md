# DB 스키마 초안 위험과 대응

> 이 문서는 DB 스키마 초안 문서다. 실제 SQL, Supabase migration, RLS 정책, API, 프론트엔드 구현은 작성하지 않는다. 테이블명과 필드명은 후보이며 최종 확정이 아니다.

## 1. 위험 관리 원칙

6단계의 스키마 초안은 BuildMap의 핵심 철학을 훼손하지 않아야 한다. 특히 Change Card 중심 구조, 내부 기록 후 공개 전환, 피드백의 판단 연결 구조를 지켜야 한다.

## 2. 위험 목록

### 1. Change Card 필수 필드가 많아져 입력 부담이 커지는 위험

- 위험 설명: Change Card 필수 필드가 많아져 입력 부담이 커지는 위험
- 왜 문제가 되는가: Builder가 기록을 포기하면 핵심 흐름이 무너진다.
- 6단계 예방 원칙: 필수 후보를 최소화하고 AI 초안/선택 입력을 둔다.
- 1차 구현 시 대응: project, builder, 유형, 제목, 요약, 상태 정도를 중심으로 시작한다.
- 후순위 검토 사항: 실제 사용 로그로 필수 필드를 재조정한다.

### 2. Change Card와 Decision Timeline을 둘 다 원천 테이블로 만드는 위험

- 위험 설명: Change Card와 Decision Timeline을 둘 다 원천 테이블로 만드는 위험
- 왜 문제가 되는가: 두 원천이 불일치할 수 있다.
- 6단계 예방 원칙: Timeline은 Change Card 표현으로 둔다.
- 1차 구현 시 대응: Timeline 테이블을 만들지 않는다.
- 후순위 검토 사항: 읽기 성능 필요 시 캐시만 검토한다.

### 3. Public Project Page를 별도 원천 테이블처럼 만드는 위험

- 위험 설명: Public Project Page를 별도 원천 테이블처럼 만드는 위험
- 왜 문제가 되는가: 일반 제품 소개 페이지로 변질된다.
- 6단계 예방 원칙: Project의 공개 뷰로 둔다.
- 1차 구현 시 대응: Project/Problem/Hypothesis/공개 카드에서 파생한다.
- 후순위 검토 사항: SEO 스냅샷은 후순위.

### 4. Problem/Hypothesis 이력이 Change Card와 중복되는 위험

- 위험 설명: Problem/Hypothesis 이력이 Change Card와 중복되는 위험
- 왜 문제가 되는가: 이력 원천이 둘로 갈라진다.
- 6단계 예방 원칙: 현재 상태만 별도 객체가 갖고, 이력은 Change Card가 원천이다.
- 1차 구현 시 대응: 별도 이력 테이블을 만들지 않는다.
- 후순위 검토 사항: 복잡한 이력 요구 확인 후 검토.

### 5. AI Draft와 Change Card 승인 상태가 충돌하는 위험

- 위험 설명: AI Draft와 Change Card 승인 상태가 충돌하는 위험
- 왜 문제가 되는가: AI 초안이 공식 기록처럼 보일 수 있다.
- 6단계 예방 원칙: AI Draft에는 승인됨을 두지 않는다.
- 1차 구현 시 대응: Change Card로 전환됨 상태와 카드 참조만 둔다.
- 후순위 검토 사항: 초안 버전 이력은 후순위.

### 6. Project 진행 상태와 공개 상태가 충돌하는 위험

- 위험 설명: Project 진행 상태와 공개 상태가 충돌하는 위험
- 왜 문제가 되는가: 운영 중/공개됨 같은 혼합 상태가 생긴다.
- 6단계 예방 원칙: 진행 상태와 공개 상태를 분리한다.
- 1차 구현 시 대응: 각각 별도 필드 후보로 둔다.
- 후순위 검토 사항: 상태 전환 이력 검토.

### 7. Feedback이 일반 댓글 테이블처럼 변질되는 위험

- 위험 설명: Feedback이 일반 댓글 테이블처럼 변질되는 위험
- 왜 문제가 되는가: BuildMap의 판단 근거 구조가 약해진다.
- 6단계 예방 원칙: Feedback Request에 연결한다.
- 1차 구현 시 대응: 비로그인 댓글창을 만들지 않는다.
- 후순위 검토 사항: 공개 Feedback 정책 세분화.

### 8. Scout Profile과 채용 데이터가 1차 범위를 과도하게 키우는 위험

- 위험 설명: Scout Profile과 채용 데이터가 1차 범위를 과도하게 키우는 위험
- 왜 문제가 되는가: 서비스가 채용 플랫폼처럼 보인다.
- 6단계 예방 원칙: Scout Profile은 선택, 채용 데이터는 장기 확장으로 둔다.
- 1차 구현 시 대응: 로그인 사용자 기반 피드백을 우선한다.
- 후순위 검토 사항: 협업/테스터 요구가 커지면 확장.

### 9. Activity Signal과 히트맵 지표가 과도하게 복잡해지는 위험

- 위험 설명: Activity Signal과 히트맵 지표가 과도하게 복잡해지는 위험
- 왜 문제가 되는가: 인기순위/점수화처럼 오해된다.
- 6단계 예방 원칙: 최근 업데이트와 배지 중심으로 시작한다.
- 1차 구현 시 대응: activity_signals를 선택 후보로 둔다.
- 후순위 검토 사항: 히트맵 산식은 별도 검증.

### 10. 선택 테이블을 너무 빨리 많이 만들어 구현 범위가 커지는 위험

- 위험 설명: 선택 테이블을 너무 빨리 많이 만들어 구현 범위가 커지는 위험
- 왜 문제가 되는가: 1차 구현이 지연된다.
- 6단계 예방 원칙: 필수 테이블과 선택 테이블을 분리한다.
- 1차 구현 시 대응: Save/Follow, tags, links도 범위에 따라 제외 가능하게 둔다.
- 후순위 검토 사항: 사용자 반응 후 도입.

### 11. 공개/비공개 상태 설계가 허술해 민감 정보가 노출되는 위험

- 위험 설명: 공개/비공개 상태 설계가 허술해 민감 정보가 노출되는 위험
- 왜 문제가 되는가: Builder 신뢰와 데이터 안전성이 훼손된다.
- 6단계 예방 원칙: 기본 비공개, 명시 공개 전환 원칙을 둔다.
- 1차 구현 시 대응: 공개 상태 필드를 분리한다.
- 후순위 검토 사항: RLS 설계 전 공개 정책 점검.

### 12. 파생/표현 데이터를 중복 저장해 불일치가 발생하는 위험

- 위험 설명: 파생/표현 데이터를 중복 저장해 불일치가 발생하는 위험
- 왜 문제가 되는가: 공개 페이지와 원천 카드가 달라진다.
- 6단계 예방 원칙: 파생 우선, 캐시는 후순위로 둔다.
- 1차 구현 시 대응: 별도 Public Page 원천 테이블을 만들지 않는다.
- 후순위 검토 사항: 성능 요구 발생 시 캐시 무효화 설계.
