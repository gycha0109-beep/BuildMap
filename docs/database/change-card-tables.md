# Change Card 관련 테이블 후보

> 이 문서는 DB 스키마 초안 문서다. 실제 SQL, Supabase migration, RLS 정책, API, 프론트엔드 구현은 작성하지 않는다. 테이블명과 필드명은 후보이며 최종 확정이 아니다.

## 1. 문서의 위치

이 문서는 6단계의 핵심 문서다. BuildMap의 DB 스키마 초안에서 Change Card는 핵심 원천 기록이며, Decision Timeline의 기반이다.

## 2. change_cards 테이블 후보

### 정의

Change Card는 프로젝트에서 발생한 하나의 의사결정 단위다. 단순 로그, Todo, 릴리즈 노트가 아니다.

### 1차 포함 여부

1차 필수다.

## 3. 주요 필드 후보 재정리

### 필수 후보

## 4. Change Card 유형 후보

- 문제 발견
- 문제 정의 수정
- 가설 생성
- 가설 반박
- 실험
- 사용자 피드백
- 기능 추가
- 기능 제거
- 판단 유지
- 판단 수정
- 방향 전환
- 릴리즈
- 인수인계 메모

상태값과 유형값은 아직 DB enum처럼 최종 확정하지 않는다. 다만 유형이 없는 카드가 Timeline에 쌓이면 판단 흐름이 흐려지므로 1차에서 유형은 필수 후보로 둔다.

## 5. 상태축 분리

Change Card 작업 상태 후보:

- 초안
- 수정 중
- 승인됨
- 보류됨

Change Card 공개 상태 후보:

- 내부 전용
- 공개 가능
- 공개됨
- 민감 정보 포함

두 상태축은 반드시 분리한다. 하나의 상태 필드에 승인과 공개를 함께 넣으면 공식 기록 여부와 외부 노출 여부가 충돌한다.

## 6. Problem Definition 연결 방식 후보

### 선택 A: change_cards에 직접 선택 참조

예상 의미: Change Card가 관련 Problem Definition 하나를 선택적으로 참조한다.

장점:

- 단순하다.
- 1차 구현에 적합하다.
- 화면 구현이 쉽다.

단점:

- 하나의 Change Card가 여러 문제 정의 후보와 연결되는 경우 한계가 있다.

### 선택 B: 연결 테이블 후보

예상 의미: Change Card와 Problem Definition을 별도 연결 구조로 연결한다.

장점:

- 유연하다.
- 다대다 연결이 가능하다.

단점:

- 1차 구현이 복잡해진다.
- 입력 부담과 쿼리 복잡도가 늘어난다.

권장: 1차 구현에서는 직접 선택 참조를 우선 검토한다.

## 7. Hypothesis 연결 방식 후보

### 선택 A: change_cards에 직접 선택 참조

장점:

- Builder가 카드 승인 시 하나의 주요 가설과 쉽게 연결할 수 있다.
- 1차 UX와 DB 구조가 단순하다.

단점:

- 여러 가설을 동시에 다루는 카드에는 부족할 수 있다.

### 선택 B: change_card_links 같은 연결 테이블 후보

장점:

- 여러 가설, 피드백, 문제 정의와 연결 가능하다.

단점:

- 1차에서 과도하다.
- Change Card 작성을 복잡하게 만든다.

권장: 1차는 직접 선택 참조를 우선하고, 복수 가설 연결은 후순위로 둔다.

## 8. Feedback 연결 방식 후보

### 선택 A: 직접 선택 참조

Change Card가 주요 Feedback 하나 또는 Feedback Request 하나를 참조한다.

장점:

- 피드백 반영 카드 작성이 단순하다.
- 1차 구현 범위를 줄인다.

단점:

- 여러 Feedback을 묶어 판단한 경우 표현이 부족하다.

### 선택 B: 연결 테이블 후보

여러 Feedback을 하나의 Change Card에 연결한다.

장점:

- 사용자 테스트 여러 건을 근거로 묶을 수 있다.

단점:

- 구현 범위가 커진다.
- 일반 댓글 관리처럼 변질될 수 있다.

권장: 1차는 직접 선택 참조 또는 Feedback Request 중심 연결을 우선한다. 여러 Feedback 묶음은 후순위다.

## 9. 1차 단순화 원칙

- 필수 필드를 최소화한다.
- 연결은 가능하면 선택 참조로 시작한다.
- 연결 테이블은 실제 복수 연결 필요가 확인된 뒤 도입한다.
- Builder가 카드 작성 부담을 느끼지 않도록 AI 초안과 선택 입력을 적극 활용한다.

## 10. 추가 검토 필요 사항

- 원문 메모를 Change Card에 복사 저장할지, Rough Note 참조만 둘지
- 승인 후 카드 수정 이력을 저장할지
- 공개 상태 변경 이력을 저장할지
- 중요도와 정렬 우선순위를 별도로 둘지
- 복수 연결이 필요한 시점의 기준
