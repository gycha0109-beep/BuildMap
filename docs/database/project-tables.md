# Project 관련 테이블 후보

> 이 문서는 DB 스키마 초안 문서다. 실제 SQL, Supabase migration, RLS 정책, API, 프론트엔드 구현은 작성하지 않는다. 테이블명과 필드명은 후보이며 최종 확정이 아니다.

## 1. 포함할 테이블 후보

- `projects`
- `project_links`
- `project_tags`
- `project_saves` 또는 `project_follows`

## 2. projects

### 1차 포함 여부

1차 필수다.

### 제품상 역할

Project는 판단 흐름이 쌓이는 중심 컨테이너다. 제품 소개글이 아니라 문제 정의, 가설, 변화 카드, Feedback Request, 공개 뷰를 묶는다.

### 주요 필드 후보

| 필드 후보 | 의미 | 필수 여부 | 공개 가능성 | 관련 화면 |
|---|---|---|---|---|
| 프로젝트명 | 프로젝트 식별 이름 | 필수 | 공개 가능 | 생성, 공개 페이지 |
| 한 줄 정의 | 짧은 프로젝트 설명 | 필수 | 공개 가능 | 생성, 카드 그리드 |
| 현재 진행 상태 | 아이디어/제작 중/테스트 중/베타/운영 중 등 | 필수 | 공개 가능 | 대시보드, 공개 페이지 |
| 공개 상태 | 비공개/링크 공개/전체 공개 | 필수 | 권한 판단 | 공개 전환 |
| 소유 Builder | 프로젝트 책임자 | 필수 | 공개 가능 | 공개 페이지 |
| 현재 필요한 것 요약 | 외부에 보여주는 현재 도움 필요 상태 | 선택/강력 권장 | 공개 가능 | 카드 그리드, 공개 페이지 |
| 최근 전환 요약 | 최근 핵심 Change Card에서 파생 | 파생 후보 | 공개 가능 | 공개 페이지 |
| 생성 시점 | 프로젝트 생성 시점 | 필수 후보 | 공개 가능 | 대시보드 |
| 최근 업데이트 시점 | 탐색 정렬 기본값 | 필수 후보 | 공개 가능 | 카드 그리드 |

### Project 진행 상태 후보

- 아이디어
- 제작 중
- 테스트 중
- 베타
- 운영 중
- 일시 중단
- 종료됨

### Project 공개 상태 후보

- 비공개
- 링크 공개
- 전체 공개

### 현재 필요한 것과 Feedback Request의 구분

Project의 현재 필요한 것은 외부에 보여주는 요약 상태다. 구체적 질문, 테스트 조건, 응답 수집은 Feedback Request에서 다룬다.

예:

- Project 요약: 첫 화면 이해도 피드백이 필요합니다.
- Feedback Request: 사이드프로젝트 경험자 5명에게 첫 화면을 보여주고 10초 안에 서비스 목적을 이해하는지 확인하고 싶습니다.

## 3. project_links

### 1차 포함 여부

1차 선택이다.

### 역할

데모 링크, GitHub 링크, Notion 링크, Figma 링크, 기타 외부 링크를 저장하는 단순 링크 후보이다. 외부 GitHub/Notion 자동 연동은 아니다.

### 주요 필드 후보

| 필드 후보 | 의미 | 필수 여부 | 공개 가능성 | 관련 화면 |
|---|---|---|---|---|
| Project 참조 | 어떤 프로젝트의 링크인지 연결 | 필수 | 비공개 | 공개 페이지 관리 |
| 링크 유형 | 데모, GitHub, Notion, Figma, 기타 | 필수 후보 | 공개 가능 | 공개 페이지 |
| URL | 실제 링크 | 필수 | 공개 가능 | 공개 페이지 |
| 표시 이름 | 사용자에게 보이는 링크명 | 선택 | 공개 가능 | 공개 페이지 |
| 공개 여부 | 링크 노출 여부 | 필수 후보 | 권한 판단 | 공개 전환 |

### 추가 검토 필요

1차 구현이 커지면 관련 링크는 Project의 선택 필드로 단순화할 수 있다.

## 4. project_tags

### 1차 포함 여부

1차 선택이다.

### 역할

프로젝트 분야, 도구, 상태 탐색에 도움을 주는 후보이다.

### 설계 후보

| 후보 | 설명 | 장점 | 단점 |
|---|---|---|---|
| 단순 문자열/배열 후보 | Project에 태그 목록을 직접 둔다. | 구현이 단순하다. | 검색/정규화 한계가 있다. |
| 별도 태그 후보 | 태그를 별도 구조로 둔다. | 탐색과 통계에 유리하다. | 1차 구현이 커진다. |

권장: 1차에서는 단순 구조를 우선 검토하고, 태그 기반 탐색 요구가 확인되면 확장한다.

## 5. project_saves 또는 project_follows

### 1차 포함 여부

1차 선택이다. 핵심 Decision Timeline 흐름보다 우선하지 않는다.

### 역할

Scout 또는 로그인 사용자가 공개 프로젝트를 재방문하거나 관심 프로젝트를 저장하는 데 유용하다.

### 주요 필드 후보

| 필드 후보 | 의미 | 필수 여부 | 공개 가능성 | 관련 화면 |
|---|---|---|---|---|
| 사용자 참조 | 저장한 사용자 | 필수 | 비공개 기본 | Scout 저장 목록 |
| Project 참조 | 저장된 프로젝트 | 필수 | 비공개 기본 | 카드 그리드 |
| 저장 유형 | 저장 또는 팔로우 구분 후보 | 선택 | 비공개 기본 | 탐색 |
| 생성 시점 | 저장 시점 | 필수 후보 | 비공개 | 저장 목록 |

### 추가 검토 필요

Save와 Follow를 분리할지 하나로 둘지 결정이 필요하다. 1차 구현이 커지면 후순위로 내릴 수 있다.
