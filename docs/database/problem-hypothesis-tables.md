# Problem Definition과 Hypothesis 테이블 후보

> 이 문서는 DB 스키마 초안 문서다. 실제 SQL, Supabase migration, RLS 정책, API, 프론트엔드 구현은 작성하지 않는다. 테이블명과 필드명은 후보이며 최종 확정이 아니다.

## 1. 포함할 테이블 후보

- `problem_definitions`
- `hypotheses`

두 후보는 프로젝트의 판단 중심축을 담지만, 과거 이력의 원천은 Change Card다.

---

## 2. problem_definitions

### 제품상 역할

현재 문제 정의를 나타낸다. BuildMap에서 문제 정의는 프로젝트가 왜 존재하는지 설명하는 중심축이다.

### 1차 포함 여부

1차 필수다.

### 주요 필드 후보

| 필드 후보 | 의미 | 필수 여부 | 공개 가능성 | 관련 화면 |
|---|---|---|---|---|
| Project 참조 | 어떤 프로젝트의 문제 정의인지 연결한다. | 필수 | 비공개/공개 판단에 영향 | 문제/가설 입력 |
| 현재 문제 정의 문장 | 현재 프로젝트가 해결하려는 문제다. | 필수 | 공개 가능 | 공개 페이지, Timeline 상단 |
| 설명 또는 맥락 | 문제 정의의 짧은 보충 설명이다. | 선택 | 공개 선택 | 문제 정의 상세 |
| 생성 시점 | 문제 정의 생성 시점이다. | 필수 후보 | 공개 가능 | 내부 관리 |
| 최근 수정 시점 | 현재 문제 정의가 바뀐 시점이다. | 필수 후보 | 공개 가능 | 대시보드 |
| 공개 여부 후보 | 문제 정의 노출 여부다. | 선택 또는 Project 공개 정책 연동 | 권한 판단 | 공개 전환 |

### 이력 처리 원칙

- 현재 Problem Definition은 현재 상태만 나타낸다.
- 과거 문제 정의 이력의 원천은 Change Card다.
- 문제 정의가 바뀌면 가능하면 “문제 정의 수정” 유형의 Change Card 생성을 유도한다.
- 별도 이력 테이블은 후순위로 보류한다.

### 추가 검토 필요

- 한 Project가 여러 문제 정의 후보를 동시에 운영할 수 있는지 여부는 후순위다.
- 1차에서는 현재 문제 정의 1개를 우선한다.

## 3. hypotheses

### 제품상 역할

Hypothesis는 기능 목록이 아니라 프로젝트가 현재 맞다고 믿고 있는 판단이다. 실험, 피드백, Change Card와 연결되어야 한다.

### 1차 포함 여부

1차 필수다.

### 주요 필드 후보

| 필드 후보 | 의미 | 필수 여부 | 공개 가능성 | 관련 화면 |
|---|---|---|---|---|
| Project 참조 | 어떤 프로젝트의 가설인지 연결한다. | 필수 | 비공개/공개 선택 | 문제/가설 화면 |
| 가설 문장 | 프로젝트가 맞다고 믿는 판단이다. | 필수 | 공개 가능 | 문제/가설 화면, 공개 페이지 |
| 가설 상태 | 가정, 검증 중, 일부 확인 등 현재 상태다. | 필수 후보 | 공개 가능 | 대시보드, 공개 페이지 |
| 근거 요약 | 현재 상태를 설명하는 간단한 근거다. | 선택 | 공개 선택 | 가설 상세 |
| 생성 시점 | 가설 생성 시점이다. | 필수 후보 | 공개 가능 | Timeline |
| 최근 수정 시점 | 최근 변경 시점이다. | 필수 후보 | 공개 가능 | 대시보드 |


### Hypothesis 상태 후보

- 가정
- 검증 중
- 일부 확인
- 확인됨
- 반박됨
- 보류

### 이력 처리 원칙

- Hypothesis는 현재 가설 문장과 현재 상태를 가진다.
- 가설 생성, 반박, 보류, 수정의 판단 흐름은 Change Card가 원천이다.
- 상태 변경 시 관련 Change Card 연결을 유도한다.
- 모든 상태 변경에 Change Card 생성을 강제하지 않는다.

### 관계

- Project에 속한다.
- Change Card와 선택적으로 연결된다.
- Feedback Request의 대상이 될 수 있다.
- 공개 프로젝트 페이지의 현재 가설 요약으로 노출될 수 있다.

### 추가 검토 필요

- 가설 여러 개를 동시에 운영하는 화면 부담을 어떻게 줄일지 검토가 필요하다.
- 상태 변경 이력을 별도 테이블로 만들지 않고 Change Card로 충분히 추적 가능한지 실제 데이터로 검증해야 한다.
