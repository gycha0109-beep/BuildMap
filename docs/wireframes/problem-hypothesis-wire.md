# 문제 정의와 가설 입력 화면 텍스트 와이어

## 1. 화면 목적

이 화면은 프로젝트가 무엇을 믿고 움직이는지 정리하는 곳이다.

문제 정의와 가설은 기능 목록이 아니다. BuildMap에서 문제 정의는 프로젝트의 중심축이고, 가설은 Builder가 현재 맞다고 믿고 있는 판단이다.

문제 정의와 가설이 바뀌면 프로젝트의 방향도 바뀐다. 따라서 이 화면은 단순 입력 폼이 아니라 Decision Timeline의 출발점이다.

## 2. 주요 사용자

- Builder
- 프로젝트를 정리하는 팀원
- 인수인계를 준비하는 기존 Builder

## 3. 진입 조건

- 프로젝트가 생성되어 있다.
- 프로젝트 대시보드, 프로젝트 상세, Decision Timeline, 프로젝트 생성 완료 화면에서 진입할 수 있다.

## 4. 문제 정의 입력 영역

### 표시 정보

- 현재 문제 정의
- 마지막 수정일
- 마지막 수정 이유가 연결된 변화 카드 여부
- 공개 페이지 노출 여부

### 입력 필드

제품 문서용 표현 기준:

- 현재 문제 정의
- 문제가 발생하는 상황
- 영향을 받는 사용자
- 문제를 의심하게 된 근거
- 아직 불확실한 부분

### 도움말 문구

> 문제 정의는 한 번 쓰고 끝나는 문장이 아닙니다. 사용자의 반응과 실험 결과에 따라 바뀔 수 있고, 바뀐 이유가 중요한 변화 카드가 됩니다.

### 예시

초기 문제 정의:

> 사람들은 장기 목표를 세우지만 오늘 할 일과 연결하지 못한다.

수정된 문제 정의:

> 사람들은 장기 목표 자체보다 이번 주에 무엇을 증명해야 하는지 몰라 실행이 끊긴다.

## 5. 문제 정의 수정 이력 안내

### 표시 정보

- 이전 문제 정의 요약
- 현재 문제 정의 요약
- 변경 이유가 있는지 여부
- 연결된 변화 카드

### 주요 행동

- 수정 이력 보기
- 문제 정의 변경을 변화 카드로 기록
- 연결된 Decision Diff 보기

### 문제 정의가 바뀔 때 안내 문구

> 문제 정의가 바뀌면 프로젝트의 방향이 바뀔 수 있습니다. 왜 바뀌었는지 변화 카드로 남기면 나중에 프로젝트 맥락을 이해하기 쉬워집니다.

## 6. 가설 입력 영역

### 표시 정보

- 현재 가설 목록
- 각 가설의 상태
- 연결된 변화 카드 수
- 연결된 피드백 수
- 다음 확인 사항

### 입력 필드

- 가설 문장
- 가설 상태
- 검증 방법 메모
- 연결된 문제 정의
- 관련 변화 카드

### 도움말 문구

> 가설은 “만들 기능”이 아니라 “프로젝트가 맞다고 믿고 있는 판단”입니다.

### 예시

가설:

> 사용자는 긴 목표 계획보다 이번 주에 확인할 수 있는 증거 단위를 더 쉽게 이해한다.

상태:

> 검증 중

다음 확인:

> 첫 화면에서 사용자가 10초 안에 목적을 이해하는지 확인한다.

## 7. 가설 상태 선택

상태 후보는 다음과 같다.

| 상태 | 의미 |
|---|---|
| 가정 | 아직 근거가 부족하지만 현재 판단의 출발점인 상태 |
| 검증 중 | 실험, 피드백, 테스트를 통해 확인 중인 상태 |
| 일부 확인 | 일부 근거는 있으나 충분히 확정하기 어려운 상태 |
| 확인됨 | 현재 기준에서 유지할 만한 근거가 있는 상태 |
| 반박됨 | 테스트나 피드백으로 약해졌거나 틀렸다고 판단한 상태 |
| 보류 | 지금 당장 검증하지 않기로 한 상태 |

## 8. 가설과 변화 카드 연결 안내

### 기본 원칙

- 변화 카드는 단독 생성이 가능하다.
- 다만 승인 시 문제 또는 가설과 연결하도록 유도한다.
- 처음부터 모든 카드에 연결을 강제하면 기록 부담이 커진다.

### 안내 문구

> 이 변화 카드가 어떤 문제나 가설과 관련되어 있다면 연결해주세요. 연결하면 나중에 Decision Timeline과 Decision Diff에서 판단 흐름을 더 쉽게 볼 수 있습니다.

## 9. 문제 정의가 바뀔 때 변화 카드 생성 유도

### 표시 조건

- 기존 문제 정의가 수정됨
- 의미 있는 문장 변경이 발생함
- 사용자가 저장 버튼을 누름

### 안내 메시지

> 문제 정의가 변경되었습니다. 변경 이유를 변화 카드로 남기시겠습니까?

### 가능한 행동

- 변화 카드로 남기기
- 지금은 저장만 하기
- 취소

### 변화 카드 초안에 포함될 수 있는 정보

- 기존 문제
- 새 문제
- 변경 이유
- 근거
- 다음 확인 사항

## 10. 빈 상태

### 문제 정의가 없는 경우

메시지:

> 아직 문제 정의가 없습니다. BuildMap에서는 프로젝트가 어떤 문제에서 시작했는지가 가장 중요한 첫 기록입니다.

행동:

- 문제 정의 입력하기
- 예시 보기

### 가설이 없는 경우

메시지:

> 아직 가설이 없습니다. 프로젝트가 성립하려면 무엇이 사실이어야 하는지 적어보세요.

행동:

- 첫 가설 추가
- 건너뛰기

## 11. 수정 상태

### 표시 정보

- 저장 전 변경 사항 있음
- 변경된 문제 정의 또는 가설 강조
- 변화 카드 생성 유도 여부

### 주요 버튼

- 저장
- 저장하고 변화 카드 만들기
- 취소

## 12. 초기 구현 요소

- 현재 문제 정의 입력/수정
- 가설 목록
- 가설 상태 선택
- 문제 정의 변경 시 변화 카드 생성 유도
- 가설과 변화 카드 연결 안내

## 13. 후순위 요소

- 문제 정의 버전 비교
- 가설별 실험 보드
- 가설 검증 히스토리 시각화
- 문제 정의 변경 자동 감지
- AI 기반 문제 정의 재정리 제안

## 14. 추가 검토 필요

- 문제 정의 변경을 얼마나 자동으로 변화 카드화할지
- 가설 상태 변경만으로 변화 카드를 생성할지
- 문제 정의와 가설의 필수 연결 수준
