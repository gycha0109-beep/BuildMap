# Builder 대시보드 텍스트 와이어

## 1. 화면 목적

Builder 대시보드는 Builder가 자신의 프로젝트 상태를 한눈에 파악하고, 다음에 어떤 판단을 기록하거나 검증해야 하는지 확인하는 화면이다.

이 화면은 단순 Todo 대시보드가 아니다. BuildMap의 대시보드는 “오늘 해야 할 일 목록”보다 다음 질문에 답해야 한다.

- 지금 어떤 프로젝트가 진행 중인가?
- 어떤 변화 카드가 승인 대기 중인가?
- 어떤 가설이 아직 검증되지 않았는가?
- 어떤 피드백이 도착했는가?
- 공개 가능한 판단 기록이 있는가?
- 지금 프로젝트는 어떤 판단 위에 서 있는가?

## 2. 주요 사용자

- Builder
- 창업자
- 학생 메이커
- 개발자
- 디자이너
- PM
- 비개발자
- 바이브코딩 유저
- 사이드프로젝트 제작자

## 3. 진입 조건

- 사용자가 Builder로 가입 또는 역할 선택을 완료했다.
- 프로젝트가 없을 수도 있고, 하나 이상 있을 수도 있다.
- 첫 진입 시에는 프로젝트 생성으로 유도한다.

## 4. 상단 요약 영역

### 표시 정보

- 현재 진행 중인 프로젝트 수
- 최근 변화 카드 수
- 승인 대기 중인 AI 초안 수
- 검증 중인 가설 수
- 공개 전환이 가능한 변화 카드 수
- 최근 받은 피드백 수

### 사용자가 이해해야 하는 것

Builder는 대시보드에 들어오자마자 프로젝트의 “활동량”이 아니라 “판단 흐름의 상태”를 이해해야 한다.

예시 메시지:

> 아직 승인되지 않은 변화 카드 초안이 2개 있습니다. 승인해야 Decision Timeline에 반영됩니다.

> 검증 중인 가설 3개 중 1개는 최근 피드백과 연결되어 있습니다.

### 주요 행동

- 승인 대기 카드 확인
- 미해결 가설 보기
- 최근 피드백 확인
- 새 메모 입력
- 새 프로젝트 생성

## 5. 현재 진행 중인 프로젝트 영역

### 표시 정보

각 프로젝트 카드에는 다음을 표시한다.

- 프로젝트명
- 한 줄 설명
- 현재 상태
- 현재 문제 정의 요약
- 현재 필요한 것
- 최근 전환 요약
- 마지막 변화 카드 작성일
- 공개 상태

### 주요 버튼

- 프로젝트 열기
- Decision Timeline 보기
- 공개 페이지 보기
- 거친 메모 입력

### 잘못 설계하면 생길 문제

프로젝트 카드를 일반 포트폴리오 카드처럼 만들면 BuildMap의 정체성이 흐려진다. 카드에는 썸네일이나 홍보 문구보다 문제 정의, 최근 전환, 현재 필요한 것이 먼저 보여야 한다.

## 6. 최근 Decision Timeline 활동 영역

### 표시 정보

- 최근 승인된 변화 카드
- 최근 핵심 전환 카드
- 최근 문제 정의 수정
- 최근 가설 상태 변경
- 최근 기능 제거 또는 보류 판단

### 카드 표시 방식

각 활동은 다음처럼 짧게 표시한다.

```text
[가설 반박] 장기 목표 선입력 흐름을 제거함
근거: 테스트 참여자들이 첫 화면을 부담스럽게 느꼈음
연결 가설: 장기 목표가 실행 동기를 높인다
```

### 주요 행동

- 변화 카드 자세히 보기
- 연결된 가설 보기
- Decision Timeline 전체 보기

## 7. 미해결 가설 영역

### 표시 정보

- 가설 제목
- 상태: 가정 / 검증 중 / 일부 확인 / 확인됨 / 반박됨 / 보류
- 연결된 변화 카드 수
- 최근 피드백 여부
- 다음 확인 사항

### 사용자가 해야 할 행동

- 실험 결과 입력
- 피드백 요청
- 변화 카드 작성
- 가설 상태 변경

### 설계 원칙

가설은 기능 목록이 아니다. 화면에서는 “이 기능을 만들자”가 아니라 “우리가 무엇을 믿고 있는가”가 드러나야 한다.

## 8. 피드백 요청/응답 영역

### 표시 정보

- 열린 피드백 요청
- 새로 도착한 피드백
- 검토 중인 피드백
- 반영된 피드백
- 반영하지 않기로 한 피드백

### 주요 행동

- 피드백 읽기
- 피드백을 변화 카드로 전환
- 반영하지 않음으로 표시
- 추가 질문 남기기

### 주의점

피드백은 인기 댓글이 아니다. 피드백은 특정 문제, 가설, 변화 카드, 공개 페이지에 연결되어야 한다.

## 9. 공개 전환이 필요한 카드 알림

### 표시 정보

- 공개 가능한 변화 카드
- 아직 내부 전용인 핵심 전환 카드
- 민감 정보가 포함되어 공개 주의가 필요한 카드

### 주요 행동

- 공개 여부 검토
- 공개 페이지 미리보기
- 공개 제외 유지

### 공개 전환 안내 예시

> 이 변화 카드는 핵심 전환으로 표시되어 있지만 아직 내부 전용입니다. 공개 프로젝트 페이지에 포함할지 검토할 수 있습니다.

## 10. 주요 버튼

### 새 프로젝트 만들기

- 이동: 프로젝트 생성 화면
- 목적: 새로운 판단 흐름 시작

### 거친 메모 입력

- 이동: 거친 메모 입력 화면
- 목적: 회의, 테스트, 작업 중 발생한 판단 재료를 빠르게 남김

### 변화 카드 작성

- 이동: 변화 카드 작성 또는 AI 초안 검토 화면
- 목적: 직접 변화 카드를 만들거나 임시 메모를 구조화

### 공개 페이지 보기

- 이동: 공개 프로젝트 페이지 미리보기 또는 실제 공개 페이지
- 조건: 프로젝트가 공개 가능 상태일 때

## 11. 빈 상태

### 프로젝트가 없는 경우

사용자에게 보여줄 메시지:

> 아직 프로젝트가 없습니다. 완성작을 등록할 필요는 없습니다. 지금 만들고 있는 아이디어나 문제의식부터 기록해보세요.

가능한 행동:

- 첫 프로젝트 만들기
- BuildMap 예시 프로젝트 보기

피해야 할 표현:

- “포트폴리오를 등록하세요”
- “완성된 제품을 업로드하세요”

## 12. 로딩 상태

표시 메시지:

> 프로젝트의 최근 판단 흐름을 불러오는 중입니다.

로딩은 단순 스피너보다 어떤 정보를 불러오는지 설명하는 문구가 적합하다.

## 13. 오류 상태

표시 메시지:

> 프로젝트 판단 흐름을 불러오지 못했습니다. 잠시 후 다시 시도해주세요.

가능한 행동:

- 다시 시도
- 임시 저장된 메모 확인
- 대시보드로 돌아가기

## 14. 초기 구현 요소

- 프로젝트 목록
- 최근 변화 카드
- 승인 대기 AI 초안
- 미해결 가설
- 최근 피드백
- 새 프로젝트 만들기 버튼
- 거친 메모 입력 버튼
- Decision Timeline 이동
- 공개 페이지 보기

## 15. 후순위 요소

- 프로젝트별 활동 지표 시각화
- 개인별 생산성 분석
- 고급 알림 설정
- 팀 단위 대시보드
- 외부 Notion/GitHub 활동 통합
- 인수인계 요약 카드
- 고급 검색

## 16. 추가 검토 필요

- 대시보드에서 여러 프로젝트를 운영하는 Builder의 우선순위를 어떻게 표시할지
- 승인 대기 변화 카드가 많아졌을 때 묶음 처리 방식
- 가설과 변화 카드 연결이 약한 프로젝트를 어떻게 안내할지
