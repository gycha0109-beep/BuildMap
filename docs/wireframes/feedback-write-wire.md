# 피드백 작성 화면 텍스트 와이어

## 1. 화면 목적

피드백 작성 화면은 사용자가 Builder의 특정 판단에 대해 응답하는 화면이다.

피드백은 프로젝트 전체 평가가 아니라 문제 정의, 가설, 변화 카드, 공개 페이지, 테스트 요청 등 특정 대상에 연결되어야 한다.

## 2. 작성자 조건

1차에서는 비로그인 피드백을 보류한다.

기본 가정:

- 최소 계정 또는 로그인 기반 피드백
- 표시 이름 선택 가능
- 무분별한 익명 피드백과 스팸성 응답을 줄이는 방향

### 안내 문구

> 피드백은 Builder의 다음 변화 카드로 이어질 수 있습니다. 최소 계정 기반으로 작성됩니다.

## 3. 피드백 대상 표시

### 표시 정보

- 프로젝트명
- 피드백 요청 질문
- 피드백 대상 유형
- 연결된 문제/가설/변화 카드 요약

### 예시

```text
피드백 대상: 변화 카드
질문: 첫 화면에서 10년 목표 입력을 제거한 판단에 동의하시나요?
연결 가설: 사용자는 긴 계획보다 이번 주 실행 단위를 더 쉽게 이해한다.
```

## 4. 피드백 유형 선택

### 유형 후보

- 공감
- 이해 안 됨
- 개선 제안
- 사용 경험 공유
- 기능 필요성 의견
- 테스터 참여 의사

### 선택 방식

- 단일 또는 복수 선택 가능 여부는 추가 검토 필요
- 초기에는 단일 선택이 단순하다.

## 5. 피드백 내용 입력

### 입력 필드

- 피드백 내용

### 도움말 문구

> 좋다/나쁘다보다 왜 그렇게 느꼈는지 적어주면 Builder가 판단을 업데이트하는 데 도움이 됩니다.

### 예시 문구

- 첫 화면에서 바로 이번 주 목표가 보여서 더 쉽게 시작할 수 있을 것 같습니다.
- 문제 정의는 공감되지만, 대상 사용자가 너무 넓어 보입니다.
- 기능 제거 판단은 이해되지만, 나중에 캘린더 연동은 다시 필요할 수 있습니다.

## 6. 테스터 참여 의사 선택

### 표시 조건

- 피드백 요청 유형에 테스터 참여가 포함된 경우
- 프로젝트가 테스터를 모집 중인 경우

### 선택 후보

- 참여 의사 있음
- 나중에 가능
- 피드백만 남김

### 후속 정보

이메일, 연락 방식, 일정 등의 상세 정보는 초기 구현에서 과도하게 확정하지 않는다.

## 7. 공개 표시 이름

### 표시 정보

- 계정 표시명
- 프로젝트 공개 페이지에서 노출 여부

### 선택 후보

- 표시명 공개
- Builder에게만 표시

### 추가 검토 필요

- 익명 또는 닉네임 피드백을 어느 범위까지 허용할지

## 8. 제출 후 상태

### 제출 완료 메시지

> 피드백이 제출되었습니다. Builder가 검토 후 변화 카드에 반영할 수 있습니다.

### 가능한 행동

- 프로젝트로 돌아가기
- 다른 피드백 요청 보기
- 프로젝트 저장

## 9. Builder가 피드백을 검토하는 흐름

Builder는 받은 피드백을 다음 상태로 처리할 수 있다.

- 새 피드백
- 검토 중
- 반영됨
- 반영하지 않음

피드백이 중요한 판단으로 이어지면 Builder는 이를 변화 카드로 구조화할 수 있다.

## 10. 피드백이 변화 카드로 이어질 수 있다는 안내

사용자에게 보여줄 문구:

> 이 피드백은 Builder가 검토한 뒤 새로운 변화 카드의 근거로 사용될 수 있습니다.

Builder에게 보여줄 문구:

> 이 피드백을 변화 카드 초안으로 구조화할 수 있습니다.

## 11. 초기 구현 요소

- 로그인 기반 피드백 작성
- 피드백 대상 표시
- 피드백 유형 선택
- 피드백 내용 입력
- 테스터 참여 의사 선택
- 공개 표시 이름
- 제출 완료 상태

## 12. 후순위 요소

- 비로그인 피드백
- 피드백 스레드
- 피드백 투표
- 피드백 품질 평가
- 테스터 일정 조율
- 피드백을 AI가 묶어 요약

## 13. 추가 검토 필요

- 비로그인 피드백을 언제 허용할지
- 피드백 공개 범위 기본값
- 부정확하거나 악성 피드백 처리 방식
