# Decision Timeline 화면 텍스트 와이어

## 1. 화면 목적

Decision Timeline은 BuildMap의 중심 화면이다.

이 화면은 일반 활동 로그가 아니다. 단순 최신순 목록처럼 보이면 안 된다. Decision Timeline은 프로젝트가 왜 지금의 모습이 되었는지 보여주는 판단 흐름이어야 한다.

## 2. 주요 사용자

- Builder
- 프로젝트 팀원
- 외부 사용자
- Scout
- 새 팀원
- 협업자
- 멘토
- 심사위원

## 3. 진입 조건

- 프로젝트가 생성되어 있다.
- 승인된 변화 카드가 없을 수도 있다.
- Builder 내부 화면과 공개 프로젝트 페이지 양쪽에서 접근할 수 있다.

## 4. 상단 요약 카드

상단은 Timeline을 읽기 전에 현재 프로젝트의 판단 상태를 요약한다.

### 현재 문제 정의

표시 예시:

> 사용자는 장기 목표와 이번 주 실행 단위를 연결하지 못한다.

사용자가 이해해야 하는 것:

- 프로젝트가 지금 어떤 문제를 중심으로 움직이는지

### 현재 가설

표시 예시:

> 사용자는 긴 계획보다 이번 주에 확인할 수 있는 증거 단위를 더 쉽게 이해한다.

사용자가 이해해야 하는 것:

- 프로젝트가 현재 무엇을 믿고 실험 중인지

### 최근 전환

표시 예시:

> 첫 화면에서 10년 목표 입력을 제거하고 이번 주 목표 선택을 우선 배치함

사용자가 이해해야 하는 것:

- 최근 어떤 판단 변화가 있었는지

### 현재 필요한 것

표시 예시:

> 첫 화면 이해도 피드백과 테스트 참여자가 필요함

사용자가 이해해야 하는 것:

- 지금 이 프로젝트에 어떤 도움을 줄 수 있는지

## 5. Timeline 본문

### 표시 원칙

Timeline은 단순 시간순 목록이 아니라 문제 발견, 가설, 실험, 피드백, 판단, 방향 전환이 연결되어 보이도록 구성한다.

### 변화 카드 표시 정보

각 변화 카드에는 다음을 표시한다.

- 유형
- 제목
- 날짜
- 구조화 요약
- 근거 요약
- 판단 요약
- 변경 내용
- 다음 확인 사항
- 연결된 문제 또는 가설
- 공개 여부
- 중요도

### 카드 예시

```text
[가설 반박] 장기 목표 선입력 흐름 제거
근거: 테스트 참여자들이 첫 화면을 부담스럽게 느꼈음
판단: 초기 진입에서는 이번 주 실행 단위를 먼저 보여준다
변경: 10년 목표 입력을 첫 화면에서 제거
다음 확인: 수정된 첫 화면에서 사용자가 10초 안에 목적을 이해하는지 확인
```

## 6. 핵심 전환 강조 방식

핵심 전환 카드는 일반 카드보다 더 강하게 표시한다.

### 강조 대상

- 문제 정의 수정
- 가설 반박
- 기능 제거
- 방향 전환
- 릴리즈 목적 변경
- 인수인계에 중요한 선택

### 표시 방식

- Timeline에서 접히지 않음
- 프로젝트 진화도에 포함
- 공개 페이지 상단 “최근 전환”에 노출 가능

### 주의점

강조 방식은 과장된 성과 표시가 아니다. 중요한 판단 변화임을 보여주는 것이다.

## 7. 카드 접기/펼치기

### 기본 표시

- 제목
- 유형
- 날짜
- 한 줄 판단 요약

### 펼쳤을 때 표시

- 원문 메모
- 근거
- 판단
- 변경 내용
- 다음 확인 사항
- 연결된 문제/가설
- 피드백

### 카드가 너무 많은 경우

- 핵심 전환만 먼저 표시
- 일반 카드는 접힘 처리
- 유형별 필터 제공

## 8. 필터

### 카드 유형

- 문제 발견
- 문제 정의 수정
- 가설 생성
- 가설 반박
- 실험
- 사용자 피드백
- 기능 추가
- 기능 제거
- 판단 유지
- 판단 수정
- 방향 전환
- 릴리즈
- 인수인계 메모

### 공개 여부

- 전체
- 내부 전용
- 공개 가능
- 공개됨

### 핵심 전환

- 전체
- 핵심 전환만
- 일반 카드만

### 연결된 가설

- 특정 가설 기준 필터
- 가설 미연결 카드 보기

### 기간

- 전체
- 최근 7일
- 최근 30일
- 직접 선택

## 9. Decision Diff로 이동

### 진입 방식

- 상단 요약 카드에서 “초기 판단과 비교”
- 특정 핵심 전환 카드에서 “전환 전후 비교”
- Timeline 필터 결과에서 “이 시점과 현재 비교”

### 이동 목적

Decision Diff는 코드 diff가 아니라 판단 차이를 보여준다.

## 10. 피드백 요청으로 이동

### 진입 방식

- 특정 변화 카드에서 “이 판단에 대한 피드백 요청”
- 상단 현재 문제 정의에서 “문제 정의 피드백 요청”
- 현재 가설에서 “가설 검증 피드백 요청”

### 원칙

피드백 요청은 막연한 평가가 아니라 특정 판단에 대한 질문이어야 한다.

## 11. 공개 페이지 미리보기로 이동

### Builder가 보는 경우

- 공개될 카드만 미리보기
- 내부 전용 카드 제외
- 민감 정보 포함 카드 경고

### 주요 행동

- 공개 페이지 보기
- 공개 카드 관리
- 공개 제외 유지

## 12. 새 팀원/협업자가 읽는 경우의 사용성

### 목표

새 팀원이나 외부 협업자가 5분 안에 프로젝트의 큰 흐름을 이해할 수 있어야 한다.

### 우선 노출 정보

- 처음 문제 정의
- 현재 문제 정의
- 주요 가설
- 핵심 전환
- 기능 제거 이유
- 현재 필요한 것

### 보조 안내

> 세부 코드는 GitHub에서, 장문 문서는 Notion에서 확인할 수 있습니다. BuildMap에서는 왜 그런 판단을 했는지 먼저 확인합니다.

## 13. 빈 상태

### 승인된 변화 카드가 없는 경우

메시지:

> 아직 Decision Timeline에 반영된 변화 카드가 없습니다. 거친 메모를 입력하고 첫 변화 카드를 만들어보세요.

행동:

- 거친 메모 입력
- 직접 변화 카드 작성
- 문제/가설 정리

## 14. 카드가 너무 많은 상태

### 메시지

> 변화 카드가 많습니다. 핵심 전환과 연결된 가설을 기준으로 먼저 볼 수 있습니다.

행동:

- 핵심 전환만 보기
- 유형별 필터
- 특정 가설 기준 보기
- 기간 선택

## 15. 초기 구현 요소

- 상단 요약 카드
- Timeline 본문
- 변화 카드 기본 표시
- 카드 펼치기
- 핵심 전환 표시
- 유형 필터
- 공개 여부 필터
- Decision Diff 이동
- 피드백 요청 이동

## 16. 후순위 요소

- 시각적 프로젝트 진화도
- 가설별 가지 구조
- 카드 간 연결 그래프
- 인수인계 요약 모드
- 고급 검색
- 외부 문서 링크 자동 연결

## 17. 추가 검토 필요

- Timeline 기본 정렬을 시간순으로 할지 판단 흐름순으로 할지
- 공개 Timeline과 내부 Timeline의 차이를 어떻게 보여줄지
- 핵심 전환 기준을 Builder 수동 선택으로만 둘지 일부 추천할지
