# 프로젝트 생성 화면 텍스트 와이어

## 1. 화면 목적

프로젝트 생성 화면은 Builder가 완성작을 등록하는 화면이 아니라, 프로젝트의 판단 흐름을 쌓기 시작하는 화면이다.

처음부터 긴 기획서, 완성된 소개글, 정교한 로드맵을 요구하지 않는다. Builder가 가지고 있는 문제의식, 아직 불확실한 가설, 현재 필요한 도움을 짧게 기록할 수 있어야 한다.

## 2. 기본 흐름

```text
프로젝트명 입력
→ 한 줄 설명 입력
→ 현재 상태 선택
→ 해결하려는 문제 입력
→ 아직 확정되지 않은 가설 입력
→ 현재 필요한 도움 입력
→ 공개 여부 선택
→ 생성 완료 후 첫 프로젝트 대시보드 진입
```

## 3. 1단계: 프로젝트명 입력

### 필수 입력

- 프로젝트명

### 선택 입력

- 임시 이름 여부

### 도움말 문구

> 아직 확정된 이름이 아니어도 괜찮습니다. 나중에 바꿀 수 있습니다.

### 예시 문구

- BuildMap
- 말하는대로
- 주간 실행 증거 플래너
- AI 피부 추천 실험

### 주요 버튼

- 다음
- 임시 저장
- 취소

### 입력하지 않았을 때 처리

프로젝트명은 최소한의 구분자이므로 입력을 요구한다.

표시 메시지:

> 프로젝트를 구분할 수 있는 이름을 입력해주세요. 임시 이름도 괜찮습니다.

### 잘못 설계하면 생길 문제

프로젝트명 단계에서 브랜드명 완성도를 요구하면 시작 장벽이 높아진다.

### 초기 구현 요소

- 프로젝트명 입력
- 글자 수 안내
- 임시 이름 허용 안내

### 후순위 요소

- 중복 이름 추천
- AI 이름 제안
- 공개 URL 미리보기

## 4. 2단계: 한 줄 설명 입력

### 필수 입력

- 한 줄 설명

### 선택 입력

- 대상 사용자
- 프로젝트 분야

### 도움말 문구

> 이 프로젝트가 무엇을 하려는지 한 문장으로 적습니다. 완성된 홍보 문구가 아니어도 됩니다.

### 예시 문구

> 프로젝트가 왜 지금의 모습이 되었는지 기록하는 Decision Timeline 플랫폼

> 장기 목표를 이번 주 실행 증거로 바꾸는 플래너

### 주요 버튼

- 이전
- 다음
- 임시 저장

### 입력하지 않았을 때 처리

표시 메시지:

> 한 줄 설명은 나중에 수정할 수 있습니다. 지금은 프로젝트의 방향을 알아볼 수 있을 정도면 충분합니다.

### 잘못 설계하면 생길 문제

마케팅 카피처럼 요구하면 BuildMap이 Product Hunt식 소개 페이지처럼 보일 수 있다.

### 초기 구현 요소

- 한 줄 텍스트 입력
- 예시 표시

### 후순위 요소

- 여러 버전의 한 줄 설명 비교
- 공개 페이지용 문구와 내부용 문구 분리

## 5. 3단계: 현재 상태 선택

### 필수 입력

- 현재 상태 선택

### 선택 가능한 상태 예시

- 아이디어 단계
- 문제 정의 중
- 가설 검증 중
- 초기 실험 중
- 테스트 중
- 공개 준비 중
- 운영 중
- 방향 전환 중
- 보류 중

### 도움말 문구

> 현재 상태는 완성도를 평가하기 위한 값이 아닙니다. 지금 어떤 판단 단계에 있는지 보여주기 위한 정보입니다.

### 주요 버튼

- 이전
- 다음
- 임시 저장

### 입력하지 않았을 때 처리

기본값을 강제로 넣기보다 선택을 유도한다.

표시 메시지:

> 지금 프로젝트가 어느 판단 단계에 가까운지 선택해주세요.

### 잘못 설계하면 생길 문제

상태를 “완성도”처럼 보여주면 Builder가 미완성 프로젝트를 올리기 부담스러워진다.

### 초기 구현 요소

- 단일 선택
- 상태 설명

### 후순위 요소

- 상태 변경 이력
- 상태별 추천 다음 행동

## 6. 4단계: 해결하려는 문제 입력

### 필수 입력

- 해결하려는 문제

### 선택 입력

- 문제가 관찰된 상황
- 문제가 발생하는 사용자
- 문제를 의심하게 된 계기

### 도움말 문구

> 이 프로젝트가 처음 본 문제를 적습니다. 정답처럼 쓸 필요는 없습니다. 문제 정의는 나중에 바뀔 수 있고, 바뀐 이유가 중요한 기록이 됩니다.

### 예시 문구

> 사람들은 장기 목표를 세우지만 오늘 할 일과 연결하지 못한다.

> 프로젝트는 만들어지지만 왜 그런 결정을 했는지는 시간이 지나면 사라진다.

### 주요 버튼

- 이전
- 다음
- 임시 저장

### 입력하지 않았을 때 처리

문제 정의는 BuildMap의 중심축이므로 최소 입력을 요구한다.

표시 메시지:

> BuildMap에서는 프로젝트의 첫 기록이 문제 정의에서 시작됩니다. 아직 불확실해도 현재 생각하는 문제를 적어주세요.

### 잘못 설계하면 생길 문제

긴 문제 정의 템플릿을 요구하면 비개발자와 초기 Builder가 이탈할 수 있다.

### 초기 구현 요소

- 짧은 텍스트 입력
- 예시 문구
- “나중에 수정 가능” 안내

### 후순위 요소

- 문제 정의 변경 이력 자동 비교
- 문제 정의 수정 시 변화 카드 자동 제안

## 7. 5단계: 아직 확정되지 않은 가설 입력

### 필수 입력

- 최소 0개 또는 1개를 허용할지 추가 검토 필요

### 선택 입력

- 첫 가설
- 가설 상태
- 검증 방법 메모

### 도움말 문구

> 가설은 기능 목록이 아닙니다. 이 프로젝트가 맞다고 믿고 있는 판단입니다.

### 예시 문구

> 사용자는 긴 목표 계획보다 이번 주에 확인할 수 있는 증거 단위를 더 쉽게 이해한다.

### 주요 버튼

- 이전
- 다음
- 건너뛰기
- 임시 저장

### 입력하지 않았을 때 처리

1차에서는 가설 입력을 강제하지 않는 것이 적합하다. 대신 프로젝트 대시보드에서 가설 입력을 유도한다.

표시 메시지:

> 아직 가설이 명확하지 않다면 건너뛸 수 있습니다. 프로젝트 생성 후 다시 정리할 수 있습니다.

### 잘못 설계하면 생길 문제

가설을 필수로 강제하면 프로젝트 생성 자체가 연구 보고서처럼 느껴질 수 있다.

### 초기 구현 요소

- 선택 입력
- 가설 상태 기본값: 가정

### 후순위 요소

- 가설 템플릿
- 가설과 실험 연결 추천

## 8. 6단계: 현재 필요한 도움 입력

### 필수 입력

- 현재 필요한 도움 선택 또는 텍스트 입력

### 선택 가능한 도움 예시

- 문제 정의 피드백
- 첫 화면 이해도 피드백
- 기능 필요성 검증
- 테스터 모집
- 협업자 필요
- 멘토링 필요
- 사용자 인터뷰 필요
- 아직 모름

### 도움말 문구

> BuildMap의 공개 페이지는 완성작 홍보보다 지금 어떤 도움이 필요한지를 보여주는 데 중요합니다.

### 예시 문구

> 첫 화면에서 사용자가 서비스 목적을 이해하는지 피드백이 필요합니다.

### 주요 버튼

- 이전
- 다음
- 임시 저장

### 입력하지 않았을 때 처리

선택을 강제하기보다 “아직 모름”을 제공한다.

### 잘못 설계하면 생길 문제

현재 필요한 도움을 단순 구인 항목처럼 만들면 Scout 흐름이 채용 플랫폼처럼 오해될 수 있다.

### 초기 구현 요소

- 도움 유형 선택
- 짧은 설명 입력

### 후순위 요소

- 도움 요청별 공개 범위
- 테스터 신청 폼 연결
- 협업자 요청 상세 조건

## 9. 7단계: 공개 여부 선택

### 필수 입력

- 프로젝트 공개 상태 선택

### 선택 가능한 상태

- 비공개
- 링크 공개
- 전체 공개

### 도움말 문구

> 처음부터 모든 기록을 공개할 필요는 없습니다. 내부 기록으로 시작한 뒤 공개 가능한 변화 카드만 외부에 보여줄 수 있습니다.

### 주요 버튼

- 이전
- 프로젝트 만들기
- 임시 저장

### 입력하지 않았을 때 처리

기본값은 비공개가 적합하다.

### 잘못 설계하면 생길 문제

첫 프로젝트 생성에서 공개를 기본값으로 두면 민감한 내부 기록이 노출될 수 있다.

### 초기 구현 요소

- 기본값: 비공개
- 공개 상태 설명
- 공개 전환은 나중에 가능하다는 안내

### 후순위 요소

- 카드별 공개 설정
- 공개 페이지 미리보기
- 민감 정보 감지

## 10. 8단계: 생성 완료 후 첫 프로젝트 대시보드 진입

### 완료 화면 표시 정보

- 프로젝트명
- 한 줄 설명
- 현재 문제 정의
- 현재 가설 또는 가설 입력 유도
- 현재 필요한 도움
- 다음 권장 행동

### 다음 권장 행동 예시

- 첫 거친 메모 입력하기
- 첫 가설 추가하기
- Decision Timeline 보기
- 공개 페이지 미리보기

### 주요 버튼

- 프로젝트 대시보드로 이동
- 거친 메모 입력
- 문제/가설 수정

## 11. 초기 구현 요소

- 단계형 프로젝트 생성
- 프로젝트명
- 한 줄 설명
- 현재 상태
- 해결하려는 문제
- 선택적 가설
- 현재 필요한 도움
- 공개 여부
- 생성 후 프로젝트 대시보드 이동

## 12. 후순위 요소

- AI 프로젝트 소개 정리
- 분야별 템플릿
- 공개 URL 설정
- 팀원 초대
- 외부 링크 자동 수집
- 프로젝트 생성 중 예시 프로젝트 참고

## 13. 추가 검토 필요

- 첫 프로젝트 생성에서 가설을 필수로 할지 선택으로 둘지
- 현재 상태 선택지를 얼마나 단순화할지
- 공개 상태를 프로젝트 단위와 카드 단위로 어떻게 분리할지
