# 3단계 화면 및 흐름 범위 결정

## 1. 문서 목적

이 문서는 BuildMap 3단계에서 확정한 화면/흐름 범위와 보류한 항목을 기록한다.

3단계는 구현 단계가 아니다. 3단계는 화면과 흐름의 제품 설계 단계다.

## 2. 3단계에서 확정한 것

## 2.1 3단계는 화면/흐름 제품 설계 단계다

이번 단계에서는 무슨 화면이 필요하고, 각 화면에서 사용자가 무엇을 보고 무엇을 해야 하는지 정리한다.

코드 구현, DB 설계, API 설계, 컴포넌트 설계, 디자인 시스템은 확정하지 않는다.

## 2.2 Builder 핵심 흐름은 프로젝트 생성부터 Decision Timeline 반영까지다

첫 구현의 Builder 핵심 흐름은 다음으로 본다.

```text
프로젝트 생성
→ 문제 정의
→ 가설
→ 거친 메모 입력
→ AI 변화 카드 초안
→ Builder 승인
→ Decision Timeline 반영
→ 공개 프로젝트 페이지
```

Scout와 히트맵은 최소 탐색 수준으로 둔다.

## 2.3 변화 카드 작성 흐름은 BuildMap의 핵심 화면이다

변화 카드는 단순 로그가 아니라 의사결정 단위다.

변화 카드 작성 흐름은 다음을 포함한다.

- 거친 메모 입력
- AI 초안 구조화
- Builder 검토
- 유형 선택 또는 수정
- 문제/가설 연결
- 공개 여부 선택
- 중요도 선택
- 승인
- Decision Timeline 반영

## 2.4 AI는 변화 카드 초안을 구조화한다

AI는 프로젝트를 대신 만들거나 성과를 꾸미지 않는다.

AI는 Builder의 텍스트 메모를 기반으로 문제, 근거, 판단, 변경, 다음 확인 사항을 구조화한다. 음성, 이미지, 회의록, 외부 문서 업로드는 후순위다.

AI 초안은 자동 공개되지 않는다.

## 2.5 Builder가 최종 승인자다

AI 초안은 Builder가 수정하고 승인해야 공식 Decision Timeline에 반영된다.

승인과 공개는 별개의 상태다.

## 2.6 공개 프로젝트 페이지는 제품 소개보다 판단 흐름을 우선한다

공개 프로젝트 페이지의 기본 정보 순서는 다음으로 둔다.

```text
프로젝트명
→ 한 줄 정의
→ 현재 상태
→ 현재 필요한 것
→ 문제 정의
→ 현재 가설
→ 최근 전환
→ 프로젝트 진화도
→ 공개 변화 카드
→ 피드백 요청
→ 관련 링크
→ Builder 정보
→ 팔로우/저장
```

공개 프로젝트 페이지는 제품 소개 페이지가 아니라 프로젝트의 판단 흐름과 현재 필요한 도움을 보여주는 페이지다.

## 2.7 내부 기록 후 공개 전환이 기본 정책이다

기본은 내부 비공개다.

Builder가 승인한 변화 카드 중 공개 가능한 것만 공개 프로젝트 페이지에 노출된다. AI 초안은 절대 자동 공개되지 않는다.

## 2.8 Scout는 초기에는 프로젝트 발견/피드백/테스터/협업 중심이다

초기 Scout 흐름은 다음으로 제한한다.

- 공개 프로젝트 탐색
- 공개 프로젝트 페이지 확인
- Decision Timeline 확인
- 특정 변화 카드에 피드백 작성
- 테스터 신청
- 프로젝트 저장
- Builder에게 맥락 기반 연락

채용/헤드헌팅 화면은 장기 확장으로 보류한다.

## 2.9 히트맵은 발견 화면이지만 산식은 보류한다

히트맵은 단순 인기 순위가 아니라 지금 살아 움직이는 프로젝트를 발견하는 입구다.

초기에는 외부 트래픽 데이터가 아니라 내부 활동 지표 기반으로 가정한다. 다만 타일 크기, 색상, 활동성 산식은 확정하지 않는다.

초기 구현에서는 기본 카드 그리드로 대체 가능하다.

## 2.10 인수인계는 초기에는 Decision Timeline의 활용 사례로 다룬다

인수인계는 BuildMap의 중요한 차별점이다.

다만 초기에는 별도 전용 모드로 확정하지 않는다. 프로젝트 상세 페이지와 Decision Timeline의 요약/타임라인 확장 기능으로 다룬다.

전용 인수인계 모드는 2차 확장 후보로 둔다.

## 3. 3단계에서 보류한 것

다음 항목은 3단계에서 확정하지 않는다.

- DB 구조
- API 구조
- 구체 컴포넌트 설계
- 디자인 시스템
- AI 프롬프트 세부 설계
- 히트맵 산식
- 권한 세부 정책
- 채용 기능
- 결제
- 외부 연동
- 보류 중인 고급 브랜딩 표현
- 역량 점수화

## 4. 추가 검토 필요 항목

다음은 4단계 이후에 검토한다.

- 변화 카드 최종 필드와 데이터 구조
- 문제/가설/변화 카드의 관계 모델
- 공개 프로젝트 페이지의 최소 UI 구성
- Builder 대시보드의 정보 우선순위
- Scout 탐색 필터의 최소 범위
- 피드백 작성에 로그인 필요 여부
- 공개/비공개 권한 정책
- 히트맵 산식과 조작 방지
- AI 구조화 프롬프트와 실패 처리
- 인수인계 요약을 별도 기능으로 분리할지 여부

## 5. 다음 단계 후보

3단계 이후에는 다음 중 하나로 진행할 수 있다.

1. 최소 데이터 구조 설계
2. 변화 카드 필드 정의
3. 화면별 텍스트 와이어 구조 작성
4. 공개 프로젝트 페이지 상세 섹션 설계
5. AI 구조화 입력/출력 계약 설계

다만 바로 구현으로 넘어가기 전에 변화 카드와 Decision Timeline의 최소 데이터 구조를 먼저 정리하는 것이 적합하다.
