# 4단계 텍스트 와이어 범위 결정

## 1. 문서 목적

이 문서는 BuildMap 4단계에서 확정한 것과 보류한 것을 기록한다.

4단계는 구현 전 화면 구조를 텍스트로 정리하는 단계다. 이번 단계의 목적은 각 화면에서 사용자가 실제로 무엇을 보고, 무엇을 입력하고, 어떤 버튼을 누르고, 다음 화면으로 어떻게 이동하는지를 명확히 하는 것이다.

## 2. 확정한 것

### 2.1 4단계는 화면별 텍스트 와이어 구조 작성 단계다

이번 단계에서는 실제 UI 코드를 작성하지 않는다. 화면의 목적, 사용자, 진입 조건, 표시 정보, 입력, 버튼, 상태, 다음 이동만 문서화한다.

### 2.2 DB 설계보다 화면 구조를 먼저 고정한다

사용자가 실제로 무엇을 보고 누르는지 아직 고정되지 않았으므로 DB 구조를 먼저 확정하지 않는다. 화면별 제품 구조가 먼저다.

### 2.3 AI 구조화 입력은 1차에서 텍스트 메모 하나로 시작한다

음성, 이미지, 파일 업로드, GitHub/Notion 연동은 후순위다. 1차에서는 Builder가 직접 작성한 텍스트 메모를 AI가 변화 카드 초안으로 구조화한다.

### 2.4 AI 초안은 Builder 승인 전까지 공식 Timeline에 반영하지 않는다

AI 초안은 자동 공개되지 않는다. Builder가 내용을 검토하고 수정한 뒤 승인해야 Decision Timeline에 공식 반영된다.

### 2.5 공개 프로젝트 페이지는 상단 요약 카드 + 하단 Timeline 구조를 기본으로 한다

처음부터 Timeline만 보여주면 읽기 어렵다. 따라서 상단에는 현재 필요한 것, 문제 정의, 현재 가설, 최근 전환을 카드형으로 요약하고, 하단에는 공개 변화 카드와 Decision Timeline을 둔다.

### 2.6 피드백 작성은 1차에서 비로그인 허용을 보류한다

무분별한 익명 피드백과 스팸성 응답을 줄이기 위해 1차에서는 최소 계정 또는 로그인 기반 피드백을 기본으로 가정한다.

### 2.7 Scout 탐색은 카드 그리드부터 시작한다

히트맵은 시각적으로 강점이 있지만 산식이 불안정하다. 1차에서는 프로젝트 카드 그리드로 시작하고, 히트맵은 후순위 실험 화면으로 둔다.

### 2.8 히트맵은 후순위 실험 화면으로 둔다

히트맵은 단순 인기 순위가 아니라 살아 움직이는 프로젝트를 발견하는 입구가 되어야 한다. 그러나 타일 크기와 색상 산식은 아직 확정하지 않는다.

### 2.9 Decision Diff는 단순 좌우 비교부터 시작한다

초기 Decision Diff는 초기 판단 vs 현재 판단, 전환 전 vs 전환 후를 좌우 비교하는 수준으로 시작한다.

## 3. 보류한 것

이번 단계에서 보류한 항목은 다음이다.

- DB 구조
- API 구조
- 상세 컴포넌트 설계
- 디자인 시스템
- AI 프롬프트 세부 설계
- 음성 입력
- 이미지/파일 업로드
- GitHub/Notion 연동
- 비로그인 피드백
- 히트맵 산식
- 채용 기능
- Project DNA
- 역량 점수화
- 결제

## 4. 추가 검토 필요

다음 항목은 다음 단계 이후 검토한다.

- 변화 카드 제품 필드를 어느 수준까지 실제 입력 항목으로 노출할지
- 문제 정의 변경 시 변화 카드 생성을 얼마나 강하게 유도할지
- 공개 프로젝트 페이지에서 “현재 필요한 것”과 “최근 전환” 중 무엇을 더 강조할지
- Scout가 Builder에게 직접 연락할 수 있는 조건
- 카드 그리드의 기본 정렬 기준
- Decision Diff를 공개 페이지에 포함할 기준

## 5. 다음 단계로 넘기는 결론

4단계 이후에는 다음 중 하나를 선택할 수 있다.

1. 화면별 입력 문구와 UX 카피를 더 구체화한다.
2. 데이터 구조 설계로 넘어간다.
3. 구현 범위를 1차 기준으로 잘라낸다.

다만 데이터 구조로 넘어가기 전에는 변화 카드, 문제 정의, 가설, 공개 상태, 피드백의 실제 화면 입력 구조가 현재 문서 수준으로 충분히 고정되었는지 확인해야 한다.
