# Problem Definition

## 핵심 문제 정의

BuildMap이 해결하려는 핵심 문제는 다음이다.

> 프로젝트의 결과물은 남지만, 그 결과물에 도달한 판단 흐름은 사라진다.

프로젝트는 코드, 문서, 대화, 피드백, 회고, 배포 링크 등 여러 형태의 기록을 남긴다. 하지만 시간이 지나면 “왜 그렇게 만들었는가”를 설명하는 맥락은 흩어지거나 사라진다.

BuildMap은 이 사라지는 맥락을 프로젝트의 중심 기록으로 되살리기 위해 존재한다.

## 프로젝트 기록이 흩어지는 문제

대부분의 프로젝트 기록은 하나의 장소에 모이지 않는다.

- 코드는 GitHub에 남는다.
- 기획 문서는 Notion이나 Google Docs에 남는다.
- 대화는 Slack, 카카오톡, Discord에 남는다.
- 피드백은 폼, 댓글, DM, 인터뷰 메모에 흩어진다.
- 배포 결과는 별도 링크나 서비스 페이지에 남는다.

이 기록들은 각각 의미가 있지만, 프로젝트의 변화 흐름으로 연결되지는 않는다.

결과적으로 나중에 프로젝트를 다시 보거나 외부에 설명하려 할 때, Builder는 기록을 다시 찾아 엮어야 한다. 이 과정에서 많은 맥락이 누락된다.

## 의사결정 맥락이 사라지는 문제

프로젝트에서 중요한 것은 단순히 “무엇을 했다”가 아니다. 더 중요한 것은 “왜 그렇게 판단했는가”다.

하지만 일반적인 기록 방식은 의사결정의 맥락을 충분히 남기지 못한다.

- 기능을 추가한 이유
- 기능을 삭제한 이유
- 가설을 바꾼 이유
- 사용자 피드백을 반영하거나 무시한 이유
- 특정 문제를 우선순위에 둔 이유
- 현재 방향을 유지하는 이유

이 맥락이 사라지면 프로젝트는 결과물만 남고, Builder의 사고 과정은 보이지 않는다.

BuildMap은 의사결정을 하나의 기록 단위로 다루어 이 문제를 해결하려 한다.

## 비개발자가 프로젝트 변화를 이해하기 어려운 문제

현재 프로젝트 기록의 많은 부분은 개발자 중심으로 남는다.

GitHub의 커밋, 브랜치, Pull Request, 이슈는 개발자에게는 익숙하지만, 비개발자에게는 프로젝트의 실제 변화를 이해하기 어려운 형식일 수 있다.

비개발자가 알고 싶은 것은 파일 변경이 아니라 다음과 같은 내용이다.

- 이 프로젝트는 처음에 어떤 문제에서 시작했는가
- 어떤 시도를 해보았는가
- 사용자는 어떤 반응을 보였는가
- 왜 기능이나 방향이 바뀌었는가
- 지금은 무엇이 필요한가

BuildMap은 개발자만 읽을 수 있는 변경 로그가 아니라, 비개발자도 이해할 수 있는 프로젝트 변화 지도를 지향한다.

## 새 팀원이나 외부 협업자가 프로젝트 맥락을 이해하기 어려운 문제

프로젝트에 새 팀원, 협업자, 멘토, 테스터가 합류할 때 가장 큰 비용은 맥락 파악이다.

기존 기록을 모두 읽어도 다음 질문에 답하기 어렵다.

- 이 결정은 왜 내려졌는가
- 이전에 이미 시도했다가 실패한 것은 무엇인가
- 현재 방향은 어떤 피드백 위에 서 있는가
- 절대 건드리면 안 되는 판단은 무엇인가
- 다시 검토해도 되는 판단은 무엇인가

BuildMap은 프로젝트 인수인계의 중요한 기반이 될 수 있다.

단, 프로젝트 인수인계의 상세 화면, 권한, 협업 기능은 이번 단계에서 확정하지 않는다. 추가 검토 필요 항목으로 남긴다.

## Builder가 자신의 성장과 판단 과정을 증명하기 어려운 문제

이력서와 포트폴리오는 결과를 보여주기 쉽다. 하지만 Builder가 실제로 어떤 판단을 했고, 어떻게 배웠고, 어떤 기준으로 프로젝트를 성장시켰는지는 보여주기 어렵다.

특히 학생 메이커, 비개발자, 바이브코딩 유저, 사이드프로젝트 제작자는 코드만으로 자신의 역량을 설명하기 어렵다.

BuildMap은 사람을 직접 평가하기보다, 프로젝트의 변화 과정이 Builder를 증명하게 하는 방향을 지향한다.

이번 단계에서는 역량 평가, 점수화, 채용 기능은 다루지 않는다. BuildMap은 먼저 프로젝트의 판단 흐름을 정확히 남기는 데 집중한다.
