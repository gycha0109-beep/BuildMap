# Core Concepts

## Builder

Builder는 프로젝트를 만든 사람이 아니라 프로젝트를 성장시키는 사람이다.

Builder에는 다음이 포함될 수 있다.

- 창업자
- 학생 메이커
- 개발자
- 디자이너
- PM
- 비개발자
- 바이브코딩 유저
- 사이드프로젝트 제작자

Builder는 프로젝트의 결과물만 보여주지 않는다. Builder는 프로젝트가 어떤 문제에서 시작했고, 어떤 가설과 실험을 거쳐 변화했는지 기록한다.

Builder의 핵심 행동은 다음이다.

- 판단 기록
- 피드백 요청
- 테스터 모집
- 협업자 탐색
- 프로젝트 맥락 공유
- 프로젝트 인수인계 준비

## Scout

Scout는 프로젝트와 Builder를 발견하는 사람이다.

Scout는 단순 채용 담당자가 아니다. 초기에는 다음 역할이 더 중요할 수 있다.

- 협업자
- 초기 사용자
- 테스터
- 멘토
- 창업자
- 팀원 탐색자
- 커뮤니티 운영자

장기적으로는 헤드헌터와 채용 담당자도 Scout에 포함될 수 있다.

Scout는 이력서보다 Builder의 프로젝트 진화 과정과 판단 기록을 보고 사람과 프로젝트를 발견한다.

## 프로젝트

BuildMap에서 프로젝트는 단순 결과물이 아니다.

프로젝트는 문제 정의, 가설, 실험, 피드백, 판단, 방향 전환이 누적되는 살아 있는 단위다.

BuildMap에서 프로젝트는 다음 질문에 답할 수 있어야 한다.

- 이 프로젝트는 어떤 문제에서 시작했는가
- 어떤 판단을 거쳐 현재 모습이 되었는가
- 무엇을 시도했고 무엇을 배웠는가
- 현재 무엇이 필요한가

## Decision Timeline

Decision Timeline은 BuildMap의 중심 구조다.

Decision Timeline은 프로젝트의 판단 흐름을 시간 순서대로 보여준다. 단순 작업 기록이나 코드 변경 목록이 아니라, 프로젝트가 왜 변화했는지를 보여주는 구조다.

Decision Timeline에는 다음 흐름이 포함될 수 있다.

```text
문제 발견 → 가설 → 실험 → 피드백 → 판단 → 방향 전환 → 현재 상태
```

BuildMap에서 중요한 것은 현재 버전 자체가 아니다. 현재 버전은 Decision Timeline의 결과일 뿐이다.

## 변화 카드

변화 카드는 프로젝트에서 발생한 하나의 의사결정 단위다.

변화 카드는 단순 로그가 아니다. 변화 카드는 프로젝트의 판단이 발생한 지점을 기록한다.

변화 카드의 유형은 다음과 같을 수 있다.

- 문제 발견
- 가설 생성
- 실험
- 사용자 피드백
- 기능 추가
- 기능 제거
- 판단 유지
- 판단 수정
- 방향 전환
- 릴리즈
- 인수인계 메모

변화 카드의 상세 필드와 입력 방식은 이번 단계에서 확정하지 않는다. 추가 검토 필요 항목으로 남긴다.

## 프로젝트 진화도

프로젝트 진화도는 프로젝트가 어떤 판단을 거쳐 변화했는지 한눈에 이해할 수 있는 지도다.

프로젝트 진화도는 완성된 포트폴리오가 아니다. 프로젝트가 살아 움직이는 동안의 변화 경로를 보여준다.

프로젝트 진화도는 다음을 드러내야 한다.

- 문제의 변화
- 가설의 변화
- 실험 결과
- 피드백 반영 여부
- 기능의 추가와 제거
- 방향 전환의 이유
- 현재 필요한 도움

## Decision Diff

Decision Diff는 단순 코드 diff가 아니라 판단의 차이를 의미한다.

예를 들어 이전 판단과 현재 판단이 다음처럼 달라졌다면, BuildMap은 그 차이를 기록해야 한다.

- 이전에는 A 문제를 핵심으로 보았지만, 지금은 B 문제를 더 중요하게 본다.
- 이전에는 기능 추가가 필요하다고 판단했지만, 지금은 기능 제거가 더 중요하다고 판단한다.
- 이전에는 특정 사용자군을 우선했지만, 피드백 이후 다른 사용자군을 우선한다.

Decision Diff는 “무엇이 바뀌었는가”와 “왜 바뀌었는가”를 함께 보여준다.

Decision Diff의 구체적 UI와 데이터 구조는 이번 단계에서 확정하지 않는다. 추가 검토 필요 항목으로 남긴다.

## 현재 필요한 것

BuildMap의 프로젝트는 현재 필요한 것을 명확히 표현할 수 있어야 한다.

예시는 다음과 같다.

- 피드백 필요
- 테스터 필요
- 협업자 필요
- 멘토 필요
- 사용자 인터뷰 필요
- 디자인 검토 필요
- 기술 검토 필요
- 인수인계 대상 필요

현재 필요한 것은 프로젝트를 발견한 Scout가 어떻게 참여할 수 있는지 알려주는 신호다.

## 프로젝트 인수인계

프로젝트 인수인계는 BuildMap의 중요한 유즈케이스 후보다.

프로젝트가 중단되거나 팀원이 바뀌거나 새로운 협업자가 들어올 때, 필요한 것은 단순한 파일 목록이 아니다. 필요한 것은 판단의 맥락이다.

BuildMap은 다음 질문에 답하는 인수인계 기반이 될 수 있다.

- 왜 이 방향으로 만들었는가
- 이전에 실패한 시도는 무엇인가
- 유지해야 할 판단은 무엇인가
- 다시 검토 가능한 판단은 무엇인가
- 현재 막힌 지점은 어디인가

프로젝트 인수인계의 상세 기능은 이번 단계에서 확정하지 않는다.

## 공개 프로젝트 페이지

공개 프로젝트 페이지는 외부인이 프로젝트의 판단 흐름을 이해하는 진입점이다.

공개 프로젝트 페이지는 단순 소개 페이지가 아니다. 프로젝트의 현재 상태와 그 상태에 도달한 이유를 함께 보여주는 페이지다.

공개 프로젝트 페이지는 다음을 포함할 수 있다.

- 프로젝트 한 줄 설명
- 현재 상태
- Decision Timeline
- 주요 변화 카드
- 현재 필요한 것
- Builder 정보
