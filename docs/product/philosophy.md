# BuildMap Philosophy

## BuildMap의 철학

BuildMap의 철학은 단순하다.

> 프로젝트는 결과물만으로 설명되지 않는다. 프로젝트는 그 결과물에 도달한 판단의 흐름으로 설명된다.

많은 프로젝트는 완성된 화면, 배포 링크, 코드 저장소, 소개 글로 평가된다. 그러나 실제 프로젝트를 성장시키는 것은 결과물 이전의 판단이다.

- 어떤 문제를 진짜 문제로 보았는가
- 어떤 가설을 세웠는가
- 무엇을 실험했는가
- 어떤 피드백을 받았는가
- 어떤 기능을 유지했는가
- 어떤 기능을 삭제했는가
- 왜 방향을 바꾸었는가

BuildMap은 이 흐름을 프로젝트의 핵심 기록으로 본다.

## 코드 변경이 아니라 판단 흐름을 기록한다는 의미

GitHub의 커밋 로그는 코드가 어떻게 바뀌었는지를 보여준다. 하지만 코드 변경만으로는 프로젝트의 의도를 충분히 이해하기 어렵다.

예를 들어 어떤 기능이 제거되었다고 해도, 커밋 로그만으로는 다음 질문에 답하기 어렵다.

- 사용자가 그 기능을 이해하지 못했는가
- 운영 비용이 너무 높았는가
- 핵심 문제와 맞지 않았는가
- 더 중요한 가설이 발견되었는가
- 단순한 임시 제거인가, 방향 전환인가

BuildMap이 기록하려는 것은 파일의 변화가 아니라 **판단의 변화**다.

BuildMap에서 중요한 것은 “무엇이 바뀌었는가”보다 “왜 바뀌었는가”다.

## 왜 정리글, 블로그, GitHub, Notion만으로 부족한가

### 정리글과 블로그의 한계

정리글과 블로그는 보통 사건이 지난 뒤 작성된다. 그래서 흐름이 매끄럽게 정리되는 대신, 실제 고민과 중간 판단이 사라지기 쉽다.

블로그는 결과를 설명하기 좋지만 프로젝트가 살아 움직이는 동안의 판단 단위를 지속적으로 쌓는 구조는 아니다.

### GitHub의 한계

GitHub는 코드 중심 도구다. 개발자에게는 강력하지만, 비개발자가 프로젝트의 판단 흐름을 이해하기에는 진입 장벽이 높다.

커밋 메시지와 Pull Request 설명만으로는 사용자 피드백, 기능 삭제 이유, 방향 전환의 맥락이 충분히 드러나지 않는다.

### Notion의 한계

Notion은 자유도가 높다. 하지만 자유도가 높기 때문에 프로젝트의 변화 흐름을 일관된 구조로 남기기 어렵다.

문서는 쌓이지만, 시간이 지나면 어떤 문서가 현재 판단과 연결되는지 파악하기 어려워진다.

### 대화 도구의 한계

Slack, 카카오톡, Discord에는 실제 의사결정의 힌트가 많이 남는다. 그러나 대화는 빠르게 묻히고, 검색 가능한 기록이 되더라도 프로젝트 변화의 구조로 정리되지는 않는다.

## 프로젝트 운영체제라는 표현의 의미

BuildMap을 프로젝트 운영체제라고 부르는 이유는 단순 기록장이 아니라 프로젝트의 판단 흐름을 운영하기 위한 기준점을 제공하기 때문이다.

BuildMap은 다음을 연결한다.

- 문제 정의
- 가설
- 실험
- 피드백
- 판단
- 방향 전환
- 현재 필요한 도움

이 연결은 프로젝트를 외부에 설명하기 위한 용도만이 아니다. Builder 자신이 프로젝트를 잃어버리지 않기 위한 운영 기준이기도 하다.

단, 이번 단계에서 “운영체제”는 제품 기능 범위를 확정하는 표현이 아니다. 상세 기능과 화면은 추가 검토 필요 항목으로 남긴다.

## 실패, 삭제, 방향 전환이 중요한 이유

일반적인 포트폴리오는 성공한 결과를 보여준다. 하지만 실제 프로젝트 성장에는 실패, 삭제, 방향 전환이 더 많은 정보를 담고 있다.

- 실패한 실험은 가설이 틀렸음을 증명한다.
- 삭제한 기능은 무엇을 중요하지 않다고 판단했는지 보여준다.
- 방향 전환은 프로젝트가 배운 것을 드러낸다.
- 유지한 판단은 프로젝트의 기준을 보여준다.

BuildMap은 실패를 감추지 않는다. 실패를 프로젝트의 판단 자산으로 바꾼다.

삭제와 전환은 부끄러운 기록이 아니라, Builder가 무엇을 배웠고 어떻게 프로젝트를 성장시켰는지를 보여주는 핵심 증거다.
