# 변화 카드 작성 화면과 흐름

## 1. 문서 목적

이 문서는 BuildMap의 핵심 화면인 변화 카드 작성 흐름을 정리한다.

BuildMap의 차별점은 변화 카드가 실제 의사결정 단위로 작동하는지에 달려 있다. 변화 카드는 단순 작업 로그, 회고 글, 업데이트 노트가 아니다. 변화 카드는 프로젝트에서 발생한 판단을 문제, 근거, 판단, 변경, 다음 확인 사항으로 구조화한 기록이다.

## 2. 전체 흐름

```text
Builder가 거친 메모 입력
→ AI가 변화 카드 초안 생성
→ Builder가 초안 검토
→ Builder가 유형 선택 또는 수정
→ 문제/가설/실험/피드백과 연결
→ 공개 여부 선택
→ 중요도 선택
→ 승인
→ Decision Timeline 반영
```

AI는 생성보다 구조화에 집중한다. AI 초안은 자동 공개되지 않으며, Builder가 승인하기 전까지 공식 Decision Timeline에 반영되지 않는다.

## 3. 변화 카드 최소 정보

초기 제품 설계에서 변화 카드 최소 정보는 다음으로 가정한다.

| 항목 | 의미 |
|---|---|
| 유형 | 문제 발견, 가설 반박, 기능 제거 등 의사결정 유형 |
| 제목 | 변화 카드의 짧은 이름 |
| 원문 메모 | Builder가 입력한 거친 원문 |
| 구조화 요약 | AI 또는 Builder가 정리한 핵심 요약 |
| 근거 | 피드백, 테스트, 관찰, 회의 등 판단의 근거 |
| 판단 | Builder가 내린 결정 |
| 변경 내용 | 실제로 바꾸거나 유지한 내용 |
| 다음 확인 사항 | 다음에 검증하거나 관찰할 질문 |
| 연결된 문제 또는 가설 | 이 카드가 어떤 문제/가설과 연결되는지 |
| 공개 여부 | 내부 전용, 공개 가능, 공개됨 등 |
| 상태 | 초안, 승인, 보류 |
| 중요도 | 일반, 핵심 전환 |

이 목록은 제품 설계용 최소 필드다. DB 스키마로 확정하지 않는다.

## 4. 변화 카드 유형 예시

변화 카드 유형은 다음을 포함할 수 있다.

- 문제 발견
- 문제 정의 수정
- 가설 생성
- 가설 반박
- 실험
- 사용자 피드백
- 기능 추가
- 기능 제거
- 판단 유지
- 판단 수정
- 방향 전환
- 릴리즈
- 인수인계 메모

유형은 Builder가 직접 선택하거나 AI가 초안에서 제안할 수 있다. 최종 유형은 Builder가 승인한다.

## 5. 단계별 화면 흐름

## 5.1 거친 메모 입력 화면

### 화면 목적

Builder가 회의, 사용자 테스트, 작업 중에 생긴 판단의 원재료를 빠르게 남기는 화면이다.

### 주요 사용자

- Builder

### 핵심 정보

- 이 메모는 바로 공개되지 않는다는 안내
- 짧고 거칠게 써도 된다는 안내
- 텍스트 메모 입력란
- 관련 프로젝트 선택
- 관련 문제/가설 선택 가능

### 핵심 행동

- 텍스트 메모 입력
- 저장
- AI 구조화 요청

### 전환 경로

- 저장 후 AI 초안 생성 화면
- 저장만 하고 나중에 구조화

### 초기 구현 범위

- 텍스트 입력
- 프로젝트 연결
- AI 구조화 버튼

### 후순위 요소

- 음성 입력
- 이미지 첨부
- 회의록 업로드
- 외부 문서 업로드

## 5.2 AI 초안 생성 화면

### 화면 목적

거친 메모를 변화 카드 초안으로 구조화한다.

### 주요 사용자

- Builder

### 핵심 정보

- AI는 내용을 꾸미지 않는다는 안내
- AI는 없는 성과를 만들지 않는다는 안내
- AI는 원문을 문제, 근거, 판단, 변경, 다음 확인 사항으로 구조화한다는 안내
- 생성 중 상태

### 핵심 행동

- AI 구조화 실행
- 생성 실패 시 다시 시도 또는 직접 작성

### 전환 경로

- 성공 시 Builder 검토 화면
- 실패 시 직접 작성 화면 또는 재시도

### 초기 구현 범위

- 단일 메모 기반 AI 구조화
- 생성 실패 메시지

### 후순위 요소

- 여러 메모 묶음 구조화
- 프로젝트 전체 맥락 기반 구조화
- 복수 초안 비교

## 5.3 Builder 초안 검토 화면

### 화면 목적

AI 초안을 공식 변화 카드로 승인하기 전에 Builder가 의미와 근거를 검토한다.

### 주요 사용자

- Builder

### 핵심 정보

- 원문 메모
- AI 구조화 결과
- 유형 제안
- 근거
- 판단
- 변경 내용
- 다음 확인 사항
- 연결된 문제/가설

### 핵심 행동

- 제목 수정
- 유형 수정
- 요약 수정
- 근거 확인
- 판단 문장 수정
- 연결 대상 선택
- 보류 또는 승인

### 전환 경로

- 승인 전 상태로 이동
- 보류 상태로 저장
- 직접 편집 계속

### 초기 구현 범위

- 주요 필드 수정
- 유형 선택
- 연결된 문제/가설 선택

### 후순위 요소

- 변경 이력 표시
- 팀원 코멘트
- AI 재구조화 요청

## 5.4 유형 선택 또는 수정 화면

### 화면 목적

변화 카드가 어떤 의사결정 단위인지 명확히 한다.

### 주요 사용자

- Builder

### 핵심 정보

- AI가 제안한 유형
- 선택 가능한 유형 목록
- 유형별 설명

### 핵심 행동

- 유형 선택
- 유형 수정
- 복수 유형 허용 여부는 추가 검토 필요

### 전환 경로

- 문제/가설/피드백 연결 단계

### 초기 구현 범위

- 단일 유형 선택

### 후순위 요소

- 복수 유형 태그
- 유형별 템플릿
- 유형별 공개 표시 방식

## 5.5 문제/가설/실험/피드백 연결 화면

### 화면 목적

변화 카드가 단절된 메모가 아니라 프로젝트 판단 흐름의 일부가 되도록 연결한다.

### 주요 사용자

- Builder

### 핵심 정보

- 현재 문제 정의
- 현재 가설 목록
- 최근 피드백
- 최근 실험 또는 변경

### 핵심 행동

- 연결된 문제 선택
- 연결된 가설 선택
- 연결된 피드백 선택 가능
- 새 문제/가설 생성 가능 여부는 추가 검토 필요

### 전환 경로

- 공개 여부 선택

### 초기 구현 범위

- 연결된 문제 또는 가설 선택

### 후순위 요소

- 실험 객체 연결
- 피드백 묶음 연결
- 연결 그래프 보기

## 5.6 공개 여부 선택 화면

### 화면 목적

승인된 변화 카드가 외부에 공개될지 결정한다.

### 주요 사용자

- Builder

### 핵심 정보

- 승인과 공개는 별개의 상태라는 안내
- 공개 가능한 정보만 노출된다는 안내
- 민감 정보가 있는 경우 공개 제외 가능

### 핵심 행동

- 내부 전용 선택
- 공개 가능 선택
- 공개됨 선택
- 민감 정보 포함 표시 가능

### 전환 경로

- 중요도 선택 또는 승인 화면

### 초기 구현 범위

- 내부 전용/공개 가능/공개됨 상태

### 후순위 요소

- 카드별 공개 범위 세분화
- 링크 공개 카드
- 특정 Scout에게만 공유

## 5.7 중요도 선택 화면

### 화면 목적

프로젝트 진화도와 Decision Timeline에서 어떤 카드가 핵심 전환점인지 구분한다.

### 주요 사용자

- Builder

### 핵심 정보

- 일반 카드
- 핵심 전환 카드
- 핵심 전환 카드는 프로젝트 진화도에서 더 강조될 수 있다는 안내

### 핵심 행동

- 중요도 선택

### 전환 경로

- 승인 화면

### 초기 구현 범위

- 일반 / 핵심 전환

### 후순위 요소

- 중요도 단계 세분화
- 자동 전환점 추천
- 요약 카드 생성

## 5.8 승인 화면

### 화면 목적

Builder가 변화 카드를 공식 기록으로 확정한다.

### 주요 사용자

- Builder

### 핵심 정보

- 승인 후 Decision Timeline에 반영된다는 안내
- 공개 여부 최종 확인
- 연결된 문제/가설 확인
- AI 초안이 아닌 Builder 승인 기록이라는 표시

### 핵심 행동

- 승인
- 보류
- 다시 수정

### 전환 경로

- 승인 시 Decision Timeline 화면
- 보류 시 초안 목록

### 초기 구현 범위

- 승인 / 보류 / 수정

### 후순위 요소

- 팀 승인
- 승인 전 체크리스트
- 승인 후 공유 메시지 생성

## 6. 화면 상태

## 6.1 빈 상태

변화 카드가 아직 없을 때 표시한다.

표시할 메시지:

> 아직 변화 카드가 없습니다. 사용자 피드백, 회의 내용, 기능 변경 이유를 거친 메모로 남겨보세요.

핵심 CTA:

- 거친 메모 입력

## 6.2 AI 초안 생성 중

AI가 메모를 구조화하는 상태다.

주의할 점:

- 생성 중에도 “자동 공개되지 않음”을 명시한다.
- 긴 대기 시간이 있을 경우 직접 작성 대안을 제공한다.

## 6.3 AI 초안 생성 실패

AI 구조화가 실패한 상태다.

제공할 행동:

- 다시 시도
- 직접 변화 카드 작성
- 원문 메모만 저장

## 6.4 Builder 수정 중

Builder가 AI 초안을 수정하는 상태다.

주의할 점:

- AI 문장을 그대로 확정하지 않아도 된다는 안내가 필요하다.
- 근거와 판단을 구분해서 보여줘야 한다.

## 6.5 승인 전

공식 Timeline 반영 전 상태다.

표시할 정보:

- 상태: 초안
- 공개 상태: 공개되지 않음
- 승인 필요

## 6.6 승인 후

Decision Timeline에 반영된 상태다.

표시할 정보:

- 상태: 승인됨
- Timeline 반영 위치
- 공개 상태

## 6.7 공개 제외 상태

내부 기록으로만 유지되는 상태다.

표시할 정보:

- 공개 제외 사유 선택 가능
- 내부 Decision Timeline에는 유지됨

## 6.8 공개 상태

공개 프로젝트 페이지에 노출되는 상태다.

표시할 정보:

- 공개됨
- 공개 페이지에서 보이는 방식 미리보기
- 민감 정보 검토 안내

## 7. AI 관련 주의점

AI 화면 전반에서 다음 원칙을 유지한다.

- AI는 기록을 꾸미지 않는다.
- AI는 없는 성과를 만들지 않는다.
- AI는 Builder의 입력을 구조화한다.
- Builder가 최종 승인한다.
- AI 초안은 자동 공개되지 않는다.
- 승인 전 변화 카드는 공식 Decision Timeline에 반영되지 않는다.

## 8. Decision Timeline 반영 방식

승인된 변화 카드는 다음 방식으로 Timeline에 반영된다.

- 시간 순서에 배치된다.
- 연결된 문제 또는 가설 아래에 묶일 수 있다.
- 핵심 전환 카드는 강조된다.
- 공개 상태에 따라 외부 노출 여부가 달라진다.
- 낮은 중요도 카드는 접히거나 요약될 수 있다.

## 9. 추가 검토 필요

- 변화 카드 최종 필드 수
- 복수 유형 선택 허용 여부
- 실험/피드백 객체를 별도 화면으로 분리할지
- 중요도 자동 추천 여부
- AI 초안의 원문 대비 변경 표시 방식
- 공개 전 민감 정보 감지 방식
