# 프로젝트 생성 흐름

## 1. 문서 목적

이 문서는 Builder가 첫 프로젝트를 생성하는 화면 흐름을 정리한다.

BuildMap에서 프로젝트 생성은 완성작 등록이 아니다. 프로젝트 생성은 판단 흐름을 쌓기 시작하는 행위다.

따라서 프로젝트 생성 화면은 긴 기획서, 완성된 소개글, 포트폴리오용 문장을 요구하지 않는다. 비개발자와 바이브코딩 유저도 짧은 문제의식과 현재 상태만으로 시작할 수 있어야 한다.

## 2. 전체 생성 흐름

```text
프로젝트명 입력
→ 한 줄 설명 입력
→ 현재 상태 선택
→ 해결하려는 문제 입력
→ 아직 확정되지 않은 가설 입력
→ 현재 필요한 도움 입력
→ 공개 여부 선택
→ 첫 프로젝트 대시보드 진입
```

## 3. 단계별 화면 설계

## 3.1 프로젝트명 입력

### 필수 입력

- 프로젝트명

### 선택 입력

- 임시 프로젝트명 여부
- 별칭 또는 가칭 표시

### 도움말 문구

> 완성된 브랜드명이 아니어도 됩니다. 지금 부르는 이름으로 시작하세요.

### 잘못 설계하면 생길 문제

- 처음부터 완성된 서비스명을 요구하면 시작 장벽이 생긴다.
- 프로젝트명을 브랜드 심사처럼 만들면 BuildMap이 포트폴리오 등록 서비스처럼 보인다.

### 초기 구현 범위

- 프로젝트명 입력
- 나중에 수정 가능 안내

### 후순위 범위

- 프로젝트명 변경 이력
- 공개 URL 자동 생성
- 중복 이름 처리 정책

## 3.2 한 줄 설명 입력

### 필수 입력

- 한 줄 설명

### 선택 입력

- 대상 사용자
- 프로젝트 분야

### 도움말 문구

> 지금 만들고 있는 것이 무엇인지 한 문장으로 적습니다. 정확하지 않아도 괜찮습니다.

예시:

> 프로젝트의 판단 흐름을 기록하고 공유하는 Decision Timeline 플랫폼

### 잘못 설계하면 생길 문제

- 홍보 문구처럼 작성하게 만들면 실제 문제의식이 가려진다.
- 너무 멋진 문장을 요구하면 AI가 과장된 소개글을 생성하는 방향으로 흐를 수 있다.

### 초기 구현 범위

- 한 줄 설명 입력
- 예시 문구 제공

### 후순위 범위

- 공개 페이지용 요약 문구 별도 관리
- AI 문장 다듬기 기능

## 3.3 현재 상태 선택

### 필수 입력

- 현재 상태

### 선택 가능한 상태 예시

- 아이디어
- 문제 정의 중
- 가설 검증 중
- 초기 제작 중
- 테스트 중
- 공개 전
- 공개됨
- 방향 전환 중
- 보류 중

### 선택 입력

- 상태 설명
- 최근 변경 요약

### 도움말 문구

> 현재 상태는 프로젝트의 완성도가 아니라 지금 어떤 판단 단계에 있는지를 나타냅니다.

### 잘못 설계하면 생길 문제

- 상태가 제품 완성도처럼 보이면 Builder가 미완성 프로젝트를 올리기 부담스러워진다.
- “출시됨/미출시”만 있으면 BuildMap의 판단 흐름 중심성이 약해진다.

### 초기 구현 범위

- 단일 상태 선택
- 상태는 나중에 변경 가능

### 후순위 범위

- 상태 변경을 변화 카드로 연결
- 상태별 추천 다음 행동

## 3.4 해결하려는 문제 입력

### 필수 입력

- 해결하려는 문제

### 선택 입력

- 문제를 발견한 상황
- 문제를 겪는 사용자
- 아직 확실하지 않은 부분

### 도움말 문구

> 솔루션보다 먼저, 어떤 문제가 반복해서 보였는지 적습니다.

예시:

> 사람들은 장기 목표를 세우지만 이번 주에 무엇을 증명해야 하는지 몰라 실행이 끊긴다.

### 잘못 설계하면 생길 문제

- 기능 설명을 문제 정의로 착각할 수 있다.
- 처음부터 완벽한 문제 정의를 요구하면 사용자가 시작하지 못한다.

### 초기 구현 범위

- 문제 정의 텍스트 입력
- “나중에 수정 가능” 안내

### 후순위 범위

- 문제 정의 변경 시 변화 카드 자동 제안
- 문제 정의 개선 질문
- 문제 정의 버전 비교

## 3.5 아직 확정되지 않은 가설 입력

### 필수 입력

- 없음. 초기에는 선택 입력으로 둔다.

### 선택 입력

- 핵심 가설
- 가설 상태
- 연결된 문제 정의

### 도움말 문구

> 가설은 기능 목록이 아니라 프로젝트가 맞다고 믿고 있는 판단입니다.

예시:

> 사용자는 긴 목표 계획보다 이번 주에 확인할 수 있는 증거 단위를 더 쉽게 이해한다.

### 잘못 설계하면 생길 문제

- 프로젝트 생성 시 가설을 필수로 강제하면 진입 장벽이 높아진다.
- 기능 목록 입력란처럼 보이면 BuildMap의 철학과 어긋난다.

### 초기 구현 범위

- 가설 0개 또는 1개 입력 허용
- 상태 기본값은 `가정` 또는 `검증 중` 중 선택 가능

### 후순위 범위

- 여러 가설 관리
- 가설별 실험 연결
- 가설 상태 변경 Timeline 반영

## 3.6 현재 필요한 도움 입력

### 필수 입력

- 현재 필요한 도움 또는 “아직 없음” 선택

### 선택 가능한 예시

- 문제 정의 피드백
- 첫 화면 이해도 피드백
- 테스터 필요
- 사용자 인터뷰 필요
- 디자이너 협업 필요
- 개발자 협업 필요
- 멘토 의견 필요
- 아직 공개 도움 요청 없음

### 도움말 문구

> 지금 프로젝트가 다음으로 확인해야 하는 것을 적습니다. 완성된 프로젝트가 아니어도 도움을 요청할 수 있습니다.

### 잘못 설계하면 생길 문제

- “문의하기” 수준으로 보이면 프로젝트 맥락 기반 연결이 약해진다.
- 너무 많은 선택지를 주면 사용자가 현재 필요한 것을 명확히 하지 못한다.

### 초기 구현 범위

- 텍스트 입력 또는 간단한 선택지
- 공개 페이지 상단에 노출 가능

### 후순위 범위

- 도움 요청 유형별 매칭
- 테스터 모집 양식
- 협업자 요청 관리

## 3.7 공개 여부 선택

### 필수 입력

- 프로젝트 공개 상태

### 선택 가능한 공개 상태

- 비공개
- 링크 공개
- 전체 공개

### 선택 입력

- 공개할 변화 카드 선택은 프로젝트 생성 이후에 진행

### 도움말 문구

> 기본은 내부 비공개입니다. 공개 가능한 내용만 나중에 선택해서 공개할 수 있습니다.

### 잘못 설계하면 생길 문제

- 처음부터 공개를 강제하면 Builder가 거친 판단 기록을 남기기 어려워진다.
- 공개와 승인 상태를 혼동하면 AI 초안이 외부에 노출될 위험이 생긴다.

### 초기 구현 범위

- 프로젝트 단위 공개 상태 선택
- 기본값은 비공개

### 후순위 범위

- 변화 카드별 세부 공개 제어
- 팀 단위 공개 승인
- 민감 정보 감지

## 3.8 첫 프로젝트 대시보드 진입

### 필수 결과

프로젝트 생성이 완료되면 Builder는 프로젝트 대시보드로 이동한다.

### 화면에서 보여야 할 정보

- 프로젝트명
- 한 줄 설명
- 현재 문제 정의
- 현재 가설
- 현재 필요한 도움
- 첫 변화 카드 작성 유도
- 거친 메모 입력 버튼
- Decision Timeline 빈 상태

### 도움말 문구

> 아직 Timeline이 비어 있습니다. 회의, 테스트, 피드백, 판단 변경이 생기면 거친 메모로 남겨보세요.

### 잘못 설계하면 생길 문제

- 빈 대시보드가 실패처럼 보이면 사용자가 이탈한다.
- 바로 많은 기능을 보여주면 첫 기록 행동이 흐려진다.

### 초기 구현 범위

- 프로젝트 요약
- 거친 메모 입력 CTA
- 문제/가설 편집 진입
- 빈 Decision Timeline 안내

### 후순위 범위

- 추천 다음 행동
- 공개 페이지 미리보기
- 피드백 요청 템플릿

## 4. 초기 구현 기준

초기 프로젝트 생성 흐름은 다음을 우선한다.

- 프로젝트명
- 한 줄 설명
- 현재 상태
- 해결하려는 문제
- 아직 확정되지 않은 가설
- 현재 필요한 도움
- 공개 여부
- 프로젝트 대시보드 진입

이 단계에서 긴 문서 작성, 상세 기획서, 화면 설계, DB 구조 입력, 팀 권한 설정은 요구하지 않는다.

## 5. 추가 검토 필요

- 가설 입력을 필수로 할지 선택으로 둘지
- 프로젝트 공개 상태의 기본값과 설명 방식
- 현재 필요한 도움의 선택지 개수
- 프로젝트 생성 후 첫 화면이 대시보드인지 거친 메모 입력인지
- 공개 URL 생성 시점
