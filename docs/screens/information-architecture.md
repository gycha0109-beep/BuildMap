# BuildMap 정보 구조

## 1. 문서 목적

이 문서는 BuildMap의 제품 관점 정보 구조를 정리한다.

이 문서는 DB 스키마가 아니다. 테이블, 컬럼, 인덱스, RLS, API 구조를 확정하지 않는다. 여기서 정리하는 것은 사용자가 어떤 정보를 만들고, 어디에서 읽고, 어떤 흐름으로 공개하는지에 대한 제품 구조다.

## 2. 전체 정보 구조

BuildMap의 정보 구조는 하나의 중심 흐름으로 정리된다.

```text
Builder
→ 프로젝트
→ 문제 정의 / 가설
→ 거친 메모
→ AI 변화 카드 초안
→ Builder 승인
→ 변화 카드
→ Decision Timeline
→ 프로젝트 진화도
→ 공개 프로젝트 페이지
→ Scout/외부 사용자 피드백
→ 새 변화 카드
```

이 구조에서 중심은 프로젝트의 현재 버전이 아니다. 현재 버전은 Decision Timeline의 결과다.

BuildMap이 저장하고 보여주려는 핵심은 다음이다.

- 어떤 문제에서 시작했는가
- 어떤 가설을 믿었는가
- 무엇을 실험했는가
- 어떤 피드백을 받았는가
- 어떤 판단을 유지하거나 수정했는가
- 무엇을 추가하거나 제거했는가
- 왜 방향을 바꾸었는가
- 지금 어떤 도움이나 검증이 필요한가

## 3. 주요 영역

### 3.1 Builder 내부 작업 공간

Builder가 프로젝트의 판단 흐름을 쌓는 내부 공간이다.

주요 역할은 다음이다.

- 프로젝트 생성
- 문제 정의 입력 및 수정
- 가설 입력 및 상태 관리
- 거친 메모 입력
- AI 변화 카드 초안 확인
- 변화 카드 수정 및 승인
- 공개 여부 선택
- 피드백 요청 작성

이 영역은 기본적으로 내부 비공개다. 공개 가능한 변화 카드만 Builder의 승인 후 공개 프로젝트 페이지에 노출된다.

### 3.2 프로젝트 상세/관리 영역

Builder가 특정 프로젝트를 운영하는 중심 화면이다.

주요 정보는 다음이다.

- 현재 문제 정의
- 현재 가설
- 최근 변화 카드
- 미해결 가설
- 피드백 요청
- 현재 필요한 도움
- Decision Timeline
- 공개 전환 상태

프로젝트 상세/관리 영역은 단순 관리 페이지가 아니다. Builder가 “지금 어떤 판단이 비어 있는가”를 확인하는 작업 공간이다.

### 3.3 공개 프로젝트 페이지

외부 사용자, Scout, 협업자, 테스터, 멘토가 프로젝트를 이해하는 화면이다.

공개 프로젝트 페이지는 제품 소개보다 다음 정보를 우선한다.

- 현재 필요한 것
- 문제 정의
- 현재 가설
- 최근 전환
- 프로젝트 진화도
- 공개 변화 카드
- 피드백 요청

공개 프로젝트 페이지는 프로젝트가 왜 지금 모습이 되었는지 빠르게 이해시키는 역할을 한다.

### 3.4 탐색/히트맵 영역

사용자가 BuildMap에 공개된 프로젝트를 발견하는 입구다.

히트맵은 단순 인기 순위가 아니다. 히트맵은 지금 살아 움직이는 프로젝트를 발견하는 방식이다.

초기에는 외부 트래픽 데이터가 아니라 플랫폼 내부 활동 지표를 기반으로 가정한다. 히트맵 산식은 확정하지 않고 추가 검토 필요로 남긴다.

### 3.5 Scout 탐색 영역

Scout가 프로젝트와 Builder를 발견하는 영역이다.

초기 Scout는 채용 담당자가 아니다. 초기 Scout는 다음 역할에 가깝다.

- 프로젝트 발견자
- 피드백 제공자
- 테스터
- 협업자
- 멘토
- 팀원 탐색자

Scout 탐색 영역은 사람을 점수화하지 않는다. 프로젝트 기록에서 드러나는 근거를 통해 사람과 프로젝트를 이해하게 한다.

### 3.6 피드백 영역

외부 사용자나 Scout가 프로젝트 전체가 아니라 특정 판단, 변화 카드, 피드백 요청에 응답하는 영역이다.

피드백은 다음과 같이 연결될 수 있다.

- 특정 변화 카드에 대한 의견
- 문제 정의 공감 여부
- 가설에 대한 의문
- 기능 제거 판단에 대한 동의/반대
- 테스터 참여 의사
- 협업 제안

Builder는 피드백을 검토하고, 필요한 경우 새 변화 카드로 구조화한다.

## 4. 핵심 객체 간 관계

### 4.1 Builder

Builder는 프로젝트를 만든 사람이 아니라 프로젝트를 성장시키는 사람이다.

Builder는 프로젝트를 만들고, 문제 정의와 가설을 기록하고, 거친 메모를 입력하고, AI 초안을 승인하여 Decision Timeline을 형성한다.

### 4.2 프로젝트

프로젝트는 완성작 등록 단위가 아니다. 프로젝트는 판단 흐름이 축적되는 단위다.

프로젝트는 다음 정보를 가진다.

- 프로젝트명
- 한 줄 설명
- 현재 상태
- 해결하려는 문제
- 현재 가설
- 현재 필요한 도움
- 공개 상태
- Decision Timeline

세부 필드는 아직 DB 스키마로 확정하지 않는다.

### 4.3 문제 정의

문제 정의는 프로젝트의 중심축이다.

문제 정의는 한 번 쓰고 끝나는 정보가 아니다. 사용자 피드백이나 실험 결과에 따라 수정될 수 있으며, 문제 정의 변경은 중요한 변화 카드가 될 수 있다.

### 4.4 가설

가설은 기능 목록이 아니라 프로젝트가 맞다고 믿고 있는 판단이다.

가설은 다음 상태를 가질 수 있다.

- 가정
- 검증 중
- 일부 확인
- 확인됨
- 반박됨
- 보류

가설은 변화 카드, 실험, 피드백과 연결되어야 한다.

### 4.5 변화 카드

변화 카드는 프로젝트에서 발생한 하나의 의사결정 단위다.

변화 카드는 단순 작업 로그가 아니다. 변화 카드는 문제, 근거, 판단, 변경, 다음 확인 사항을 포함하여 프로젝트가 왜 바뀌었는지 설명한다.

### 4.6 Decision Timeline

Decision Timeline은 변화 카드가 시간과 맥락에 따라 연결된 흐름이다.

Decision Timeline은 일반 활동 로그가 아니다. 프로젝트가 왜 지금 모습이 되었는지를 보여주는 판단 흐름이다.

### 4.7 피드백

피드백은 프로젝트 전체 평가보다 특정 질문, 변화 카드, 문제 정의, 가설과 연결될 때 의미가 커진다.

피드백은 Builder가 검토한 뒤 새 변화 카드로 이어질 수 있다.

### 4.8 공개 프로젝트 페이지

공개 프로젝트 페이지는 프로젝트의 공개 가능한 판단 흐름을 외부에 보여주는 화면이다.

내부 기록 전체를 공개하지 않는다. Builder가 승인하고 공개 가능한 정보만 노출한다.

### 4.9 Scout

Scout는 프로젝트와 Builder를 발견하는 사람이다.

초기 Scout는 피드백 제공자, 테스터, 협업자, 멘토, 팀원 탐색자에 가깝다. 채용 담당자와 헤드헌터는 장기 확장 사용자로 남긴다.

## 5. 객체별 생성 화면과 읽기 화면

| 객체 | 생성/수정 화면 | 읽기 화면 |
|---|---|---|
| Builder 역할 | 가입 및 역할 선택 | Builder 프로필, 공개 프로젝트 페이지 |
| 프로젝트 | 프로젝트 생성 화면 | Builder 대시보드, 공개 프로젝트 페이지, 탐색 화면 |
| 문제 정의 | 프로젝트 생성, 문제/가설 입력 화면 | Decision Timeline, 공개 프로젝트 페이지 |
| 가설 | 프로젝트 생성, 문제/가설 입력 화면 | Decision Timeline, 공개 프로젝트 페이지 |
| 거친 메모 | 거친 메모 입력 화면 | Builder 내부 작업 공간 |
| AI 변화 카드 초안 | AI 변화 카드 초안 화면 | 승인 전 내부 화면 |
| 변화 카드 | 변화 카드 승인 화면 | Decision Timeline, 공개 프로젝트 페이지 |
| 피드백 | 피드백 요청/작성 화면 | Builder 대시보드, 변화 카드, 공개 프로젝트 페이지 |
| 현재 필요한 것 | 프로젝트 생성/관리 화면 | 공개 프로젝트 페이지, Scout 탐색 화면 |
| 히트맵 타일 | 공개 프로젝트 정보 기반 | 탐색/히트맵 화면 |

## 6. 내부 기록과 공개 노출의 관계

BuildMap의 기본 정책은 내부 기록 후 공개 전환이다.

```text
내부 메모
→ AI 초안
→ Builder 수정
→ Builder 승인
→ 내부 Decision Timeline 반영
→ 공개 가능 여부 선택
→ 공개 프로젝트 페이지 노출
```

AI 초안은 자동 공개되지 않는다. Builder가 승인하지 않은 변화 카드는 공식 Decision Timeline에 반영되지 않는다.

민감한 내부 메모, 미확정 판단, 외부 공개에 부적합한 내용은 공개 제외 상태로 남을 수 있어야 한다.

## 7. 초기 구현에서 필요한 정보 구조

초기 구현에서 필요한 정보 구조는 다음으로 제한하는 것이 적합하다.

- Builder
- 프로젝트
- 문제 정의
- 가설
- 거친 메모
- AI 변화 카드 초안
- 승인된 변화 카드
- Decision Timeline
- 공개 프로젝트 페이지
- 피드백 요청
- 피드백 작성
- 간단한 Scout 탐색
- 기본 히트맵 또는 카드 그리드

## 8. 후순위 정보 구조

후순위로 남길 정보 구조는 다음이다.

- 인수인계 전용 모드
- 고급 Scout 저장 목록
- 테스터 매칭 관리
- 협업자 요청 관리
- 팀 단위 프로젝트 운영
- 외부 GitHub/Notion 연동
- 고급 활동 지표
- 채용/헤드헌팅 흐름
- 투자자 탐색 흐름
- 조직용 온보딩 구조

## 9. 추가 검토 필요

다음 항목은 아직 확정하지 않는다.

- 변화 카드의 최종 필드 구성
- 히트맵 산식
- 공개/비공개 권한의 세부 단계
- Scout와 Builder 간 연락 방식
- 피드백의 공개 여부
- 프로젝트 팔로우 알림 방식
- AI 초안 생성 실패 시 재시도 정책
