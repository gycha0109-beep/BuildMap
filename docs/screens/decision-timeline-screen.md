# Decision Timeline 화면

## 1. 화면 목적

Decision Timeline은 BuildMap의 중심 화면이다.

이 화면의 목적은 프로젝트의 일반 활동 로그를 보여주는 것이 아니다. 목적은 프로젝트가 왜 지금의 모습이 되었는지, 어떤 판단이 이어져 현재 상태에 도달했는지를 보여주는 것이다.

> Decision Timeline은 코드 diff가 아니라 판단 흐름이다.

## 2. 주요 사용자

Decision Timeline은 사용자 유형에 따라 다른 의미를 가진다.

| 사용자 | 보는 목적 |
|---|---|
| Builder | 자신의 프로젝트가 어떤 판단 위에 서 있는지 확인 |
| 외부 사용자 | 프로젝트가 왜 현재 모습이 되었는지 이해 |
| Scout | Builder와 프로젝트의 판단 과정을 근거 중심으로 파악 |
| 새 팀원/협업자 | 프로젝트 맥락을 빠르게 이해 |

## 3. Builder가 보는 경우

### 목적

Builder는 Decision Timeline을 통해 프로젝트의 판단 공백과 전환 흐름을 확인한다.

### 화면에서 보여야 할 정보

- 현재 문제 정의
- 현재 가설
- 최근 변화 카드
- 핵심 전환점
- 미해결 가설
- 최근 피드백 요청
- 공개되지 않은 내부 카드
- 공개 가능한 카드

### 핵심 행동

- 새 변화 카드 작성
- 변화 카드 수정
- 공개 여부 변경
- Decision Diff 확인
- 피드백 요청 생성

### 주의점

Builder 화면에서는 내부 비공개 카드도 볼 수 있어야 한다. 단, 공개 페이지에서 보이는 정보와 명확히 구분해야 한다.

## 4. 외부 사용자가 보는 경우

### 목적

외부 사용자는 프로젝트 소개글보다 프로젝트가 왜 지금 모습이 되었는지 빠르게 이해해야 한다.

### 화면에서 보여야 할 정보

- 공개된 핵심 변화 카드
- 현재 문제 정의
- 현재 가설
- 최근 방향 전환
- 현재 필요한 도움
- 피드백 요청

### 핵심 행동

- 공개 변화 카드 읽기
- 피드백 작성
- 프로젝트 팔로우 또는 저장
- 관련 링크 이동

### 주의점

외부 사용자에게 너무 많은 내부 기록을 보여주면 흐름이 흐려진다. 공개 Timeline은 핵심 카드 중심으로 압축되어야 한다.

## 5. Scout가 보는 경우

### 목적

Scout는 프로젝트 결과물보다 판단 기록에서 드러나는 행동과 근거를 본다.

### 화면에서 보여야 할 정보

- 문제 정의 변경 기록
- 사용자 피드백 반영 기록
- 기능 제거 판단
- 실험 반복 기록
- 최근 업데이트
- 현재 필요한 도움

### 핵심 행동

- 특정 변화 카드에 피드백 작성
- 테스터 신청
- 프로젝트 저장
- 맥락 기반 연락

### 주의점

Scout 화면은 사람 평가 화면이 아니다. 점수, 등급, 자동 역량 평가 대신 기록 기반 근거를 보여준다.

## 6. 새 팀원/협업자가 보는 경우

### 목적

새 팀원이나 협업자는 Decision Timeline을 통해 5분 안에 프로젝트의 큰 흐름을 이해해야 한다.

### 화면에서 보여야 할 정보

- 처음 문제 정의
- 주요 가설
- 핵심 방향 전환
- 기능 제거 이유
- 현재 필요한 것
- 최근 결정
- 관련 링크

### 핵심 행동

- 핵심 전환점만 보기
- 인수인계 메모 카드 보기
- 관련 Notion/GitHub 링크로 이동

### 주의점

인수인계 전용 복잡한 모드는 초기에는 만들지 않는다. 초기에는 프로젝트 상세 페이지와 Decision Timeline의 요약/타임라인 확장 기능으로 다룬다.

## 7. Timeline에 표시되는 카드 유형

Decision Timeline에는 다음 변화 카드 유형이 표시될 수 있다.

- 문제 발견
- 문제 정의 수정
- 가설 생성
- 가설 반박
- 실험
- 사용자 피드백
- 기능 추가
- 기능 제거
- 판단 유지
- 판단 수정
- 방향 전환
- 릴리즈
- 인수인계 메모

카드 유형은 단순 분류가 아니라 프로젝트 판단 흐름의 맥락을 이해시키기 위한 장치다.

## 8. 핵심 전환점 강조 방식

핵심 전환점은 프로젝트의 방향이나 핵심 판단이 바뀐 지점이다.

예시:

- 문제 정의가 바뀐 카드
- 핵심 가설이 반박된 카드
- 주요 기능을 제거한 카드
- 타깃 사용자가 바뀐 카드
- 공개 릴리즈의 목적이 바뀐 카드

표현 방식은 아직 디자인으로 확정하지 않는다. 제품 설계상 가능한 방식은 다음이다.

- 카드 상단에 `핵심 전환` 표시
- Timeline에서 더 크게 표시
- 프로젝트 진화도에서 노드로 강조
- 공개 프로젝트 페이지의 최근 전환 영역에 노출

## 9. 중요도가 낮은 카드 처리 방식

모든 기록을 동일한 무게로 보여주면 판단 흐름이 묻힌다.

중요도가 낮은 카드는 다음 방식으로 처리할 수 있다.

- 기본 접힘 처리
- 월별/주별 요약 처리
- 필터에서만 노출
- Builder 내부에서는 보이고 공개 화면에서는 숨김

정확한 요약 기준은 추가 검토 필요다.

## 10. 문제 정의, 가설, 실험, 피드백, 판단의 연결 표현

Decision Timeline은 단순 최신순 목록이 아니어야 한다.

각 변화 카드는 다음 연결을 보여줄 수 있어야 한다.

```text
문제 정의
→ 가설
→ 실험 또는 피드백
→ 판단
→ 변경
→ 다음 확인 사항
```

예시:

```text
문제 정의: 사용자는 장기 목표와 이번 주 실행 단위를 연결하지 못한다.
가설: 이번 주 증거 단위가 실행을 돕는다.
피드백: 첫 화면의 장기 목표 입력이 부담스럽다는 반응이 있었다.
판단: 장기 목표 입력을 첫 화면에서 제거한다.
변경: 이번 주 목표 선택을 우선 배치한다.
다음 확인: 사용자가 10초 안에 목적을 이해하는지 확인한다.
```

## 11. 필터

초기 필터 후보는 다음이다.

### 카드 유형

- 문제 발견
- 가설 반박
- 사용자 피드백
- 기능 제거
- 방향 전환
- 릴리즈 등

### 공개/비공개

- 전체
- 내부 전용
- 공개 가능
- 공개됨

### 핵심 전환

- 전체
- 핵심 전환만

### 연결된 가설

- 특정 가설과 연결된 변화 카드만 보기

### 기간

- 최근 7일
- 최근 30일
- 전체
- 직접 선택

필터는 Timeline을 활동 로그가 아니라 판단 흐름으로 읽기 위한 보조 장치다.

## 12. Timeline 상단 요약 정보

Timeline 상단에는 현재 판단 상태를 요약해야 한다.

필수 후보는 다음이다.

1. 현재 문제 정의
2. 현재 가설
3. 최근 전환
4. 현재 필요한 것

이 순서는 Builder와 Scout가 프로젝트의 현재 맥락을 빠르게 이해하기 위해 필요하다.

## 13. Decision Diff로 이동하는 흐름

Decision Diff는 코드 diff가 아니라 판단 diff다.

Timeline에서 다음 경우 Decision Diff로 이동할 수 있다.

- 문제 정의가 바뀐 카드
- 가설이 반박된 카드
- 방향 전환 카드
- 기능 제거 카드
- 릴리즈 카드

Decision Diff는 다음을 비교할 수 있다.

- 초기 판단과 현재 판단
- 전환 전과 전환 후
- 이전 문제 정의와 현재 문제 정의
- 이전 타깃 사용자와 현재 타깃 사용자
- 이전 성공 기준과 현재 성공 기준

초기 구현에서는 간단한 Decision Diff만 후보로 둔다.

## 14. 프로젝트 인수인계에 사용되는 방식

Decision Timeline은 초기 인수인계의 핵심 자료가 될 수 있다.

새 팀원이나 협업자는 다음 순서로 프로젝트를 이해한다.

1. 현재 문제 정의 확인
2. 현재 가설 확인
3. 핵심 전환점 확인
4. 기능 제거 이유 확인
5. 인수인계 메모 카드 확인
6. 현재 필요한 것 확인
7. 세부 코드는 GitHub, 세부 문서는 Notion에서 확인

전용 인수인계 모드는 2차 확장 후보로 둔다.

## 15. 초기 구현 범위

초기 Decision Timeline 화면은 다음을 포함한다.

- 상단 요약
  - 현재 문제 정의
  - 현재 가설
  - 최근 전환
  - 현재 필요한 것
- 변화 카드 목록
- 핵심 전환 표시
- 카드 유형 필터
- 공개/비공개 상태 표시
- 새 변화 카드 작성 진입
- 공개 프로젝트 페이지에서 공개 카드만 노출

## 16. 후순위 요소

- 그래프형 프로젝트 진화도
- 고급 Decision Diff
- 인수인계 요약 화면
- Timeline 자동 요약
- 연결 관계 시각화
- 팀 단위 Timeline 관리

## 17. 추가 검토 필요

- Timeline 기본 정렬 방식
- 핵심 전환 자동 추천 여부
- 공개 Timeline의 압축 기준
- 낮은 중요도 카드 요약 방식
- Decision Diff 최소 화면 범위
