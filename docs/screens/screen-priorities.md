# 화면 우선순위

## 1. 문서 목적

이 문서는 BuildMap 화면의 우선순위를 정리한다.

우선순위의 기준은 다음이다.

- BuildMap의 핵심 철학을 직접 구현하는가
- Builder의 판단 흐름 기록에 필수인가
- 변화 카드와 Decision Timeline을 성립시키는가
- 공개 프로젝트 페이지와 피드백 흐름으로 이어지는가
- 초기 구현에서 과도한 복잡성을 만들지 않는가

## 2. 핵심 화면

핵심 화면은 BuildMap의 정체성을 성립시키기 위해 반드시 필요한 화면이다.

| 화면 | 이유 |
|---|---|
| Builder 대시보드 | Builder가 프로젝트의 판단 흐름과 현재 필요한 행동을 확인하는 중심 |
| 프로젝트 생성 화면 | 완성작 등록이 아니라 판단 흐름 시작점 |
| 문제/가설 입력 화면 | 프로젝트의 중심 판단을 기록하는 화면 |
| 거친 메모 입력 화면 | Builder의 실제 기록 습관이 시작되는 화면 |
| AI 변화 카드 초안 화면 | AI가 생성보다 구조화에 집중하는 핵심 화면 |
| 변화 카드 승인 화면 | Builder가 최종 승인자임을 보장하는 화면 |
| Decision Timeline 화면 | 프로젝트가 왜 현재 모습이 되었는지 보여주는 중심 화면 |
| 공개 프로젝트 페이지 | 외부에 판단 흐름과 현재 필요한 도움을 보여주는 화면 |
| 피드백 요청/작성 화면 | 외부 피드백이 새 판단 흐름으로 이어지는 화면 |

## 3. 초기 구현 화면

초기 구현 화면은 핵심 화면을 보조하고, 공개/탐색 흐름을 최소 수준으로 성립시키는 화면이다.

| 화면 | 초기 범위 |
|---|---|
| 프로젝트 목록 | Builder가 자신의 프로젝트를 확인 |
| 간단한 탐색 화면 | 공개 프로젝트를 목록/카드로 탐색 |
| 간단한 Scout 탐색 화면 | 프로젝트 발견, 피드백, 테스터 신청 중심 |
| 기본 히트맵 또는 카드 그리드 | 살아 움직이는 프로젝트 발견 입구 |
| 내부/공개 전환 화면 | 내부 기록 후 공개 전환 기본 정책 구현 |
| 간단한 Decision Diff 화면 | 초기 판단과 현재 판단 비교 |

초기에는 히트맵이 복잡한 시각화일 필요가 없다. 프로젝트 카드 그리드와 필터만으로도 목적을 일부 달성할 수 있다.

## 4. 2차 확장 화면

2차 확장 화면은 핵심 흐름이 안정된 뒤 추가할 수 있는 화면이다.

| 화면 | 이유 |
|---|---|
| 인수인계 요약 화면 | 새 팀원/협업자가 프로젝트 맥락을 빠르게 이해 |
| 협업자 요청 화면 | 맥락 기반 협업 제안 흐름 강화 |
| 테스터 매칭 화면 | 테스트 목적과 참여자 연결 강화 |
| Scout 저장 목록 | Scout가 관심 프로젝트를 지속 추적 |
| 프로젝트 활동 지표 화면 | 내부 활동성 이해 보조 |
| 공개/비공개 세부 권한 관리 화면 | 카드별 공개 제어 고도화 |

인수인계는 초기에는 별도 모드가 아니라 프로젝트 상세 페이지와 Decision Timeline의 요약/타임라인 확장 기능으로 다룬다. 전용 인수인계 모드는 2차 확장이다.

## 5. 장기 확장 화면

장기 확장 화면은 BuildMap의 핵심 흐름이 충분히 검증된 뒤 고려한다.

| 화면 | 이유 |
|---|---|
| 채용/헤드헌팅 화면 | Scout의 장기 확장 |
| 투자자 탐색 화면 | 프로젝트 발견의 장기 확장 |
| 팀 플랜 관리 화면 | 조직/팀 단위 운영 |
| 조직용 온보딩 화면 | 인수인계와 교육 확장 |
| 외부 GitHub/Notion 연동 화면 | 코드/문서와 판단 흐름 연결 |
| 고급 히트맵 분석 화면 | 활동성 탐색 고도화 |
| AI 고급 요약/검색 화면 | 긴 Timeline 이해 보조 |

이 단계에서는 장기 확장 화면을 초기 핵심으로 과장하지 않는다.

## 6. 이번 단계에서 제외할 화면

이번 단계에서는 다음 화면을 제외한다.

- 결제
- 가격 정책
- 지원자 관리 시스템
- 투자 매칭 관리
- AI 역량 평가 대시보드
- 보류 중인 고급 브랜딩 개념 화면
- 실시간 협업 편집 화면

제외 이유는 다음이다.

- BuildMap이 채용/평가 플랫폼처럼 보일 위험이 있다.
- Builder의 판단 흐름 기록이라는 핵심에서 멀어진다.
- 초기 화면 구조를 과도하게 복잡하게 만든다.
- 아직 DB, API, 권한, 결제 정책이 확정되지 않았다.

## 7. 첫 구현 화면 흐름 후보

첫 구현은 다음 화면 흐름을 중심으로 두는 것이 적합하다.

```text
Builder 시작
→ 프로젝트 생성
→ 문제/가설 입력
→ 거친 메모 입력
→ AI 변화 카드 초안
→ 변화 카드 승인
→ Decision Timeline
→ 공개 프로젝트 페이지
→ 피드백 요청/작성
```

Scout와 히트맵은 다음 수준으로 제한한다.

```text
공개 프로젝트 탐색
→ 공개 프로젝트 페이지
→ Decision Timeline 확인
→ 피드백 작성 / 테스터 신청 / 저장
```

## 8. 추가 검토 필요

- Builder 대시보드의 첫 화면 정보 우선순위
- 프로젝트 생성과 문제/가설 입력을 한 화면에 둘지 분리할지
- AI 변화 카드 초안 화면의 입력/출력 상세
- 공개 프로젝트 페이지와 Decision Timeline의 중복 정보 처리
- 기본 히트맵을 카드 그리드로 시작할지
- 피드백 작성에 로그인 필요 여부
