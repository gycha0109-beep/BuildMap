# Builder 전체 화면 흐름

## 1. 문서 목적

이 문서는 Builder가 BuildMap에서 프로젝트 판단 흐름을 기록하고 공개하는 전체 화면 흐름을 정리한다.

Builder 핵심 흐름은 다음이다.

```text
가입 및 역할 선택
→ 첫 프로젝트 생성
→ 문제 정의 입력
→ 가설 입력
→ 거친 메모 입력
→ AI 변화 카드 초안 생성
→ Builder 수정 및 승인
→ Decision Timeline 반영
→ 프로젝트 진화도 확인
→ 공개 전환
→ 피드백 요청
→ 피드백 반영 후 새 변화 카드 작성
```

이 흐름의 중심은 완성작 등록이 아니다. 중심은 프로젝트가 어떤 판단을 거쳐 변화했는지 축적하는 것이다.

## 2. 전체 흐름 요약

| 단계 | 화면 목적 | 핵심 행동 |
|---|---|---|
| 1 | Builder로 시작 | 역할 선택 |
| 2 | 프로젝트 시작 | 프로젝트명, 문제, 현재 필요한 도움 입력 |
| 3 | 문제 정의 | 프로젝트의 중심 문제 기록 |
| 4 | 가설 입력 | 검증할 판단 기록 |
| 5 | 거친 메모 | 회의/테스트/작업 중 메모 입력 |
| 6 | AI 초안 | 메모를 변화 카드 초안으로 구조화 |
| 7 | 승인 | Builder가 수정하고 최종 승인 |
| 8 | Timeline 반영 | 공식 판단 흐름에 반영 |
| 9 | 진화도 확인 | 핵심 전환 흐름 확인 |
| 10 | 공개 전환 | 공개 가능한 카드 선택 |
| 11 | 피드백 요청 | 특정 질문에 대한 피드백 요청 |
| 12 | 피드백 반영 | 피드백을 새 변화 카드로 연결 |

## 3. 단계별 화면 흐름

## 3.1 가입 및 역할 선택

### 사용자의 상황

사용자는 프로젝트를 만들거나 성장시키는 사람으로 BuildMap을 사용하려 한다. 사용자는 개발자일 수도 있고, 비개발자일 수도 있고, 바이브코딩 유저일 수도 있다.

### 화면에서 보여야 할 핵심 정보

- Builder는 프로젝트를 만든 사람이 아니라 프로젝트를 성장시키는 사람이라는 설명
- 선택 가능한 역할 예시
  - 창업자
  - 학생 메이커
  - 개발자
  - 디자이너
  - PM
  - 비개발자
  - 바이브코딩 유저
  - 사이드프로젝트 제작자
- 역할 선택은 신분 인증이 아니라 탐색과 프로필 구성을 위한 메타데이터라는 안내

### 사용자가 해야 할 핵심 행동

- Builder로 시작하기 선택
- 역할 또는 현재 위치 선택
- 관심 프로젝트 유형 선택 가능

### 다음 화면으로 넘어가는 조건

- 최소 1개의 역할을 선택하거나, 역할 선택을 건너뛰고 프로젝트 생성으로 이동한다.

### 실패하거나 막힐 수 있는 지점

- 사용자가 자신을 개발자/비개발자 중 하나로 고정해야 한다고 느낄 수 있다.
- 역할 선택이 자격 검증처럼 보이면 진입 장벽이 생긴다.

### 후순위 요소

- 역할별 추천 템플릿
- 역할별 공개 프로필 구성
- 팀 단위 역할 관리

## 3.2 첫 프로젝트 생성

### 사용자의 상황

Builder는 아직 완성된 제품이 없지만 아이디어와 문제의식을 기록하기 시작하려 한다.

### 화면에서 보여야 할 핵심 정보

- 프로젝트 생성은 포트폴리오 등록이 아니라 판단 흐름을 시작하는 행위라는 안내
- 긴 기획서를 요구하지 않는다는 메시지
- 나중에 수정 가능하다는 안내

### 사용자가 해야 할 핵심 행동

- 프로젝트명 입력
- 한 줄 설명 입력
- 현재 상태 선택
- 해결하려는 문제 입력
- 아직 확정되지 않은 가설 입력
- 현재 필요한 도움 입력
- 공개 여부 선택

### 다음 화면으로 넘어가는 조건

- 프로젝트명과 최소 문제 정의 또는 한 줄 설명이 입력된다.

### 실패하거나 막힐 수 있는 지점

- 처음부터 너무 많은 항목을 요구하면 사용자가 이탈한다.
- 완성도 높은 소개글을 요구하면 BuildMap이 포트폴리오 등록 서비스처럼 보인다.

### 후순위 요소

- 프로젝트 템플릿
- 분야별 질문 자동 추천
- 외부 링크 자동 가져오기

## 3.3 문제 정의 입력

### 사용자의 상황

Builder는 프로젝트가 어떤 문제에서 시작했는지 기록한다.

### 화면에서 보여야 할 핵심 정보

- 문제 정의는 프로젝트의 중심축이라는 설명
- 문제 정의는 나중에 수정될 수 있다는 안내
- 문제 정의가 바뀌면 변화 카드로 남길 수 있다는 안내

### 사용자가 해야 할 핵심 행동

- 현재 문제 정의 입력
- 문제를 관찰한 근거 입력 가능
- 아직 애매한 부분 표시 가능

### 다음 화면으로 넘어가는 조건

- 문제 정의 초안이 저장된다.

### 실패하거나 막힐 수 있는 지점

- 사용자가 문제 정의를 완벽하게 써야 한다고 느낄 수 있다.
- 솔루션 설명과 문제 설명이 섞일 수 있다.

### 후순위 요소

- 문제 정의 개선 질문
- 유사 문제 정의 예시
- 문제 정의 변경 이력 비교

## 3.4 가설 입력

### 사용자의 상황

Builder는 프로젝트가 성립하려면 사실이어야 하는 판단을 기록한다.

### 화면에서 보여야 할 핵심 정보

- 가설은 기능 목록이 아니라 검증할 판단이라는 설명
- 가설 상태 예시
  - 가정
  - 검증 중
  - 일부 확인
  - 확인됨
  - 반박됨
  - 보류
- 가설은 실험, 피드백, 변화 카드와 연결될 수 있다는 안내

### 사용자가 해야 할 핵심 행동

- 핵심 가설 입력
- 현재 상태 선택
- 연결할 문제 정의 선택 가능

### 다음 화면으로 넘어가는 조건

- 최소 1개 가설이 저장되거나, 가설 입력을 보류한다.

### 실패하거나 막힐 수 있는 지점

- 기능 아이디어를 가설로 착각할 수 있다.
- 상태가 지나치게 복잡하면 기록 부담이 커진다.

### 후순위 요소

- 가설 검증 체크리스트
- 실험 설계 보조
- 가설별 피드백 모음

## 3.5 거친 메모 입력

### 사용자의 상황

Builder는 회의 후, 사용자 테스트 후, 작업 중에 짧은 메모를 남긴다.

### 화면에서 보여야 할 핵심 정보

- 완성된 회고문을 쓰지 않아도 된다는 안내
- 짧고 거친 메모도 충분하다는 안내
- 메모는 곧바로 공개되지 않는다는 안내

### 사용자가 해야 할 핵심 행동

- 텍스트 메모 입력
- 관련 프로젝트 선택
- 필요하면 관련 문제/가설 선택

초기 구현에서는 텍스트 메모 입력에서 시작한다. 음성, 이미지, 회의록, 외부 문서 업로드는 후순위다.

### 다음 화면으로 넘어가는 조건

- 메모가 저장되고 AI 구조화 요청이 가능해진다.

### 실패하거나 막힐 수 있는 지점

- 사용자가 메모가 너무 짧아도 되는지 확신하지 못할 수 있다.
- 메모가 공개되는 것으로 오해할 수 있다.

### 후순위 요소

- 음성 메모
- 이미지 첨부
- 회의록 업로드
- 외부 문서 업로드

## 3.6 AI 변화 카드 초안 생성

### 사용자의 상황

Builder가 남긴 거친 메모를 변화 카드 초안으로 구조화하려 한다.

### 화면에서 보여야 할 핵심 정보

- AI는 내용을 꾸미지 않는다는 안내
- AI는 없는 성과를 만들지 않는다는 안내
- AI는 메모를 문제, 근거, 판단, 변경, 다음 확인 사항으로 구조화한다는 안내
- AI 초안은 승인 전까지 공개되지 않는다는 안내

### 사용자가 해야 할 핵심 행동

- AI 구조화 실행
- 생성된 초안 확인
- 부족한 부분 수정

### 다음 화면으로 넘어가는 조건

- 초안이 생성되고 Builder가 검토 화면으로 이동한다.

### 실패하거나 막힐 수 있는 지점

- AI가 원문 의미를 과도하게 해석할 수 있다.
- 근거가 약한 내용을 확정처럼 표현할 수 있다.
- AI 실패 시 사용자가 직접 변화 카드를 작성할 수 있어야 한다.

### 후순위 요소

- AI 구조화 옵션
- 여러 초안 비교
- 프로젝트 맥락 기반 고급 요약

## 3.7 Builder 수정 및 승인

### 사용자의 상황

Builder는 AI가 만든 초안을 확인하고 공식 변화 카드로 반영할지 결정한다.

### 화면에서 보여야 할 핵심 정보

- Builder가 최종 승인자라는 안내
- 승인 전에는 Decision Timeline에 반영되지 않는다는 안내
- 공개 여부를 별도로 선택해야 한다는 안내

### 사용자가 해야 할 핵심 행동

- 변화 카드 제목 수정
- 유형 선택 또는 수정
- 근거/판단/변경 내용 확인
- 연결된 문제 또는 가설 선택
- 공개 여부 선택
- 승인 또는 보류 선택

### 다음 화면으로 넘어가는 조건

- Builder가 승인하면 Decision Timeline에 반영된다.
- Builder가 보류하면 내부 초안 상태로 남는다.

### 실패하거나 막힐 수 있는 지점

- 승인과 공개가 같은 행동으로 오해될 수 있다.
- AI 초안이 공식 기록처럼 보일 수 있다.

### 후순위 요소

- 승인 전 변경 이력
- 팀 승인 플로우
- 리뷰어 지정

## 3.8 Decision Timeline 반영

### 사용자의 상황

승인된 변화 카드가 프로젝트의 공식 판단 흐름에 반영된다.

### 화면에서 보여야 할 핵심 정보

- 변화 카드가 Timeline의 어느 위치에 반영되었는지
- 어떤 문제/가설/실험/피드백과 연결되었는지
- 공개 상태가 무엇인지

### 사용자가 해야 할 핵심 행동

- Timeline에서 반영 결과 확인
- 필요하면 중요도 수정
- 필요하면 공개 전환 준비

### 다음 화면으로 넘어가는 조건

- Timeline 확인 후 프로젝트 진화도 또는 공개 전환 화면으로 이동한다.

### 실패하거나 막힐 수 있는 지점

- Timeline이 단순 최신순 목록처럼 보이면 BuildMap의 차별점이 약해진다.

### 후순위 요소

- Timeline 압축 보기
- 전환점 자동 강조
- Decision Diff 연결

## 3.9 프로젝트 진화도 확인

### 사용자의 상황

Builder는 프로젝트가 처음 문제에서 현재 상태까지 어떻게 변화했는지 한눈에 보고 싶다.

### 화면에서 보여야 할 핵심 정보

- 핵심 변화 카드
- 문제 정의 변화
- 가설 상태 변화
- 주요 방향 전환
- 최근 피드백과 반영 여부

### 사용자가 해야 할 핵심 행동

- 핵심 전환 흐름 확인
- 불필요한 카드 접기
- 공개할 핵심 흐름 선택

### 다음 화면으로 넘어가는 조건

- 공개 전환 또는 피드백 요청 화면으로 이동한다.

### 실패하거나 막힐 수 있는 지점

- 모든 카드를 동일하게 보여주면 흐름이 묻힌다.

### 후순위 요소

- 그래프형 진화도
- 핵심 전환 자동 요약
- 인수인계 요약 생성

## 3.10 공개 전환

### 사용자의 상황

Builder는 내부적으로 기록한 프로젝트를 외부에 공개하려 한다.

### 화면에서 보여야 할 핵심 정보

- 기본 정책은 내부 비공개라는 안내
- 공개 가능한 변화 카드만 선택해야 한다는 안내
- AI 초안은 자동 공개되지 않는다는 안내
- 민감한 내부 메모는 공개 제외할 수 있다는 안내

### 사용자가 해야 할 핵심 행동

- 프로젝트 공개 상태 선택
  - 비공개
  - 링크 공개
  - 전체 공개
- 공개 변화 카드 선택
- 현재 필요한 도움 확인
- 공개 페이지 미리보기

### 다음 화면으로 넘어가는 조건

- 공개 설정을 저장하면 공개 프로젝트 페이지가 생성 또는 갱신된다.

### 실패하거나 막힐 수 있는 지점

- 내부 기록이 실수로 공개되면 신뢰가 깨진다.
- 공개 여부와 승인 상태가 혼동될 수 있다.

### 후순위 요소

- 카드별 세부 공개 권한
- 팀 단위 승인
- 공개 전 민감 정보 자동 감지

## 3.11 피드백 요청

### 사용자의 상황

Builder는 프로젝트 전체가 아니라 특정 판단이나 화면, 문제 정의에 대해 피드백을 받고 싶다.

### 화면에서 보여야 할 핵심 정보

- 피드백 요청은 특정 질문에 연결되어야 한다는 안내
- 피드백 요청 예시
  - 첫 화면 이해도
  - 문제 정의 공감도
  - 기능 필요성
  - 테스터 모집
  - 협업자 요청

### 사용자가 해야 할 핵심 행동

- 피드백 질문 작성
- 연결할 변화 카드 또는 문제/가설 선택
- 공개 여부 선택

### 다음 화면으로 넘어가는 조건

- 공개 프로젝트 페이지나 탐색 화면에 피드백 요청이 노출된다.

### 실패하거나 막힐 수 있는 지점

- 질문이 모호하면 피드백이 프로젝트 평가처럼 흩어진다.

### 후순위 요소

- 피드백 템플릿
- 테스터 모집 양식
- 피드백 분류

## 3.12 피드백 반영 후 새 변화 카드 작성

### 사용자의 상황

Builder는 외부 피드백을 검토한 뒤 프로젝트 판단을 유지하거나 수정하려 한다.

### 화면에서 보여야 할 핵심 정보

- 피드백이 어떤 변화 카드 또는 질문에 연결되어 있는지
- 반영함/반영하지 않음/보류 상태
- 피드백을 새 변화 카드로 구조화할 수 있다는 안내

### 사용자가 해야 할 핵심 행동

- 피드백 검토
- 반영 여부 선택
- 새 변화 카드 작성 또는 AI 구조화 요청

### 다음 화면으로 넘어가는 조건

- 새 변화 카드가 승인되면 Decision Timeline에 반영된다.

### 실패하거나 막힐 수 있는 지점

- 모든 피드백을 반영해야 한다는 압박이 생길 수 있다.
- 반영하지 않는 판단도 기록해야 한다는 점이 드러나야 한다.

### 후순위 요소

- 피드백 묶음 요약
- 반복 피드백 감지
- Scout별 피드백 이력

## 4. 초기 구현에서 반드시 필요한 Builder 흐름

첫 구현에서 Builder 핵심 흐름은 다음까지 포함하는 것이 적합하다.

```text
프로젝트 생성
→ 문제 정의
→ 가설
→ 거친 메모 입력
→ AI 변화 카드 초안
→ Builder 승인
→ Decision Timeline 반영
→ 공개 프로젝트 페이지
```

Scout와 히트맵은 최소 탐색 수준으로 둔다.

## 5. 추가 검토 필요

- 문제 정의와 가설 입력을 프로젝트 생성 화면에 얼마나 포함할지
- 변화 카드 승인 전 수정 경험의 세부 수준
- 공개 전환 체크리스트의 구체 항목
- 피드백 반영 상태를 어디까지 공개할지
- Builder 대시보드에서 할 일과 판단 공백을 어떻게 구분할지
