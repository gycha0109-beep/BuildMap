# 공개/비공개 상태와 화면 상태

## 1. 문서 목적

이 문서는 BuildMap에서 공개/비공개 상태와 주요 화면 상태를 정리한다.

BuildMap은 내부 기록 후 공개 전환을 기본 정책으로 한다. Builder가 승인하지 않은 AI 초안이나 민감한 내부 기록이 자동으로 공개되어서는 안 된다.

## 2. 기본 정책

기본 정책은 다음이다.

> 모든 기록은 내부 비공개에서 시작한다.

공개는 별도의 선택과 확인을 거쳐야 한다.

```text
내부 메모
→ AI 초안
→ Builder 수정
→ Builder 승인
→ 내부 Timeline 반영
→ 공개 가능 여부 선택
→ 공개 프로젝트 페이지 노출
```

승인과 공개는 다른 개념이다.

- 승인: 공식 Decision Timeline에 반영한다.
- 공개: 외부 사용자에게 노출한다.

## 3. 프로젝트 공개 상태

| 상태 | 의미 | 외부 노출 |
|---|---|---|
| 비공개 | Builder 내부에서만 관리 | 없음 |
| 링크 공개 | 링크를 아는 사용자만 접근 | 제한적 |
| 전체 공개 | 탐색/히트맵에 노출 가능 | 있음 |

### 비공개

기본 상태다. 프로젝트 생성 직후에는 비공개로 시작하는 것이 적합하다.

### 링크 공개

Builder가 일부 사람에게만 프로젝트를 공유하고 싶을 때 사용한다.

### 전체 공개

탐색 화면, 히트맵, Scout 탐색에 노출될 수 있는 상태다.

### 추가 검토 필요

- 링크 공개 프로젝트의 검색 노출 여부
- 전체 공개 전 필수 체크리스트
- 프로젝트 공개 상태 변경 이력 기록 여부

## 4. 변화 카드 공개 상태

| 상태 | 의미 |
|---|---|
| 내부 전용 | Builder 내부에서만 보이는 카드 |
| 공개 가능 | 공개 후보지만 아직 외부 노출 전 |
| 공개됨 | 공개 프로젝트 페이지에 노출되는 카드 |
| 민감 정보 포함 | 공개 전 검토가 필요한 카드 |

### 내부 전용

회의 내용, 민감한 판단, 미확정 방향 등 외부에 공개하기 어려운 기록이다.

### 공개 가능

Builder가 공개해도 된다고 판단했지만 아직 공개 페이지에 반영하지 않은 상태다.

### 공개됨

공개 프로젝트 페이지와 공개 Decision Timeline에 노출된다.

### 민감 정보 포함

고객명, 내부 수치, 개인 정보, 계약 내용, 미공개 전략 등이 포함될 수 있는 상태다.

## 5. AI 초안 상태

| 상태 | 의미 |
|---|---|
| 초안 | AI가 생성했거나 Builder가 작성 중인 상태 |
| 수정 중 | Builder가 내용을 검토하고 수정하는 상태 |
| 승인됨 | Builder가 공식 변화 카드로 확정한 상태 |
| 보류됨 | 아직 Timeline에 반영하지 않기로 한 상태 |

AI 초안은 절대 자동 공개하지 않는다.

승인 전 변화 카드는 공식 Decision Timeline에 반영되지 않는다.

## 6. 피드백 상태

| 상태 | 의미 |
|---|---|
| 새 피드백 | 아직 Builder가 확인하지 않은 피드백 |
| 검토 중 | Builder가 반영 여부를 판단 중인 피드백 |
| 반영됨 | 변화 카드나 판단 수정으로 이어진 피드백 |
| 반영하지 않음 | 의도적으로 반영하지 않기로 결정한 피드백 |

피드백을 반영하지 않는 결정도 중요한 판단이다. 필요한 경우 `판단 유지` 또는 `기능 보류` 변화 카드로 남길 수 있다.

## 7. 가설 상태

| 상태 | 의미 |
|---|---|
| 가정 | 아직 검증되지 않은 판단 |
| 검증 중 | 실험이나 피드백으로 확인 중 |
| 일부 확인 | 일부 근거가 있으나 확정은 아님 |
| 확인됨 | 충분한 근거로 유지 중 |
| 반박됨 | 실험이나 피드백으로 약해지거나 틀렸다고 판단됨 |
| 보류 | 지금은 판단하지 않고 뒤로 미룸 |

가설 상태 변경은 변화 카드와 연결될 수 있다.

## 8. 사용자에게 상태를 보여주는 방식

상태는 단순 라벨이 아니라 사용자가 다음 행동을 이해하기 위한 장치여야 한다.

예시:

- `초안`: 아직 공식 Timeline에 반영되지 않았습니다.
- `승인됨`: Builder가 확인했고 내부 Timeline에 반영되었습니다.
- `공개됨`: 공개 프로젝트 페이지에 노출됩니다.
- `민감 정보 포함`: 공개 전 내용을 다시 확인하세요.
- `반영하지 않음`: 피드백을 검토했지만 현재 판단을 유지했습니다.

## 9. 공개 전환 체크리스트

프로젝트나 변화 카드를 공개하기 전에 확인해야 할 항목은 다음이다.

- AI 초안이 아니라 Builder 승인 카드인가?
- 민감한 내부 정보가 포함되어 있지 않은가?
- 고객명, 개인 정보, 계약 내용이 포함되어 있지 않은가?
- 공개해도 되는 문제 정의인가?
- 공개해도 되는 가설인가?
- 공개할 필요가 있는 변화 카드인가?
- 현재 필요한 도움이 명확한가?
- 공개 페이지에서 맥락이 왜곡되지 않는가?

초기 구현에서는 체크리스트를 단순 안내로 제공하고, 강제 정책은 추가 검토 필요로 둔다.

## 10. 실수 공개 방지 원칙

실수로 민감한 기록이 공개되지 않도록 다음 원칙을 둔다.

- 기본값은 내부 비공개다.
- AI 초안은 자동 공개되지 않는다.
- 승인과 공개를 분리한다.
- 공개 전 미리보기를 제공한다.
- 민감 정보 포함 상태를 둘 수 있다.
- 공개 전환 시 확인 문구를 제공한다.

## 11. 초기 구현 범위

초기 구현 후보는 다음이다.

- 프로젝트 공개 상태: 비공개 / 링크 공개 / 전체 공개
- 변화 카드 공개 상태: 내부 전용 / 공개됨
- AI 초안 상태: 초안 / 승인됨 / 보류됨
- 피드백 상태: 새 피드백 / 반영됨 / 반영하지 않음
- 가설 상태: 가정 / 검증 중 / 확인됨 / 반박됨 / 보류
- 공개 전 간단한 확인 안내

## 12. 후순위 요소

- 공개 가능 상태와 공개됨 상태의 세부 분리
- 민감 정보 자동 감지
- 팀 승인 플로우
- 카드별 링크 공개
- 특정 Scout에게만 공개
- 공개 변경 이력

## 13. 추가 검토 필요

- 링크 공개의 접근 범위
- 공개 상태 변경 권한
- 팀 프로젝트에서 승인자 역할
- 피드백 공개 여부
- 민감 정보 감지 기준
- 공개된 카드 수정 시 처리 방식
