# 공개 프로젝트 페이지 구조

## 1. 문서 목적

이 문서는 BuildMap의 공개 프로젝트 페이지 구조를 정리한다.

공개 프로젝트 페이지는 제품 소개 페이지가 아니다. 공개 프로젝트 페이지는 프로젝트의 판단 흐름과 현재 필요한 도움을 보여주는 페이지다.

외부 사용자는 이 페이지를 통해 다음을 빠르게 이해해야 한다.

- 이 프로젝트는 어떤 문제에서 시작했는가
- 현재 어떤 가설 위에 서 있는가
- 최근 어떤 전환이 있었는가
- 지금 어떤 도움이나 피드백이 필요한가
- 프로젝트가 왜 지금 모습이 되었는가

## 2. 기본 정보 순서

공개 프로젝트 페이지의 기본 정보 순서는 다음으로 둔다.

1. 프로젝트명
2. 한 줄 정의
3. 현재 상태
4. 현재 필요한 것
5. 문제 정의
6. 현재 가설
7. 최근 전환
8. 프로젝트 진화도
9. 공개 변화 카드
10. 피드백 요청
11. 관련 링크
12. Builder 정보
13. 팔로우 또는 저장

이 순서는 제품 홍보보다 프로젝트 맥락과 참여 가능성을 먼저 보여주기 위한 구조다.

## 3. 영역별 구조

## 3.1 프로젝트명

### 영역의 목적

프로젝트를 식별한다.

### 왜 이 순서가 필요한가

사용자는 가장 먼저 어떤 프로젝트인지 알아야 한다.

### 비개발자가 이해하기 쉽게 보여주는 방법

- 짧은 이름
- 가칭인 경우 가칭 표시 가능

### Builder가 공개 여부를 제어해야 하는 정보

- 비공개 프로젝트명 또는 내부 코드명은 공개 전 수정 가능해야 한다.

### 초기 구현 범위

- 프로젝트명 노출

### 후순위 요소

- 공개용 이름과 내부 이름 분리
- 프로젝트명 변경 이력

## 3.2 한 줄 정의

### 영역의 목적

프로젝트가 무엇인지 한 문장으로 설명한다.

### 왜 이 순서가 필요한가

Scout와 외부 사용자가 프로젝트를 계속 볼지 빠르게 판단할 수 있다.

### 비개발자가 이해하기 쉽게 보여주는 방법

- 기술 설명보다 사용자 문제와 프로젝트 방향을 중심으로 작성
- 과장된 홍보 문구보다 현재 판단을 반영

### Builder가 공개 여부를 제어해야 하는 정보

- 내부 전략이나 미공개 방향이 포함된 문구는 공개 전 수정 가능해야 한다.

### 초기 구현 범위

- 한 줄 정의 노출

### 후순위 요소

- 공개용 요약과 내부 요약 분리
- 다국어 요약

## 3.3 현재 상태

### 영역의 목적

프로젝트가 지금 어떤 판단 단계에 있는지 보여준다.

### 왜 이 순서가 필요한가

완성 여부보다 현재 단계와 맥락이 중요하다.

### 비개발자가 이해하기 쉽게 보여주는 방법

상태 예시:

- 문제 정의 중
- 가설 검증 중
- 초기 제작 중
- 테스트 중
- 공개 전
- 공개됨
- 방향 전환 중

### Builder가 공개 여부를 제어해야 하는 정보

- 내부적으로만 쓰는 상태 설명은 공개 제외 가능해야 한다.

### 초기 구현 범위

- 단일 상태 표시

### 후순위 요소

- 상태 변경 이력
- 상태별 추천 피드백 요청

## 3.4 현재 필요한 것

### 영역의 목적

외부 사용자가 지금 어떻게 도울 수 있는지 보여준다.

### 왜 이 순서가 필요한가

BuildMap의 공개 페이지는 단순 소개가 아니라 피드백, 테스트, 협업으로 이어지는 입구다. 따라서 현재 필요한 것은 상단에 있어야 한다.

### 비개발자가 이해하기 쉽게 보여주는 방법

예시:

- 첫 화면 이해도 피드백 필요
- 문제 정의 공감 여부 확인 필요
- 테스터 5명 필요
- UX 피드백 필요
- 협업 디자이너 필요

### Builder가 공개 여부를 제어해야 하는 정보

- 모집 대상과 연락 가능 정보
- 아직 공개하고 싶지 않은 도움 요청

### 초기 구현 범위

- 현재 필요한 것 텍스트 또는 태그 표시
- 피드백 요청으로 이동

### 후순위 요소

- 테스터 신청 양식
- 협업자 요청 관리
- 멘토링 요청 관리

## 3.5 문제 정의

### 영역의 목적

프로젝트가 어떤 문제에서 시작했는지 보여준다.

### 왜 이 순서가 필요한가

제품 설명보다 문제 정의가 먼저 보여야 프로젝트의 판단 흐름을 이해할 수 있다.

### 비개발자가 이해하기 쉽게 보여주는 방법

- 기술 용어보다 사용자 상황 중심
- “무엇을 만들었다”보다 “어떤 문제가 반복되었다” 중심

### Builder가 공개 여부를 제어해야 하는 정보

- 내부 고객명
- 민감한 인터뷰 내용
- 아직 공개하지 않을 문제 정의

### 초기 구현 범위

- 현재 문제 정의 노출

### 후순위 요소

- 문제 정의 변경 이력
- 이전 문제 정의와 현재 문제 정의 비교

## 3.6 현재 가설

### 영역의 목적

프로젝트가 지금 무엇을 믿고 검증 중인지 보여준다.

### 왜 이 순서가 필요한가

현재 가설을 알아야 변화 카드와 실험의 의미를 이해할 수 있다.

### 비개발자가 이해하기 쉽게 보여주는 방법

- 기능 목록이 아니라 검증할 판단으로 표현
- 상태 표시
  - 가정
  - 검증 중
  - 일부 확인
  - 확인됨
  - 반박됨
  - 보류

### Builder가 공개 여부를 제어해야 하는 정보

- 공개하면 경쟁상 민감할 수 있는 가설
- 검증 전 내부 가설

### 초기 구현 범위

- 핵심 가설 1~3개 표시

### 후순위 요소

- 가설별 Timeline 연결
- 가설 상태 변경 이력

## 3.7 최근 전환

### 영역의 목적

프로젝트가 최근 왜 바뀌었는지 보여준다.

### 왜 이 순서가 필요한가

최근 전환은 프로젝트가 살아 움직이고 있음을 보여주는 핵심 정보다.

### 비개발자가 이해하기 쉽게 보여주는 방법

- “기능 추가”보다 “왜 바꿨는지”를 짧게 표시
- 전환 전/후를 나란히 보여줄 수 있음

### Builder가 공개 여부를 제어해야 하는 정보

- 공개 가능한 전환 카드만 노출

### 초기 구현 범위

- 최근 핵심 전환 카드 1~3개 표시

### 후순위 요소

- Decision Diff 연결
- 전환 흐름 자동 요약

## 3.8 프로젝트 진화도

### 영역의 목적

프로젝트가 처음 문제에서 현재 상태까지 어떻게 변화했는지 한눈에 보여준다.

### 왜 이 순서가 필요한가

긴 글 목록을 읽지 않아도 프로젝트의 흐름을 이해할 수 있어야 한다.

### 비개발자가 이해하기 쉽게 보여주는 방법

- 핵심 변화 카드만 연결
- 문제 발견 → 가설 → 피드백 → 판단 → 전환 구조로 표시
- 기술 변경보다 판단 변경을 우선 표시

### Builder가 공개 여부를 제어해야 하는 정보

- 공개할 변화 카드
- 숨길 내부 카드

### 초기 구현 범위

- 핵심 변화 카드 기반 간단한 Timeline 또는 카드 흐름

### 후순위 요소

- 그래프형 진화도
- 인터랙티브 흐름
- 인수인계 요약

## 3.9 공개 변화 카드

### 영역의 목적

공개 가능한 의사결정 기록을 보여준다.

### 왜 이 순서가 필요한가

프로젝트 진화도에서 관심이 생긴 사용자가 실제 판단 근거를 확인할 수 있다.

### 비개발자가 이해하기 쉽게 보여주는 방법

각 카드에 다음을 간결히 표시한다.

- 유형
- 제목
- 근거
- 판단
- 다음 확인 사항

### Builder가 공개 여부를 제어해야 하는 정보

- 공개 가능한 카드만 노출
- 내부 메모와 AI 초안은 공개하지 않음

### 초기 구현 범위

- 공개 상태의 승인된 변화 카드 목록
- 카드 유형 필터

### 후순위 요소

- 카드별 댓글
- 카드 공유
- 카드별 공개 범위 세분화

## 3.10 피드백 요청

### 영역의 목적

외부 사용자가 특정 질문에 피드백을 남기도록 유도한다.

### 왜 이 순서가 필요한가

피드백 요청은 프로젝트 전체 평가가 아니라 특정 판단에 연결되어야 한다.

### 비개발자가 이해하기 쉽게 보여주는 방법

질문 예시:

- 첫 화면에서 이 서비스가 무엇인지 이해되나요?
- 이 문제 정의에 공감되나요?
- 이 기능 제거 판단에 동의하나요?
- 테스터로 참여할 의향이 있나요?

### Builder가 공개 여부를 제어해야 하는 정보

- 질문 공개 여부
- 피드백 공개 여부

### 초기 구현 범위

- 피드백 요청 표시
- 간단한 피드백 작성

### 후순위 요소

- 피드백 유형 분류
- 테스터 신청과 연결
- Scout 신뢰도 표시

## 3.11 관련 링크

### 영역의 목적

외부 사용자가 실제 제품, GitHub, Notion, 데모, 설문 등으로 이동할 수 있게 한다.

### 왜 이 순서가 필요한가

BuildMap은 GitHub나 Notion을 대체하지 않는다. 판단 흐름은 BuildMap에서 보고, 세부 코드와 문서는 외부 링크에서 확인할 수 있다.

### 비개발자가 이해하기 쉽게 보여주는 방법

- 링크 목적 표시
  - 데모 보기
  - GitHub 보기
  - 문서 보기
  - 설문 참여

### Builder가 공개 여부를 제어해야 하는 정보

- 내부 문서 링크
- 비공개 저장소 링크

### 초기 구현 범위

- 관련 링크 목록

### 후순위 요소

- 링크 유형별 아이콘
- 외부 연동

## 3.12 Builder 정보

### 영역의 목적

프로젝트를 성장시키는 사람을 보여준다.

### 왜 이 순서가 필요한가

BuildMap은 이력서보다 프로젝트 기록이 사람을 증명하게 하는 플랫폼이다. Builder 정보는 프로젝트 판단 흐름 뒤에 배치되는 것이 적합하다.

### 비개발자가 이해하기 쉽게 보여주는 방법

- 이름 또는 닉네임
- 역할
- 관련 프로젝트
- 연락 가능 여부

### Builder가 공개 여부를 제어해야 하는 정보

- 개인 프로필 공개 수준
- 연락 가능 여부

### 초기 구현 범위

- 기본 Builder 정보

### 후순위 요소

- Builder 공개 프로필
- 복수 프로젝트 포트폴리오
- 팀 구성 표시

## 3.13 팔로우 또는 저장

### 영역의 목적

사용자가 프로젝트의 이후 변화를 계속 볼 수 있게 한다.

### 왜 이 순서가 필요한가

BuildMap의 대상은 완성된 제품이 아니라 진행 중인 프로젝트다. 따라서 사용자는 변화 자체를 팔로우할 수 있어야 한다.

### 비개발자가 이해하기 쉽게 보여주는 방법

- 새 변화 카드 알림
- 새 피드백 요청 알림
- 테스터 모집 알림
- 방향 전환 알림

### Builder가 공개 여부를 제어해야 하는 정보

- 알림 대상이 되는 공개 업데이트

### 초기 구현 범위

- 저장 또는 팔로우 버튼

### 후순위 요소

- 알림 설정
- 팔로워 관리
- Scout 저장 목록

## 4. 초기 구현 구조 요약

초기 공개 프로젝트 페이지는 다음을 우선한다.

```text
프로젝트명
→ 한 줄 정의
→ 현재 상태
→ 현재 필요한 것
→ 문제 정의
→ 현재 가설
→ 최근 전환
→ 프로젝트 진화도
→ 공개 변화 카드
→ 피드백 요청
→ 관련 링크
→ Builder 정보
→ 팔로우/저장
```

## 5. 추가 검토 필요

- 공개 프로젝트 페이지의 기본 공개 상태
- 공개 변화 카드의 최대 노출 개수
- 진화도와 Timeline의 관계
- 피드백 공개 여부
- Builder 연락 방식
- 팔로우 알림 범위
