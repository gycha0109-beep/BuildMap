# 유즈케이스 우선순위

## 개요

이 문서는 2단계에서 도출한 유즈케이스를 우선순위별로 나눈다.

우선순위는 구현 계획이 아니다. 이 문서는 어떤 흐름이 BuildMap의 정체성에 더 가까운지 판단하기 위한 제품 방향 문서다.

BuildMap의 중심은 Builder의 판단 흐름 기록이다. Scout, 히트맵, 채용, 평가 기능은 확장 가치이며 초기 핵심처럼 과장하지 않는다.

---

## 1. 핵심 유즈케이스

핵심 유즈케이스는 BuildMap의 존재 이유와 직접 연결된다.

- Builder의 첫 프로젝트 생성
- 문제 정의 기록
- 가설 기록
- 거친 메모 입력
- AI 변화 카드 초안 생성
- 변화 카드 승인
- Decision Timeline 반영
- 프로젝트 진화도 확인
- 공개 프로젝트 페이지 확인
- 피드백 요청
- 피드백 작성

### 판단 이유

이 흐름이 없으면 BuildMap은 단순 프로젝트 소개 서비스나 문서 관리 도구처럼 보인다. BuildMap의 중심은 거친 기록이 판단 단위로 구조화되고, 승인된 변화 카드가 Decision Timeline에 쌓이는 것이다.

---

## 2. 초기 구현 후보

초기 구현 후보는 핵심 유즈케이스를 실제로 사용할 수 있게 만드는 주변 흐름이다.

- 내부 기록 후 공개 전환
- 간단한 Decision Diff
- 프로젝트 팔로우
- 기본 Scout 탐색
- 변화 카드 유형 필터
- 기본 히트맵 탐색

### 판단 이유

이 항목들은 핵심 흐름을 확장하지만, 없어도 BuildMap의 기본 정체성은 설명할 수 있다. 다만 공개 전환, 기본 탐색, 피드백 흐름은 Builder 기록이 외부 발견으로 이어지는 데 필요하다.

### 추가 검토 필요

- 초기 구현 후보 중 어느 범위까지 먼저 다룰지
- 기본 히트맵이 너무 인기 순위처럼 보이지 않게 하는 기준
- Decision Diff를 얼마나 단순하게 시작할지

---

## 3. 2차 확장 후보

2차 확장 후보는 BuildMap의 사용 폭을 넓히는 기능이다.

- 프로젝트 인수인계 모드
- 협업자 요청
- 테스터 매칭
- Scout 저장 목록
- 프로젝트 활동 지표
- 공개/비공개 카드 세부 제어

### 판단 이유

이 항목들은 BuildMap의 차별점을 강화한다. 특히 프로젝트 인수인계는 장기적으로 중요하다. 다만 2단계에서는 복잡한 전용 기능이 아니라 유즈케이스 수준으로만 확정한다.

---

## 4. 장기 확장 후보

장기 확장 후보는 BuildMap이 충분한 기록과 사용자 흐름을 확보한 뒤 검토할 수 있다.

- 채용/헤드헌팅
- 투자자 탐색
- AI 기반 고급 요약
- 외부 GitHub/Notion 연동
- 외부 트래픽 연동
- 고급 히트맵 지표
- 팀 플랜
- 조직용 온보딩 기능

### 판단 이유

이 항목들은 BuildMap의 확장 가능성을 보여주지만, 지금 핵심으로 잡으면 제품이 채용 플랫폼, 분석 도구, 협업툴처럼 흐려질 위험이 있다.

---

## 5. 이번 단계에서 제외할 것

이번 단계에서는 다음을 제외한다.

- DB 설계
- 화면 와이어프레임
- API 설계
- 가격 정책
- 결제
- 권한 상세 설계
- 점수화된 역량 평가
- Project DNA 브랜딩
- 실제 채용 플로우
- 복잡한 추천 알고리즘

### 제외 이유

아직 BuildMap의 핵심은 기능 범위 확정이 아니라 유즈케이스 방향 확정이다. 지금 너무 많은 기능을 확정하면 BuildMap이 GitHub, Notion, Product Hunt, LinkedIn, 블로그, 채용 플랫폼의 조합처럼 보일 위험이 있다.

BuildMap의 독립적 위치는 프로젝트의 판단 흐름을 비개발자도 이해할 수 있는 Decision Timeline과 프로젝트 진화도로 보여주는 것이다.
