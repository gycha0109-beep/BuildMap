# 프로젝트 인수인계 유즈케이스

## 개요

프로젝트 인수인계는 BuildMap의 중요한 차별점이다.

GitHub에는 코드가 있고, Notion에는 문서가 있어도, 새 팀원이나 협업자가 가장 먼저 궁금해하는 것은 “왜 이렇게 되었는가”다. BuildMap은 Decision Timeline과 변화 카드를 통해 프로젝트의 운영 맥락을 전달한다.

이번 단계에서는 인수인계 전용 복잡한 기능을 구현 범위로 확정하지 않는다. 다만 프로젝트 인수인계를 BuildMap의 핵심 유즈케이스 후보로 문서화한다.

---

## UC-H01. 새 팀원이 프로젝트 맥락을 이해하는 흐름

### 상황

새 팀원이 프로젝트에 합류했다. GitHub와 Notion은 있지만 왜 그런 판단을 했는지 한눈에 이해하기 어렵다.

### 유즈케이스 설명

새 팀원은 Decision Timeline을 먼저 본다.

확인해야 할 정보는 다음과 같다.

- 처음 문제 정의
- 주요 가설
- 핵심 전환
- 기능 제거 이유
- 현재 필요한 것
- 최근 판단 변경
- 다음에 확인해야 할 질문

목표는 5분 안에 프로젝트의 큰 흐름을 이해하는 것이다.

세부 코드는 GitHub에서 확인하고, 문서는 Notion에서 확인할 수 있다. 하지만 판단 흐름은 BuildMap에서 본다.

### Decision Timeline과의 연결

새 팀원은 시간 순서대로 모든 기록을 읽는 것이 아니라 핵심 변화 카드가 연결된 프로젝트 진화도를 먼저 확인한다. 이후 필요한 경우 특정 변화 카드의 원문, 근거, 관련 가설을 더 깊게 본다.

### 추가 검토 필요

- 새 팀원용 요약 보기 필요 여부
- 5분 요약에 포함할 핵심 카드 선정 기준
- 비공개 내부 기록 접근 권한

## UC-H02. 외부 협업자가 프로젝트를 빠르게 파악하는 흐름

### 상황

디자이너, 개발자, 마케터, 멘토가 프로젝트에 잠깐 참여하려 한다.

### 유즈케이스 설명

협업자는 프로젝트 소개가 아니라 현재 판단 흐름을 본다.

확인할 정보는 다음과 같다.

- 현재 어떤 문제가 검증 중인지
- 어떤 피드백이 필요한지
- 최근에 어떤 방향 전환이 있었는지
- 어떤 기능이 추가되거나 제거되었는지
- 자신의 기여가 어디에 필요한지

협업자는 BuildMap을 통해 프로젝트의 현재 맥락을 빠르게 파악하고, 자신의 기여 위치를 판단한다.

### 예시

UX 디자이너가 프로젝트를 도우려 한다면 단순히 화면 목록을 보는 것보다 다음을 먼저 이해해야 한다.

> 왜 첫 화면에서 장기 목표 입력을 제거했는가?

이 판단을 이해해야 이후 화면 제안이 프로젝트 방향과 맞는다.

### 추가 검토 필요

- 외부 협업자에게 공개할 카드 범위
- 협업 요청과 현재 필요한 것의 연결 방식
- 멘토/협업자용 읽기 전용 링크 필요 여부

## UC-H03. 프로젝트를 이어받는 경우

### 상황

기존 Builder가 빠지거나, 다른 사람이 프로젝트를 이어받는다.

### 유즈케이스 설명

Decision Timeline은 프로젝트의 운영 맥락을 전달한다.

이어받는 사람은 다음을 확인할 수 있어야 한다.

- 왜 특정 기능이 제거되었는가
- 왜 특정 기술이나 방향을 선택했는가
- 어떤 가설은 확인되었고 어떤 가설은 반박되었는가
- 어떤 피드백이 아직 반영되지 않았는가
- 현재 프로젝트가 어떤 판단 위에 서 있는가

이 유즈케이스는 장기적으로 중요하지만 초기 구현에서는 핵심 문서화 수준으로 다룬다.

### Decision Timeline과의 연결

프로젝트를 이어받는 사람은 코드나 문서보다 먼저 판단 흐름을 이해해야 한다. Decision Timeline은 프로젝트가 단순 파일 묶음이 아니라 누적된 판단의 결과임을 보여준다.

### 추가 검토 필요

- 프로젝트 소유권 이전 흐름
- 기록 작성자 표시 방식
- 이전 Builder와 새 Builder의 판단 구분
- 인수인계 메모의 필수 항목

## UC-H04. 멘토 또는 심사위원이 프로젝트를 평가하는 경우

### 상황

학원 프로젝트, 해커톤, 창업 프로그램에서 멘토나 심사위원이 프로젝트를 평가한다.

### 유즈케이스 설명

멘토는 결과물만 보는 것이 아니라 프로젝트가 어떤 판단을 거쳐 발전했는지 본다.

확인할 정보는 다음과 같다.

- 문제 정의가 바뀐 이유
- 사용자 피드백 반영 여부
- 기능 제거 판단
- 실험 반복 여부
- 현재 검증 중인 가설
- 프로젝트가 지금 필요로 하는 도움

단순 결과물보다 문제 해결 과정의 성숙도를 볼 수 있다.

### 원칙

점수화 기능은 아직 다루지 않는다. BuildMap은 멘토나 심사위원에게 점수를 자동 산출해주는 서비스가 아니다. 대신 프로젝트 판단 기록에서 확인 가능한 근거를 보여준다.

### 추가 검토 필요

- 멘토용 요약 보기
- 심사위원에게 공개할 정보 범위
- 평가 코멘트를 변화 카드나 피드백으로 연결할지 여부
