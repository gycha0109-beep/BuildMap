# 변화 카드 시나리오

## 개요

BuildMap이 Notion이나 블로그와 달라지는 핵심은 변화 카드가 실제 의사결정 단위로 작동하는지에 달려 있다.

변화 카드는 단순 로그가 아니다. 변화 카드는 프로젝트에서 발생한 하나의 의사결정 단위다. 하나의 변화 카드는 문제, 가설, 근거, 판단, 변경, 다음 확인 사항을 연결한다.

AI는 변화 카드를 자동으로 확정하지 않는다. AI는 Builder의 원문 입력을 구조화하고, Builder가 최종 수정·승인한다.

---

## SC-01. 사용자 피드백으로 문제 정의가 바뀌는 경우

### 상황

Builder가 사용자 인터뷰를 통해 처음 생각한 문제 정의가 실제 사용자 어려움과 다르다는 것을 발견한다.

### Builder의 원문 입력

> 처음엔 사용자가 장기 목표를 못 세워서 문제라고 생각했다. 그런데 인터뷰해보니 목표는 있는데 이번 주에 뭘 해야 할지 몰라서 멈추는 경우가 많았다.

### AI 구조화 결과

유형:

> 문제 정의 수정

기존 문제:

> 사용자는 장기 목표를 세우지 못한다.

새 문제:

> 사용자는 장기 목표와 이번 주 실행 단위를 연결하지 못한다.

근거:

> 인터뷰에서 장기 목표보다 이번 주 실행 단위에 대한 어려움이 반복적으로 나타났다.

판단:

> 서비스의 중심을 장기 목표 생성에서 이번 주 실행 증거 설정으로 이동한다.

### Builder가 확인해야 할 점

- 인터뷰 수와 대상이 충분한지
- 문제 정의 변경이 너무 성급하지 않은지
- 기존 문제를 완전히 버리는지, 하위 문제로 남기는지
- 새 문제 정의가 공개 가능한 표현인지

### Decision Timeline에 반영되는 방식

이 변화 카드는 초기 문제 정의와 새 문제 정의 사이의 전환점으로 기록된다. 프로젝트 진화도에서는 문제 축이 바뀐 핵심 카드로 강조될 수 있다.

### 공개 가능 여부

공개 가능성이 높다. 다만 인터뷰 대상자의 개인정보나 구체 발언은 익명화해야 한다.

### 추가 검토 필요 사항

- 문제 정의 변경의 중요도 기준
- 인터뷰 근거를 어느 수준까지 공개할지

## SC-02. 가설이 반박되는 경우

### 상황

Builder가 테스트를 통해 핵심 가설이 예상과 다르게 작동한다는 것을 확인한다.

### Builder의 원문 입력

> 10년 후 목표를 먼저 쓰게 하면 동기부여가 될 줄 알았는데, 테스트한 사람들은 오히려 부담스럽다고 했다.

### AI 구조화 결과

유형:

> 가설 반박

가설:

> 10년 후 목표를 먼저 쓰면 실행 동기가 높아진다.

근거:

> 테스트 참여자들이 장기 목표 입력을 부담스럽게 느꼈다.

판단:

> 장기 목표 입력을 첫 화면에서 제거한다.

다음 확인:

> 이번 주 목표를 먼저 보여주면 사용자가 더 쉽게 시작하는지 확인한다.

### Builder가 확인해야 할 점

- 가설을 완전히 반박된 것으로 볼지, 일부 사용자군에서만 반박된 것으로 볼지
- 장기 목표 입력을 제거할지, 뒤 단계로 이동할지
- 테스트 참여자의 특성이 결과에 영향을 주었는지

### Decision Timeline에 반영되는 방식

가설 상태가 “검증 중”에서 “반박됨” 또는 “일부 반박”으로 변경된다. 연결된 화면 변경이나 우선순위 변경이 후속 변화 카드로 이어질 수 있다.

### 공개 가능 여부

공개 가능하다. BuildMap에서는 반박된 가설도 중요한 성장 기록이다.

### 추가 검토 필요 사항

- 가설 상태 분류의 세부 기준
- 반박된 가설을 공개 페이지에서 부정적으로 보이지 않게 설명하는 방식

## SC-03. 기능을 제거하는 경우

### 상황

Builder가 사용 결과를 보고 기능을 추가하는 대신 제거한다.

### Builder의 원문 입력

> AI가 3개월 계획을 자동으로 만들어주는 기능을 뺐다. 계획이 너무 길어서 아무도 끝까지 안 읽었다.

### AI 구조화 결과

유형:

> 기능 제거

제거한 기능:

> AI 3개월 자동 계획 생성

이유:

> 결과물이 너무 길고 실행으로 이어지지 않았다.

영향받은 가설:

> 자세한 계획이 실행을 돕는다는 가설이 약해졌다.

새 방향:

> 짧은 주간 실험 단위로 변경한다.

### Builder가 확인해야 할 점

- 기능 제거의 근거가 충분한지
- 기능을 완전 제거할지, 보류할지, 축소할지
- 제거 결정이 다른 기능 흐름에 미치는 영향
- 공개 시 사용자에게 부정적으로 보이지 않도록 설명할 수 있는지

### Decision Timeline에 반영되는 방식

기능 제거 카드는 프로젝트가 무언가를 덜어낸 판단으로 기록된다. 이는 프로젝트 진화도에서 방향을 좁히고 명확히 한 중요한 변화로 표시될 수 있다.

### 공개 가능 여부

공개 가능하다. 기능 제거는 성숙한 판단의 근거가 될 수 있다.

### 추가 검토 필요 사항

- 제거와 보류의 구분
- 제거된 기능의 재검토 날짜 또는 조건

## SC-04. 피드백을 반영하지 않기로 결정하는 경우

### 상황

Builder가 사용자 제안을 받았지만 현재 핵심 흐름과 맞지 않아 보류한다.

### Builder의 원문 입력

> 한 사용자가 캘린더 연동이 필요하다고 했지만, 지금은 핵심이 아니라고 판단했다. 아직 문제 정의 검증이 먼저다.

### AI 구조화 결과

유형:

> 판단 유지 / 기능 보류

제안:

> 캘린더 연동

보류 이유:

> 현재 단계에서는 문제 정의와 핵심 흐름 검증이 우선이다.

판단:

> 캘린더 연동은 후순위로 보류한다.

다음 확인:

> 반복적으로 같은 요청이 나오는지 관찰한다.

### Builder가 확인해야 할 점

- 피드백을 무시한 것이 아니라 보류한 이유가 명확한지
- 반복 요청이 발생했을 때 다시 검토할 기준이 있는지
- 제안한 사용자에게 어떤 방식으로 설명할지

### Decision Timeline에 반영되는 방식

피드백이 있었지만 반영하지 않은 판단도 기록된다. 이는 프로젝트가 요구사항을 무조건 수용하지 않고 우선순위를 관리했다는 근거가 된다.

### 공개 가능 여부

부분 공개가 적합하다. 특정 사용자 정보는 제외하고, 제안과 보류 이유만 공개할 수 있다.

### 추가 검토 필요 사항

- 피드백 미반영 기록의 공개 범위
- 보류된 기능의 재검토 트리거

## SC-05. 방향 전환이 발생하는 경우

### 상황

Builder가 프로젝트의 타깃 사용자나 문제 영역이 바뀌었다고 판단한다.

### Builder의 원문 입력

> 처음엔 자기계발 플래너라고 생각했는데, 점점 창업자나 사이드프로젝트 제작자에게 더 맞는 느낌이다.

### AI 구조화 결과

유형:

> 방향 전환

기존 방향:

> 일반 자기계발 플래너

새 방향:

> 창업자와 사이드프로젝트 제작자를 위한 실행 증거 플래너

근거:

> 사용자 반응과 서비스 흐름이 일반 목표 관리보다 프로젝트 실행에 더 적합했다.

판단:

> 타깃 사용자와 랜딩 메시지를 프로젝트 실행 중심으로 수정한다.

### Builder가 확인해야 할 점

- 실제 사용자 반응이 방향 전환을 뒷받침하는지
- 기존 사용자를 완전히 버리는지, 하위 세그먼트로 남기는지
- 랜딩 메시지, 기능 우선순위, 피드백 요청이 함께 바뀌어야 하는지

### Decision Timeline에 반영되는 방식

방향 전환 카드는 프로젝트 진화도에서 가장 중요한 분기점 중 하나다. 이전 판단과 새 판단을 Decision Diff로 비교할 수 있다.

### 공개 가능 여부

공개 가능성이 높다. 방향 전환은 외부 사용자가 프로젝트 맥락을 이해하는 데 중요하다.

### 추가 검토 필요 사항

- 방향 전환과 단순 메시지 수정의 구분
- 방향 전환 카드의 강조 기준

## SC-06. 릴리즈가 발생하는 경우

### 상황

Builder가 새 버전을 배포했지만, BuildMap에서는 버전 번호보다 그 버전의 판단 목적을 기록한다.

### Builder의 원문 입력

> v0.2를 배포했다. 이번 버전은 목표 설정보다 변화 카드 작성 흐름을 먼저 테스트하려고 만들었다.

### AI 구조화 결과

유형:

> 릴리즈

릴리즈 내용:

> v0.2 배포

핵심 목적:

> 변화 카드 작성 흐름 검증

포함된 변경:

> 목표 설정 단순화, 변화 카드 입력 추가

다음 확인:

> 사용자가 거친 메모를 변화 카드로 전환하는 과정을 이해하는지 확인한다.

### Builder가 확인해야 할 점

- 릴리즈의 목적이 명확한지
- 단순 배포 기록이 아니라 어떤 판단을 확인하기 위한 배포인지
- 다음 확인 사항이 테스트 가능한 형태인지

### Decision Timeline에 반영되는 방식

릴리즈 카드는 버전 기록이 아니라 판단 흐름의 결과로 반영된다. 릴리즈에 포함된 변경은 관련 가설, 실험, 피드백 카드와 연결될 수 있다.

### 공개 가능 여부

공개 가능하다. 공개 프로젝트 페이지에서 최근 업데이트로 표시될 수 있다.

### 추가 검토 필요 사항

- 릴리즈 카드와 기능 추가 카드의 구분
- 릴리즈 노트와 Decision Timeline의 관계

## SC-07. 인수인계 메모가 필요한 경우

### 상황

새 팀원이 프로젝트에 합류했을 때 과거 기술 선택이나 방향 선택의 이유를 빠르게 이해해야 한다.

### Builder의 원문 입력

> 새 팀원이 들어오면 왜 Firebase를 안 쓰고 Supabase를 선택했는지 설명해야 한다. 예전에 Firebase Auth와 DB 구조 때문에 고민하다가 Supabase로 바꿨다.

### AI 구조화 결과

유형:

> 인수인계 메모

주제:

> Firebase에서 Supabase로 전환한 이유

기존 선택:

> Firebase

새 선택:

> Supabase

전환 이유:

> 인증과 DB 구조를 프로젝트 요구사항에 맞게 더 단순하게 관리하기 위해

관련 판단:

> 초기 구현 속도와 데이터 구조 통제 가능성을 우선했다.

새 팀원이 알아야 할 것:

> 현재 인증과 DB 흐름은 Supabase 기준으로 설계되어 있다.

### Builder가 확인해야 할 점

- 기술 선택 이유가 과거 맥락에 맞게 정확한지
- 새 팀원이 알아야 할 현재 기준이 명확한지
- 민감한 내부 사유나 보안 정보가 포함되어 있지 않은지

### Decision Timeline에 반영되는 방식

인수인계 메모는 기술 선택 변경이나 운영 판단의 이유를 설명하는 카드로 기록된다. 새 팀원은 이 카드를 통해 현재 구조가 어떤 판단에서 나왔는지 이해한다.

### 공개 가능 여부

부분 공개가 적합하다. 기술 선택의 큰 이유는 공개 가능하지만, 보안 정보나 내부 세부 구조는 비공개로 남겨야 한다.

### 추가 검토 필요 사항

- 인수인계 메모의 공개/비공개 기본값
- 기술 선택 기록과 보안 정보의 경계
- 인수인계 전용 보기 모드 필요 여부
