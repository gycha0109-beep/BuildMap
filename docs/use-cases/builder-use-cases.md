# Builder 유즈케이스

## 개요

Builder는 프로젝트를 만든 사람이 아니라 프로젝트를 성장시키는 사람이다.

창업자, 학생 메이커, 개발자, 디자이너, PM, 비개발자, 바이브코딩 유저, 사이드프로젝트 제작자는 모두 Builder가 될 수 있다. 중요한 것은 직함이 아니라 프로젝트를 진행하며 문제를 발견하고, 가설을 세우고, 실험하고, 피드백을 반영하고, 판단을 바꾸는 행위다.

Builder 유즈케이스의 중심은 완성된 포트폴리오 등록이 아니다. 중심은 프로젝트가 현재 모습에 도달하기까지의 판단 흐름을 쌓는 것이다.

---

## UC-B01. Builder 가입 및 역할 선택

### 상황

사용자는 프로젝트를 만들거나 성장시키는 사람으로 BuildMap을 사용하려 한다. 사용자는 개발자일 수도 있고, 비개발자일 수도 있고, 바이브코딩 유저일 수도 있다.

### 유즈케이스 설명

Builder는 프로젝트를 만든 사람이라는 좁은 의미가 아니다. Builder는 프로젝트를 성장시키는 사람이다.

역할 선택은 사용자의 신분을 고정하기 위한 절차가 아니다. 역할은 탐색, 추천, 프로필 구성, 프로젝트 맥락 이해를 돕기 위한 메타데이터다.

선택 가능한 유형 예시는 다음과 같다.

- 창업자
- 학생 메이커
- 개발자
- 디자이너
- PM
- 비개발자
- 바이브코딩 유저
- 사이드프로젝트 제작자

### 원칙

초기 단계에서는 과도한 검증이나 인증을 요구하지 않는다. BuildMap은 사용자가 어떤 자격을 갖고 있는지보다 어떤 프로젝트 판단 흐름을 남기는지를 우선한다.

### Decision Timeline과의 연결

가입 자체는 Decision Timeline의 핵심 기록이 아니다. 다만 Builder의 역할 정보는 이후 프로젝트 페이지에서 “이 프로젝트를 어떤 관점의 사람이 성장시키고 있는지”를 이해하는 보조 맥락이 된다.

### 추가 검토 필요

- 역할을 복수 선택할 수 있는지
- 역할을 프로젝트별로 다르게 설정할 수 있는지
- 역할 표시가 공개 페이지에 노출되는 범위

## UC-B02. 첫 프로젝트 생성

### 상황

Builder는 아직 프로젝트가 완성되지 않았지만, 아이디어와 문제의식을 기록하기 시작하려 한다.

### 입력 정보 후보

첫 프로젝트 생성에서 받을 수 있는 정보는 다음과 같다.

- 프로젝트명
- 한 줄 설명
- 현재 상태
- 해결하려는 문제
- 아직 확정되지 않은 가설
- 현재 필요한 도움
- 프로젝트 공개 여부

### 핵심 원칙

첫 프로젝트 생성은 포트폴리오 등록이 아니다.

완성작을 올리는 것이 아니라 판단 흐름을 쌓기 시작하는 행위다. 따라서 처음부터 긴 문서 작성을 요구하지 않는다. Builder가 프로젝트를 아직 명확히 설명하지 못하더라도, 문제의식과 현재 상태를 짧게 남길 수 있어야 한다.

### Decision Timeline과의 연결

첫 프로젝트 생성 시점의 문제의식, 현재 상태, 초기 가설은 이후 Decision Timeline의 출발점이 된다.

### 추가 검토 필요

- 필수 입력을 어디까지 최소화할지
- 공개 여부 기본값을 어떻게 둘지
- 프로젝트 상태 값의 구체 분류

## UC-B03. 문제 정의 기록

### 상황

Builder는 프로젝트가 어떤 문제에서 시작했는지 기록한다.

### 유즈케이스 설명

문제 정의는 프로젝트의 중심축이다. BuildMap에서 프로젝트는 기능 목록이 아니라 해결하려는 문제와 그 문제에 대한 판단 흐름으로 이해된다.

문제 정의는 한 번 쓰고 끝나는 항목이 아니다. 프로젝트가 진행되면서 사용자 반응, 인터뷰, 실험 결과에 따라 수정될 수 있다. 문제 정의가 수정되면 왜 수정되었는지 기록되어야 한다.

문제 정의 변경은 중요한 변화 카드가 될 수 있다.

### 예시

초기 문제 정의:

> 사람들은 장기 목표를 세우지만 오늘 할 일과 연결하지 못한다.

수정된 문제 정의:

> 사람들은 장기 목표 자체보다 이번 주에 무엇을 증명해야 하는지 몰라 실행이 끊긴다.

변경 이유:

> 초기 인터뷰에서 장기 계획보다 당장 실행 가능한 단위가 더 중요하다는 반응이 나왔다.

### Decision Timeline과의 연결

문제 정의 변경은 프로젝트 방향의 핵심 전환이다. 이전 문제 정의와 새 문제 정의, 변경 근거가 함께 기록되어야 한다.

### 추가 검토 필요

- 문제 정의의 버전 관리 방식
- 이전 문제 정의를 숨길 수 있는지
- 공개 페이지에서 문제 정의 변경 이력을 어떻게 요약할지

## UC-B04. 가설 기록 및 상태 관리

### 상황

Builder는 프로젝트가 성립하려면 사실이어야 하는 가설을 기록한다.

### 유즈케이스 설명

가설은 기능 목록이 아니다. 가설은 프로젝트가 맞다고 믿고 있는 판단이다.

예를 들어 “알림 기능 추가”는 기능 목록이고, “사용자는 실행을 미루는 이유를 알림 부족으로 느낀다”는 가설이다.

가설 상태 예시는 다음과 같다.

- 가정
- 검증 중
- 일부 확인
- 확인됨
- 반박됨
- 보류

가설은 실험, 피드백, 변화 카드와 연결되어야 한다. 가설이 반박되거나 수정되면 그 자체가 중요한 판단 기록이 된다.

### 예시

가설:

> 사용자는 긴 목표 계획보다 이번 주에 확인할 수 있는 증거 단위를 더 쉽게 이해한다.

상태:

> 검증 중

연결된 변화 카드:

> 첫 화면에서 10년 계획을 제거하고 이번 주 증거를 먼저 보여준 결정

### 추가 검토 필요

- 가설 상태의 최종 분류
- 가설과 변화 카드의 연결 방식
- 반박된 가설을 공개 페이지에 어떻게 보여줄지

## UC-B05. 거친 메모 입력

### 상황

Builder는 회의 후, 사용자 테스트 후, 작업 중에 짧은 메모를 남긴다.

### 유즈케이스 설명

Builder는 처음부터 완성된 회고문을 쓰지 않는다. 실제 프로젝트 진행 중의 기록은 대체로 짧고 거칠다.

입력 예시는 다음과 같다.

- 회의 메모
- 사용자 테스트 결과
- 사용자 반응
- 기능 변경 이유
- 피드백 요약
- 방향 전환을 고민한 이유
- 다음에 확인해야 할 질문

이 메모는 곧바로 공개되지 않는다. 원문은 AI 구조화 전 기록으로 보관될 수 있다.

### 예시 입력

> 테스터 3명이 첫 화면이 너무 복잡하다고 했다. 10년 후 목표를 먼저 쓰라고 하니까 부담스럽다고 했다. 그래서 첫 화면에서는 이번 주 목표부터 고르게 바꾸는 게 맞을 것 같다.

### Decision Timeline과의 연결

거친 메모 자체는 공식 변화 카드가 아니다. AI 구조화와 Builder 승인 과정을 거친 뒤 공식 Decision Timeline에 반영된다.

### 추가 검토 필요

- 원문 메모 보관 여부
- 원문과 구조화 카드의 연결 방식
- 음성, 이미지, 회의록 입력 지원 여부

## UC-B06. AI 변화 카드 초안 생성

### 상황

Builder가 남긴 거친 메모를 AI가 변화 카드 초안으로 구조화한다.

### 유즈케이스 설명

AI는 내용을 꾸미지 않는다. 과장된 성과를 만들지 않는다. Builder의 기록을 문제, 근거, 판단, 변경, 다음 확인 사항으로 정리한다.

AI가 생성하는 것은 공식 기록이 아니라 초안이다. Builder는 AI 초안을 수정하고 승인해야 한다. 승인 전 변화 카드는 공식 Decision Timeline에 반영되지 않는다.

### 예시 구조화

유형:

> 사용자 피드백 / 판단 수정

문제:

> 첫 화면의 장기 목표 입력이 사용자에게 부담을 준다.

근거:

> 테스터 3명이 첫 화면이 복잡하고 부담스럽다고 반응했다.

판단:

> 초기 진입에서는 장기 계획보다 이번 주 실행 단위를 먼저 보여주는 것이 적합하다.

변경:

> 첫 화면에서 10년 후 목표 입력을 제거하고, 이번 주 목표 선택을 우선 배치한다.

다음 확인 사항:

> 수정된 첫 화면에서 사용자가 10초 안에 서비스 목적을 이해하는지 확인한다.

### 추가 검토 필요

- AI 초안의 필드 구성
- AI가 원문에 없는 내용을 얼마나 보수적으로 처리해야 하는지
- AI 구조화 실패 시 Builder가 직접 작성하는 흐름

## UC-B07. 변화 카드 승인 및 Decision Timeline 반영

### 상황

Builder가 AI 초안을 확인하고 변화 카드로 저장한다.

### 유즈케이스 설명

Builder가 최종 승인자다. AI가 만든 초안은 Builder의 확인 없이 공식 기록이 되지 않는다.

변화 카드는 Decision Timeline에 반영된다. 변화 카드 유형과 연결된 가설, 문제, 실험을 선택할 수 있다. 중요도가 높은 변화 카드는 프로젝트 진화도에서 강조될 수 있고, 중요도가 낮은 변화 카드는 접히거나 요약될 수 있다.

### Decision Timeline과의 연결

승인된 변화 카드는 프로젝트의 공식 판단 기록이 된다. 이 카드들은 나중에 프로젝트 진화도, 공개 프로젝트 페이지, 인수인계 흐름, Scout 탐색의 근거가 된다.

### 추가 검토 필요

- 승인 전 초안의 저장 상태
- 승인 후 수정 이력
- 중요도 기준
- 카드 공개 여부 기본값

---

## UC-B08. 기능 추가보다 기능 제거를 기록하는 흐름

### 상황

Builder가 사용자의 반응이나 실험 결과를 보고 기능을 제거한다.

### 유즈케이스 설명

BuildMap은 기능 추가만 기록하지 않는다. 기능 제거, 축소, 보류도 중요한 판단으로 기록한다.

많은 프로젝트에서 기능 제거는 기록되지 않는다. 하지만 기능을 제거한 이유는 프로젝트의 판단 수준을 보여주는 중요한 근거다.

### 예시

기능 제거:

> 자동 10년 계획 생성 기능 제거

이유:

> 사용자가 계획이 너무 거대하다고 느꼈다.

영향받은 가설:

> 장기 계획이 실행력을 높일 것이라는 가설이 일부 반박됨

새 방향:

> 이번 주 증거 중심 플래너로 전환

### Decision Timeline과의 연결

기능 제거 카드는 프로젝트가 무엇을 만들었는지보다 왜 만들지 않기로 했는지를 보여준다. 이는 프로젝트 진화도에서 중요한 방향 전환 근거가 된다.

### 추가 검토 필요

- 제거, 축소, 보류의 구분
- 기능 제거가 공개될 때의 표현 방식
- 제거된 기능을 나중에 재검토하는 방식

## UC-B09. 프로젝트 진화도 확인

### 상황

Builder는 프로젝트가 처음 문제에서 현재 상태까지 어떻게 변화했는지 한눈에 보고 싶다.

### 유즈케이스 설명

프로젝트 진화도는 긴 글 목록이 아니다. 핵심 변화 카드만 연결된 구조로 보여주는 프로젝트 변화 지도다.

프로젝트 진화도에는 다음 흐름이 드러나야 한다.

```text
문제 발견 → 가설 → 실험 → 피드백 → 판단 → 방향 전환 → 현재 상태
```

프로젝트 진화도는 Builder 자신뿐 아니라 외부 사용자도 이해할 수 있어야 한다.

### Decision Timeline과의 연결

Decision Timeline이 시간 순서의 판단 기록이라면, 프로젝트 진화도는 그중 핵심 전환을 요약해 보여주는 구조다.

### 추가 검토 필요

- 진화도에 어떤 카드를 자동 포함할지
- Builder가 직접 강조 카드를 선택할 수 있는지
- 공개 페이지에서 진화도를 어느 정도 요약할지

---

## UC-B10. Decision Diff 확인

### 상황

Builder는 프로젝트의 이전 판단과 현재 판단이 어떻게 달라졌는지 비교하고 싶다.

### 유즈케이스 설명

Decision Diff는 코드 diff가 아니다. Decision Diff는 문제 정의, 가설, 타깃 사용자, 핵심 기능, 우선순위, 성공 기준의 차이를 보여준다.

비교 대상은 반드시 v0.1과 v0.4 같은 버전일 필요가 없다. “초기 판단”과 “현재 판단”, “전환 전”과 “전환 후”를 비교할 수 있다.

### 예시

초기 판단:

> 사용자는 10년 후 목표를 먼저 정하면 실행력이 높아진다.

현재 판단:

> 사용자는 장기 목표보다 이번 주에 무엇을 증명해야 하는지를 먼저 이해해야 실행한다.

### Decision Timeline과의 연결

Decision Diff는 여러 변화 카드가 누적된 결과를 비교 가능한 형태로 요약한다. 이는 Builder의 회고, 공개 설명, 인수인계에 모두 쓰일 수 있다.

### 추가 검토 필요

- 비교 기준 선택 방식
- Diff 필드의 최종 범위
- 자동 요약과 수동 편집의 비율

---

## UC-B11. 피드백 요청

### 상황

Builder는 프로젝트의 특정 판단이나 화면, 문제 정의에 대해 피드백을 받고 싶다.

### 유즈케이스 설명

피드백 요청은 프로젝트 전체가 아니라 특정 질문에 연결되어야 한다.

좋은 피드백 요청은 “좋아 보이나요?”가 아니라 “이 문제 정의가 실제로 공감되는가?”, “첫 화면에서 서비스 목적이 이해되는가?”처럼 판단 단위와 연결된다.

피드백 요청 예시는 다음과 같다.

- 첫 화면 이해도
- 문제 정의 공감도
- 기능 필요성
- 테스터 모집
- 협업자 요청

피드백 요청은 공개 프로젝트 페이지와 연결될 수 있다.

### 추가 검토 필요

- 피드백 요청의 공개 범위
- 피드백을 변화 카드로 전환하는 기준
- 피드백 남용 방지 방식

## UC-B12. 내부 기록 후 공개 전환

### 상황

Builder는 처음에는 내부적으로 기록하다가, 어느 정도 정리된 뒤 프로젝트를 공개하려 한다.

### 유즈케이스 설명

BuildMap은 모든 기록을 처음부터 공개하도록 강제하지 않는다. 기본 흐름은 내부 기록 → 정리 → 공개 전환이다.

Builder는 공개 가능한 변화 카드만 선택할 수 있어야 한다. 민감한 내부 메모나 미확정 판단은 제외할 수 있어야 한다.

공개 프로젝트 페이지는 제품 소개보다 다음 정보를 우선 보여준다.

- 문제 정의
- 현재 가설
- 최근 전환
- 현재 필요한 것
- 핵심 변화 카드

### 추가 검토 필요

- 공개 전환 시 검토 체크리스트
- 카드별 공개/비공개 정책
- 공개 후 수정 이력 노출 범위

## UC-B13. Builder 대시보드 확인

### 상황

Builder는 자신의 프로젝트와 판단 흐름, 미해결 가설, 최근 피드백, 필요한 행동을 한 화면에서 확인한다.

### 유즈케이스 설명

대시보드는 단순 Todo 앱이 아니다. 할 일은 문제, 가설, 실험, 피드백, 결정과 연결되어야 한다.

대시보드는 “오늘 무엇을 해야 하는가”보다 “지금 어떤 판단이 비어 있는가”를 보여줘야 한다.

예를 들어 다음과 같은 질문을 보여줄 수 있다.

- 검증 중인 가설은 무엇인가?
- 최근 피드백 중 아직 판단으로 반영되지 않은 것은 무엇인가?
- 공개 가능한 변화 카드는 무엇인가?
- 현재 필요한 도움은 명확히 작성되어 있는가?
- 프로젝트의 최근 방향 전환은 설명 가능한가?

### 추가 검토 필요

- 대시보드 구성 요소
- Todo와 판단 흐름의 연결 방식
- Builder가 여러 프로젝트를 운영할 때의 우선순위 표시
