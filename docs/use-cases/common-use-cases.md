# 공통 유즈케이스

## 개요

공통 유즈케이스는 Builder와 Scout뿐 아니라 일반 방문자, 잠재 테스터, 협업자, 커뮤니티 참여자가 BuildMap의 공개 영역을 사용하는 상황을 다룬다.

이 문서의 중심은 탐색 기능 자체가 아니다. 중심은 공개된 Decision Timeline과 프로젝트 진화도를 통해 프로젝트가 왜 지금의 모습이 되었는지 이해하는 것이다.

---

## UC-C01. 프로젝트 히트맵 탐색

### 상황

사용자는 BuildMap에 올라온 프로젝트들을 한눈에 탐색한다.

### 유즈케이스 설명

히트맵은 주식 히트맵처럼 프로젝트 타일 크기와 색상으로 활동성을 보여줄 수 있다.

다만 초기에는 외부 트래픽 데이터를 사용하지 않는다. 플랫폼 내부에서 측정 가능한 활동 지표만 사용한다.

지표 후보는 다음과 같다.

- 최근 변화 카드 수
- 피드백 수
- 테스터 신청 수
- 팔로우 수
- 저장 수
- 공개 업데이트 수

히트맵은 단순 인기 순위가 아니다. 히트맵은 지금 살아 움직이는 프로젝트를 발견하는 입구다.

### Decision Timeline과의 연결

히트맵의 핵심은 “많이 본 프로젝트”가 아니라 “최근 판단 흐름이 활발한 프로젝트”를 발견하게 하는 것이다.

### 추가 검토 필요

- 타일 크기와 색상의 의미
- 활동성 지표 산정 방식
- 지표가 인기 경쟁처럼 왜곡되지 않도록 하는 기준
- 내부 지표 외 외부 트래픽을 나중에 연결할지 여부

## UC-C02. 공개 프로젝트 페이지 확인

### 상황

사용자가 프로젝트 타일이나 링크를 통해 공개 프로젝트 페이지에 들어온다.

### 유즈케이스 설명

공개 프로젝트 페이지의 첫 화면은 제품 소개보다 프로젝트 맥락을 보여준다.

우선 노출 정보 후보는 다음과 같다.

- 문제 정의
- 현재 가설
- 최근 전환
- 현재 필요한 것
- 핵심 변화 카드
- 프로젝트 진화도 요약

긴 글보다 요약된 프로젝트 진화도가 중요하다. 사용자는 프로젝트가 왜 지금 모습이 되었는지 빠르게 이해해야 한다.

### BuildMap의 차별점

일반 랜딩 페이지는 설득을 위해 제품의 장점과 결과를 앞세운다. BuildMap의 공개 프로젝트 페이지는 판단 흐름을 먼저 보여준다.

### 추가 검토 필요

- 공개 페이지의 정보 우선순위
- 비공개 카드가 있을 때 진화도 연결이 끊기지 않도록 표시하는 방식
- 공개 페이지 URL 정책

## UC-C03. 변화 카드에 피드백 작성

### 상황

사용자가 특정 변화 카드에 피드백을 남긴다.

### 유즈케이스 설명

피드백은 프로젝트 전체가 아니라 특정 판단에 연결될 수 있다.

예시는 다음과 같다.

- “이 문제 정의는 공감된다.”
- “이 가설은 약해 보인다.”
- “이 기능 제거 판단에 동의한다.”
- “이 전환은 타깃 사용자가 바뀐 것처럼 보인다.”
- “이 실험은 샘플 수가 부족해 보인다.”

이 피드백은 Builder의 다음 변화 카드로 이어질 수 있다.

### Decision Timeline과의 연결

피드백이 특정 변화 카드에 연결되면, 나중에 Builder가 “피드백을 반영한 판단 수정” 카드를 만들 수 있다. 이때 피드백은 근거가 된다.

### 추가 검토 필요

- 변화 카드별 댓글/피드백 구조
- 피드백 공개 범위
- 악성 피드백 관리
- Builder가 피드백을 채택하거나 보류하는 표시 방식

## UC-C04. 프로젝트 팔로우 및 업데이트 수신

### 상황

사용자는 관심 있는 프로젝트의 변화를 계속 보고 싶다.

### 유즈케이스 설명

팔로우 대상은 완성된 제품이 아니라 진행 중인 프로젝트다.

알림 대상 예시는 다음과 같다.

- 새 변화 카드
- 새 피드백 요청
- 테스터 모집
- 방향 전환
- 공개 업데이트
- 현재 필요한 것 변경

알림 기능의 구체 구현은 이번 단계에서 확정하지 않는다.

### Decision Timeline과의 연결

팔로우는 프로젝트의 결과물이 아니라 판단 흐름을 구독하는 행위다. 사용자는 프로젝트가 어떤 판단으로 변하는지 지속적으로 본다.

### 추가 검토 필요

- 알림 채널
- 알림 빈도
- 팔로우와 저장의 차이
- 비공개 카드가 있는 경우 알림 처리

## UC-C05. 프로젝트를 통한 사람 이해

### 상황

사용자는 Builder의 자기소개보다 프로젝트 진화 과정을 보고 그 사람을 이해한다.

### 유즈케이스 설명

BuildMap은 이력서보다 프로젝트 기록을 통해 사람을 보여준다. 다만 사람을 점수화하지 않는다.

프로젝트에서 드러나는 실제 행동과 판단 근거를 보여준다.

예시는 다음과 같다.

- Builder가 사용자 피드백을 들었는가
- 문제 정의를 수정할 수 있었는가
- 기능을 추가하는 것만 아니라 제거할 수 있었는가
- 반박된 가설을 인정했는가
- 현재 필요한 도움을 명확히 표현했는가
- 프로젝트 맥락을 다른 사람이 이해할 수 있게 남겼는가

### BuildMap의 차별점

이력서는 사람이 자신을 설명한다. BuildMap은 프로젝트가 사람을 증명하게 한다. 이 증명은 점수가 아니라 기록에서 드러나는 근거다.

### 추가 검토 필요

- Builder 프로필과 프로젝트 페이지의 연결 방식
- 개인 역량 평가처럼 보이지 않도록 하는 표현
- 여러 프로젝트를 가진 Builder의 대표 기록 선택 방식
