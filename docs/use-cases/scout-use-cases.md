# Scout 유즈케이스

## 개요

Scout는 단순 채용 담당자가 아니다.

초기 Scout는 프로젝트를 발견하고, 피드백을 주고, 테스터가 되고, 협업자를 찾고, 멘토링하거나 팀원을 탐색하는 사람이다. 채용 담당자와 헤드헌터는 장기 확장 사용자로 다룬다.

Scout 유즈케이스의 중심은 이력서 검색이 아니다. 중심은 프로젝트의 판단 흐름과 Builder의 실제 행동을 근거로 사람과 프로젝트를 발견하는 것이다.

---

## UC-S01. Scout 가입 및 역할 선택

### 상황

사용자는 프로젝트나 Builder를 발견하기 위해 BuildMap을 사용한다.

### 유즈케이스 설명

Scout는 다음과 같은 역할을 포함할 수 있다.

초기 Scout 유형:

- 테스터
- 초기 사용자
- 협업자
- 멘토
- 창업자
- 팀원 탐색자
- 커뮤니티 운영자

장기 Scout 유형:

- 채용 담당자
- 헤드헌터
- 투자자

초기 문서에서는 채용을 핵심으로 과장하지 않는다. BuildMap의 초기 발견 흐름은 피드백, 테스트, 협업, 멘토링에 더 가깝다.

### Decision Timeline과의 연결

Scout가 선택한 역할은 어떤 프로젝트를 발견하고 싶은지, 어떤 방식으로 참여하려는지 이해하는 보조 정보다. Scout의 핵심 행동은 프로젝트의 Decision Timeline을 읽고 맥락 기반으로 참여하는 것이다.

### 추가 검토 필요

- Scout와 Builder 역할을 동시에 가질 수 있는지
- Scout 유형을 공개 프로필에 표시할지
- 장기 Scout 유형의 권한 분리 여부

---

## UC-S02. 프로젝트 발견

### 상황

Scout는 완성된 제품이 아니라 진행 중인 프로젝트를 발견한다.

### 유즈케이스 설명

Scout는 프로젝트의 결과물보다 판단 흐름을 본다.

Scout는 다음을 확인한다.

- 프로젝트가 어떤 문제에서 시작했는가
- 어떤 가설을 검증하고 있는가
- 어떤 사용자 피드백을 받았는가
- 어떤 기능을 추가하거나 제거했는가
- 어떤 방향 전환을 거쳤는가
- 현재 어떤 도움이 필요한가

Scout는 프로젝트의 현재 필요한 것을 보고 피드백, 테스트, 협업 참여 여부를 결정한다.

### BuildMap의 차별점

일반 프로젝트 소개 페이지는 “무엇을 만들었는가”를 보여준다. BuildMap의 공개 프로젝트 페이지는 “왜 지금의 모습이 되었는가”를 보여준다.

### 추가 검토 필요

- 프로젝트 탐색 필터
- 공개 프로젝트 페이지의 기본 요약 구조
- 진행 중인 프로젝트와 완성 프로젝트의 구분 방식

---

## UC-S03. Builder 발견

### 상황

Scout는 이력서가 아니라 프로젝트 진화 과정을 통해 Builder를 발견한다.

### 유즈케이스 설명

Scout는 Builder의 학력이나 경력보다 프로젝트 기록에서 드러나는 행동을 본다.

예시는 다음과 같다.

- 문제 정의를 수정한 기록
- 사용자 피드백을 반영한 기록
- 기능을 제거한 판단
- 실험을 반복한 흔적
- 최근 업데이트
- 미확정 가설을 공개적으로 관리한 기록
- 협업자나 테스터에게 명확한 요청을 한 기록

초기에는 AI 역량 점수나 자동 등급을 사용하지 않는다. Scout가 보는 정보는 점수가 아니라 기록에서 드러나는 근거 중심 정보다.

예시 근거:

- 사용자 테스트 4회
- 피드백 반영 변화 카드 6개
- 최근 30일 업데이트 3회
- 기능 제거 판단 기록 있음

### 추가 검토 필요

- Builder 프로필에 어떤 근거 요약을 보여줄지
- 근거 요약이 점수처럼 오해되지 않도록 하는 표현 방식
- Scout가 Builder를 저장하거나 팔로우하는 방식

## UC-S04. 피드백 제공자로 참여

### 상황

Scout가 프로젝트의 특정 질문에 피드백을 남긴다.

### 유즈케이스 설명

피드백은 프로젝트 전체 평가가 아니라 특정 질문에 답하는 방식이 적합하다.

피드백 유형 예시는 다음과 같다.

- 공감
- 이해 안 됨
- 개선 제안
- 사용 경험 공유
- 테스트 참여 의사

Builder는 피드백이 실제 변화 카드로 이어졌는지 표시할 수 있다.

### Decision Timeline과의 연결

Scout의 피드백은 Builder의 판단 흐름에 영향을 줄 수 있다. Builder가 피드백을 반영하면 해당 피드백은 변화 카드의 근거로 연결될 수 있다.

### 추가 검토 필요

- 피드백의 공개 여부
- 피드백을 변화 카드 근거로 연결하는 방식
- 피드백 품질 관리

---

## UC-S05. 테스터로 참여

### 상황

Scout가 프로젝트의 테스터 모집 요청을 보고 참여한다.

### 유즈케이스 설명

테스터 신청은 프로젝트의 현재 필요한 것과 연결된다. 테스트 목적이 명확해야 한다.

테스트 요청 예시는 다음과 같다.

- 첫 화면 이해도 테스트
- 기능 필요성 테스트
- 문제 공감도 테스트
- 사용 흐름 혼란 지점 확인
- 특정 타깃 사용자 반응 확인

테스트 결과는 Builder가 변화 카드로 구조화할 수 있다.

### Decision Timeline과의 연결

테스터 참여 자체보다 중요한 것은 테스트 결과가 어떤 판단으로 이어졌는지다. BuildMap은 테스트 결과를 문제, 근거, 판단, 변경, 다음 확인 사항으로 연결한다.

### 추가 검토 필요

- 테스터 신청과 승인 흐름
- 테스트 결과 입력 방식
- 테스터 개인정보와 공개 범위

---

## UC-S06. 협업자 또는 팀원 후보로 접근

### 상황

Scout가 프로젝트의 방향과 판단 흐름을 보고 협업을 제안한다.

### 유즈케이스 설명

협업 제안은 단순 DM이 아니라 프로젝트 맥락 기반이어야 한다.

부적절한 접근 예시:

> 같이 해보고 싶습니다.

BuildMap에 더 적합한 접근 예시:

> 첫 화면에서 장기 목표 입력을 제거한 판단이 좋았습니다. 해당 전환 이후 사용자가 실제로 더 쉽게 시작하는지 UX 테스트를 도와보고 싶습니다.

초기 핵심은 채용 제안보다 협업 제안에 더 가깝다.

### Decision Timeline과의 연결

협업자는 프로젝트의 최근 판단 흐름을 읽고 자신의 기여 위치를 찾는다. 협업 제안은 특정 변화 카드, 피드백 요청, 현재 필요한 것과 연결될 수 있다.

### 추가 검토 필요

- 협업 제안 양식
- Builder의 수락/거절 흐름
- 제안 메시지가 공개되는지 비공개인지

---

## UC-S07. 장기 확장으로서의 채용/헤드헌팅

### 상황

나중에 채용 담당자나 헤드헌터가 BuildMap을 통해 Builder를 탐색한다.

### 유즈케이스 설명

이 기능은 초기 핵심이 아니라 장기 확장이다.

Scout가 보는 것은 이력서가 아니라 프로젝트 판단 기록이다. 점수화된 역량 평가보다 기록 기반 근거가 우선이다.

채용 관점에서 의미 있는 근거 예시는 다음과 같다.

- 문제 정의를 수정한 이유가 명확함
- 사용자 피드백을 기능 변경으로 연결함
- 기능 제거 판단을 기록함
- 가설 검증 상태를 관리함
- 프로젝트 인수인계를 위한 맥락 기록이 있음

### 원칙

채용 기능은 유즈케이스에 포함하되, 초기 구현 범위로 확정하지 않는다. BuildMap이 채용 플랫폼처럼 보이지 않도록 주의한다.

### 추가 검토 필요

- 채용 Scout의 권한
- Builder의 구직 의사 표시 여부
- Scout 검색과 개인정보 보호
- 기록 기반 근거가 점수처럼 오해되지 않도록 하는 UI 표현
