# Visibility와 State 모델

## 1. 기본 원칙

BuildMap의 기본 정책은 내부 기록 후 공개 전환이다. 따라서 모든 주요 객체는 공개 상태와 작업 상태를 분리해 생각해야 한다.

상태값은 제품 설계용 후보이며 DB enum으로 확정하지 않는다.

## 2. Project 공개 상태

| 상태 | 의미 | 변경 주체 | 영향 화면 | 초기 필요 |
|---|---|---|---|---|
| 비공개 | Builder 내부에서만 접근 가능 | Builder | Builder 대시보드, 내부 Timeline | 필요 |
| 링크 공개 | 링크를 가진 외부 사용자가 접근 가능 | Builder | Public Project Page | 후보 |
| 전체 공개 | 탐색 화면과 공개 페이지에 노출 가능 | Builder | Project Card Grid, 공개 페이지 | 필요 후보 |

추가 검토 필요: 링크 공개와 전체 공개의 검색 노출 차이를 명확히 해야 한다.

## 3. Change Card 공개 상태

| 상태 | 의미 | 변경 주체 | 영향 화면 | 초기 필요 |
|---|---|---|---|---|
| 내부 전용 | 공개 페이지에 노출하지 않음 | Builder | 내부 Timeline | 필요 |
| 공개 가능 | 공개 후보지만 아직 노출하지 않음 | Builder | 공개 전환 검토 | 필요 |
| 공개됨 | Public Project Page에 노출 | Builder | 공개 Timeline, 공개 페이지 | 필요 |
| 민감 정보 포함 | 공개 전 주의 필요 | Builder 또는 시스템 후보 | 공개 경고 | 후보 |

주의: Project가 전체 공개여도 내부 전용 Change Card는 공개되지 않는다.

## 4. AI Draft 상태

| 상태 | 의미 | 변경 주체 | 영향 화면 | 초기 필요 |
|---|---|---|---|---|
| 생성 중 | AI가 구조화 중 | 시스템 | AI 처리 중 화면 | 필요 |
| 생성됨 | 초안이 생성됨 | 시스템 | Change Card Review | 필요 |
| 수정 중 | Builder가 수정 중 | Builder | Review 화면 | 필요 |
| 승인됨 | Change Card로 전환됨 | Builder | Timeline 반영 | 필요 |
| 보류됨 | 공식 기록으로 반영하지 않음 | Builder | 보류 목록 후보 | 필요 후보 |
| 실패 | AI 구조화 실패 | 시스템 | 실패 상태 | 필요 |

AI Draft는 자동 공개되지 않는다. 승인 전까지 공식 Timeline에 반영되지 않는다.

## 5. Feedback 상태

| 상태 | 의미 | 변경 주체 | 영향 화면 | 초기 필요 |
|---|---|---|---|---|
| 새 피드백 | Builder가 아직 확인하지 않음 | 시스템 | Builder 대시보드 | 필요 |
| 검토 중 | Builder가 검토 중 | Builder | 피드백 관리 | 필요 후보 |
| 반영됨 | Change Card 또는 수정으로 이어짐 | Builder | 관련 카드 연결 | 필요 |
| 반영하지 않음 | 반영하지 않기로 판단 | Builder | 판단 유지 후보 | 필요 후보 |

`반영하지 않음`은 실패가 아니라 판단이다. 필요한 경우 Change Card로 남길 수 있다.

## 6. Hypothesis 상태

| 상태 | 의미 | 변경 주체 | 영향 화면 | 초기 필요 |
|---|---|---|---|---|
| 가정 | 아직 검증 전 | Builder | 문제/가설 화면 | 필요 |
| 검증 중 | 근거를 모으는 중 | Builder | 대시보드, Timeline | 필요 |
| 일부 확인 | 일부 근거가 있음 | Builder | 가설 요약 | 필요 후보 |
| 확인됨 | 현재는 맞다고 보는 상태 | Builder | 공개 페이지 후보 | 필요 후보 |
| 반박됨 | 근거상 약해진 상태 | Builder | 핵심 전환 후보 | 필요 |
| 보류 | 지금 판단하지 않음 | Builder | 내부 관리 | 필요 후보 |

가설 상태 변경은 Change Card로 남기도록 유도하는 것이 적합하다.

## 7. Project 상태

| 상태 | 의미 | 변경 주체 | 영향 화면 | 초기 필요 |
|---|---|---|---|---|
| 아이디어 | 문제의식과 초기 가설 정리 중 | Builder | Project Card, 대시보드 | 필요 |
| 제작 중 | 산출물 제작 중 | Builder | Project Card | 필요 |
| 테스트 중 | 사용자 테스트/검증 중 | Builder | 탐색 필터 | 필요 |
| 베타 | 제한 공개 검증 중 | Builder | 배지 후보 | 필요 후보 |
| 공개됨 | 외부 공개 가능한 상태 | Builder | 공개 페이지 | 필요 후보 |
| 일시 중단 | 진행 중단 | Builder | 내부 관리 | 후보 |
| 종료됨 | 더 이상 진행하지 않음 | Builder | 인수인계/회고 후보 | 후보 |

주의: Project 상태와 Project 공개 상태는 구분한다.

## 8. 사용자에게 상태를 보여주는 방식

상태는 내부 관리용 딱딱한 값으로만 보여주면 안 된다. 사용자가 이해할 수 있는 문구가 필요하다.

예시:

- “이 카드는 아직 공개되지 않았습니다.”
- “AI 초안입니다. 승인 전까지 Timeline에 반영되지 않습니다.”
- “이 가설은 현재 검증 중입니다.”
- “이 피드백은 아직 반영 여부가 결정되지 않았습니다.”

## 9. 공개 전환 체크리스트

공개 전환 시 확인해야 할 항목:

- 원문 메모가 공개되지 않는지
- AI 초안이 그대로 공개되는 것은 아닌지
- 민감한 사용자 정보가 포함되어 있지 않은지
- 내부 인물, 팀원, 고객 정보가 노출되지 않는지
- 공개할 Change Card만 선택했는지
- 문제 정의와 가설 공개 여부를 확인했는지
- 관련 링크가 공개 가능한지

## 10. 민감 정보 공개 방지 원칙

- 기본값은 내부 비공개다.
- 공개는 Builder의 명시적 행동으로만 발생한다.
- AI 초안은 자동 공개하지 않는다.
- 민감 정보 포함 후보는 공개 전 경고한다.
- 공개 페이지는 공개 상태 객체만 읽는다.

## 11. 추가 검토 필요

- 민감 정보 포함 상태를 Builder가 직접 선택할지, 시스템이 추천할지.
- 링크 공개와 전체 공개의 차이를 어느 수준까지 둘지.
- 공개된 Change Card를 나중에 비공개로 돌릴 수 있는지.
- 피드백 내용을 공개할 때 작성자 동의를 어떻게 받을지.
