# Builder와 Scout 모델

## 1. 기본 원칙

BuildMap의 사용자 모델은 단순 작성자와 구경꾼으로 나뉘지 않는다. 한 사용자는 Builder이면서 Scout일 수 있다. 다만 제품 흐름에서는 두 역할의 목적이 다르므로 프로필과 데이터 책임을 구분한다.

- Builder: 프로젝트를 성장시키는 사람.
- Scout: 프로젝트를 발견하고 피드백, 테스트, 협업, 멘토링으로 연결되는 사람.

채용 담당자와 헤드헌터는 장기 확장 Scout에 포함될 수 있으나, 1차 데이터 모델에서는 핵심으로 두지 않는다.

## 2. Builder Profile

### 의미

Builder는 프로젝트를 만든 사람이 아니라 프로젝트를 성장시키는 사람이다. 창업자, 학생 메이커, 개발자, 디자이너, PM, 비개발자, 바이브코딩 유저, 사이드프로젝트 제작자가 모두 Builder가 될 수 있다.

### 역할 태그

역할 태그는 신분 인증이 아니라 프로젝트 맥락과 탐색을 돕는 메타데이터다.

예시 후보:

- 창업자
- 학생 메이커
- 개발자
- 디자이너
- PM
- 비개발자
- 바이브코딩 유저
- 사이드프로젝트 제작자

추가 검토 필요: 역할 태그를 고정 선택으로 둘지, 자유 입력을 허용할지 결정해야 한다.

### 관심 분야

Builder가 다루는 프로젝트 분야 또는 관심 영역이다.

예시:

- AI 도구
- 교육
- 생산성
- 커뮤니티
- 뷰티/커머스
- 헬스케어
- 개발자 도구

관심 분야는 탐색과 공개 프로필에 활용될 수 있다. 1차에서는 선택 정보로 둔다.

### 보유 프로젝트

Builder Profile은 하나 이상의 Project와 연결될 수 있다.

1차에서는 개인 Builder가 소유한 프로젝트 목록을 보여주는 수준으로 충분하다. 팀 기반 권한, 공동 소유, 프로젝트별 역할 권한은 후순위다.

### 공개 프로필 정보

공개 가능 정보 후보:

- 표시 이름
- 짧은 소개
- 역할 태그
- 공개 프로젝트 목록
- 관심 분야
- 외부 링크 후보

공개 프로필은 이력서가 아니다. Builder가 어떤 판단 흐름을 쌓아왔는지 프로젝트를 통해 드러나야 한다.

### 비공개 내부 정보

비공개 정보 후보:

- 이메일 또는 로그인 식별 정보
- 비공개 프로젝트 목록
- 내부 메모
- 승인 전 AI Draft
- 비공개 Change Card
- 민감 정보 포함 기록

### 초기 구현 필수 정보

- User와의 연결
- 표시 이름
- 역할 태그 후보
- 보유 Project 목록
- 공개 가능 여부 기본값

### 후순위 정보

- 상세 자기소개
- 팀/조직 소속
- 다중 프로젝트 권한
- 외부 프로필 연동
- 포트폴리오형 정렬

## 3. Scout Profile

### 의미

Scout는 단순 채용 담당자가 아니다. 초기 Scout는 프로젝트 발견자, 피드백 제공자, 테스터, 협업자, 멘토, 팀원 탐색자다.

Scout가 보는 것은 AI 점수나 역량 등급이 아니라 프로젝트 기록에서 드러나는 근거다. 예를 들면 사용자 테스트 반영 카드, 기능 제거 판단, 최근 업데이트, 피드백 요청 등이 있다.

### 관심 분야

Scout가 보고 싶은 프로젝트 분야다. Project Card Grid의 필터 또는 추천 후보로 활용될 수 있다.

1차에서는 간단한 관심 분야 선택만 두고, 복잡한 추천 알고리즘은 만들지 않는다.

### 탐색 목적

Scout의 초기 탐색 목적 후보:

- 피드백 제공
- 테스터 참여
- 협업 프로젝트 찾기
- 멘토링 대상 찾기
- 팀원 후보 탐색
- 관심 프로젝트 저장

채용, 헤드헌팅, 투자 탐색은 장기 확장으로만 기록한다.

### 저장한 프로젝트

Scout가 저장하거나 팔로우한 프로젝트 목록이다.

1차에서는 저장 또는 팔로우 중 하나로 단순화하는 것이 적절하다. 둘을 분리할지는 추가 검토 필요다.

### 피드백 작성 이력

Scout가 남긴 Feedback 목록이다.

피드백 이력은 Scout 개인 활동이지만, 공개 표시 여부는 별도 검토가 필요하다. 1차에서는 Builder가 피드백을 검토할 수 있는 수준이 핵심이다.

### 테스터 신청 이력

테스터 신청은 2차 확장 후보로 둔다. 1차에서는 Feedback의 `테스터 참여 의사` 정도로 표현해도 충분하다.

### 초기 구현 필수 정보

- User와의 연결 후보
- 관심 분야 후보
- 저장 프로젝트 후보
- 피드백 작성 가능 여부

Scout Profile 자체는 1차 필수라기보다 선택으로 시작할 수 있다. 로그인 기반 피드백만 요구하고, Scout 전용 프로필은 나중에 강화해도 된다.

### 후순위 정보

- 테스터 신청 이력
- 협업 제안 이력
- 멘토링 이력
- 저장 목록 고도화
- Scout 신뢰도 관리
- 알림 설정

## 4. 보류할 것

다음은 5단계에서 데이터 모델로 확정하지 않는다.

- 채용 계약 정보
- 헤드헌팅 계약
- 지원자 관리 정보
- 투자자 인증
- 기업 계정 인증
- AI 역량 점수
- Project DNA
- 자동 등급화

## 5. 데이터 모델상 주의점

Builder와 Scout 모델이 채용 플랫폼처럼 보이면 BuildMap의 중심이 흔들린다. 사람을 점수화하거나 이력서처럼 정리하는 대신, Project와 Change Card에서 드러나는 판단 근거를 중심으로 연결해야 한다.
