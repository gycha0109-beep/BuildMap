# 핵심 도메인 객체

이 문서는 BuildMap의 제품 관점 도메인 객체를 정리한다. 이 목록은 DB 테이블 목록이 아니다. 실제 테이블명, 컬럼명, 인덱스, 권한 정책은 아직 확정하지 않는다.

## 객체 요약 표

| 객체 | 정의 | 초기 포함 |
|---|---|---|
| User | BuildMap 계정의 기본 사용자 | 필수 |
| Builder Profile | 프로젝트를 성장시키는 사람의 프로필 | 필수 |
| Scout Profile | 프로젝트를 발견하고 피드백/테스트/협업하는 사람의 프로필 | 선택 |
| Project | 판단 흐름이 쌓이는 중심 단위 | 필수 |
| Problem Definition | 프로젝트가 해결하려는 문제의 현재/이전 정의 | 필수 |
| Hypothesis | 프로젝트가 믿고 검증하는 가설 | 필수 |
| Rough Note | Builder가 남긴 거친 원문 기록 | 필수 |
| AI Structured Draft | Rough Note를 구조화한 AI 변화 카드 초안 | 필수 |
| Change Card | 의사결정 단위이자 Timeline의 원천 기록 | 필수 |
| Decision Timeline | 승인된 Change Card의 흐름 표현 | 필수 |
| Decision Diff | 판단의 전후 차이를 보여주는 파생 표현 | 선택 |
| Feedback Request | 특정 판단에 대한 피드백 요청 | 필수 |
| Feedback | 로그인 사용자가 남기는 판단 연결 피드백 | 필수 |
| Public Project Page | Project의 공개 뷰 | 필수 |
| Project Link | 외부 관련 링크 | 선택 |
| Project Follow 또는 Save | 프로젝트 저장/팔로우 | 후보 |
| Visibility State | 공개/비공개 상태 표현 | 필수 |
| Activity Signal | 탐색용 내부 활동 신호 | 선택 |
| Notification Candidate | 알림 후보 이벤트 | 후순위 |

## User

- 정의: BuildMap에 로그인하는 기본 계정이다.
- 왜 필요한가: Builder와 Scout 역할을 하나의 계정에서 가질 수 있게 하기 위해 필요하다.
- 누가 생성하는가: 사용자가 가입할 때 생성한다.
- 언제 생성되는가: 첫 로그인 또는 계정 생성 시점.
- 어디에서 읽히는가: 대시보드, 피드백 작성, 저장/팔로우, 프로필 영역.
- 공개 가능 여부: 기본 계정 정보는 비공개가 원칙이며, 공개 이름 등 일부만 공개될 수 있다.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 소셜 로그인 세부 프로필, 조직 계정은 후순위.
- 추가 검토 필요: Builder와 Scout를 별도 계정으로 분리할지, 하나의 User가 역할을 겸하도록 할지 구현 단계에서 확인한다. 제품 방향상 하나의 User가 복수 역할을 가질 수 있는 편이 적합하다.

## Builder Profile

- 정의: 프로젝트를 만든 사람이 아니라 프로젝트를 성장시키는 사람의 공개/내부 프로필이다.
- 왜 필요한가: Project와 Change Card의 책임 주체를 표현하기 위해 필요하다.
- 누가 생성하는가: User가 Builder 역할을 선택할 때 생성한다.
- 언제 생성되는가: 가입 후 역할 선택 또는 첫 프로젝트 생성 시점.
- 어디에서 읽히는가: Builder 대시보드, Public Project Page, Project Card, Change Card 작성 화면.
- 공개 가능 여부: 공개 프로필 정보만 공개된다.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 팀 기반 Builder, 조직 소속, 상세 포트폴리오 정보는 후순위.
- 추가 검토 필요: 역할 태그를 어느 수준까지 자유 입력으로 둘지 검토가 필요하다.

## Scout Profile

- 정의: 프로젝트를 발견하고, 피드백을 주고, 테스터 또는 협업자로 참여하는 사용자의 탐색 프로필이다.
- 왜 필요한가: Scout 탐색 목적, 관심 분야, 저장 프로젝트, 피드백 이력을 표현하기 위해 필요하다.
- 누가 생성하는가: User가 Scout 역할을 선택하거나 탐색 기능을 사용할 때 생성한다.
- 언제 생성되는가: Scout 온보딩 또는 첫 저장/피드백 시점.
- 어디에서 읽히는가: Scout 탐색, 피드백 작성, 저장 목록.
- 공개 가능 여부: 기본은 비공개 또는 제한 공개.
- 초기 구현 포함 여부: 1차 선택.
- 후순위 여부: 채용 담당자/헤드헌터/투자자 인증은 장기 확장.
- 추가 검토 필요: Scout Profile 없이도 로그인 사용자가 피드백을 남길 수 있게 할지 검토한다.

## Project

- 정의: 문제 정의, 가설, 변화 카드, 피드백, 공개 페이지를 묶는 판단 흐름 컨테이너다.
- 왜 필요한가: BuildMap에서 모든 판단 기록이 속하는 중심 단위이기 때문이다.
- 누가 생성하는가: Builder.
- 언제 생성되는가: 첫 프로젝트 생성 시점.
- 어디에서 읽히는가: Builder 대시보드, Decision Timeline, Public Project Page, 탐색 카드.
- 공개 가능 여부: 비공개, 링크 공개, 전체 공개 후보.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 다중 소유자, 팀 프로젝트 권한은 후순위.
- 추가 검토 필요: Project 상태와 공개 상태를 분리해야 한다.

## Problem Definition

- 정의: 프로젝트가 해결하려는 문제의 현재 정의와 변경 이력이다.
- 왜 필요한가: 프로젝트 판단 흐름의 중심축이다.
- 누가 생성하는가: Builder.
- 언제 생성되는가: 프로젝트 생성 또는 문제 정의 입력 시점.
- 어디에서 읽히는가: 프로젝트 생성, 문제/가설 화면, Timeline 상단, Public Project Page.
- 공개 가능 여부: 공개 가능하나 민감한 문제 정의는 비공개 유지 가능해야 한다.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 다중 문제 정의 트리, 세부 세그먼트 관리는 후순위.
- 추가 검토 필요: 문제 정의 수정 시 Change Card 생성을 얼마나 강하게 유도할지 검토한다.

## Hypothesis

- 정의: 프로젝트가 맞다고 믿고 검증하는 판단 문장이다.
- 왜 필요한가: 기능 목록이 아니라 프로젝트가 무엇을 믿고 움직이는지 보여주기 위해 필요하다.
- 누가 생성하는가: Builder.
- 언제 생성되는가: 프로젝트 생성 또는 가설 입력 시점.
- 어디에서 읽히는가: 문제/가설 화면, Timeline, Public Project Page, Change Card 연결 영역.
- 공개 가능 여부: 공개 가능하나 내부 전략 가설은 비공개 가능해야 한다.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 복잡한 실험 관리, 가설 계층 구조는 후순위.
- 추가 검토 필요: 가설 상태 변경을 Change Card로 자동 유도할지 검토한다.

## Rough Note

- 정의: Builder가 회의 후, 테스트 후, 작업 중에 남기는 거친 원문 기록이다.
- 왜 필요한가: Builder가 처음부터 완성된 회고문을 쓰지 않도록 진입 부담을 낮춘다.
- 누가 생성하는가: Builder.
- 언제 생성되는가: 거친 메모 입력 화면에서 생성된다.
- 어디에서 읽히는가: AI 구조화 화면, Change Card 검토 화면.
- 공개 가능 여부: 기본 비공개.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 음성, 이미지, 파일, 외부 문서 입력은 후순위.
- 추가 검토 필요: AI 구조화 후에도 원문을 얼마 동안 보존할지 검토한다.

## AI Structured Draft

- 정의: Rough Note를 문제, 근거, 판단, 변경, 다음 확인 사항 등으로 구조화한 AI 초안이다.
- 왜 필요한가: Builder의 거친 입력을 Change Card 후보로 전환하기 위해 필요하다.
- 누가 생성하는가: AI가 생성하고 Builder가 검토한다.
- 언제 생성되는가: Builder가 “AI로 변화 카드 초안 만들기”를 실행할 때.
- 어디에서 읽히는가: Change Card Review 화면.
- 공개 가능 여부: 자동 공개 불가.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 고급 요약, 다중 입력 병합은 후순위.
- 추가 검토 필요: AI 실패/재시도 기록을 어느 정도 남길지 검토한다.

## Change Card

- 정의: 프로젝트에서 발생한 하나의 의사결정 단위다.
- 왜 필요한가: Decision Timeline의 원천 기록이기 때문이다.
- 누가 생성하는가: Builder가 직접 만들거나 AI Draft를 승인해 생성한다.
- 언제 생성되는가: Builder 승인 시점에 공식 기록이 된다.
- 어디에서 읽히는가: Builder 대시보드, Decision Timeline, Public Project Page, Decision Diff, 인수인계 흐름.
- 공개 가능 여부: 내부 전용, 공개 가능, 공개됨, 민감 정보 포함 등으로 제어한다.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 고급 연결 그래프, 다중 승인 워크플로우는 후순위.
- 추가 검토 필요: 필수 입력을 너무 늘리면 기록률이 떨어질 수 있다.

## Decision Timeline

- 정의: 승인된 Change Card들이 시간, 중요도, 관계에 따라 배열된 판단 흐름 표현이다.
- 왜 필요한가: 프로젝트가 왜 현재 모습이 되었는지 한눈에 보여주기 위해 필요하다.
- 누가 생성하는가: 별도 작성자가 생성하지 않는다. Change Card에서 파생된다.
- 언제 생성되는가: 승인된 Change Card가 쌓이면서 구성된다.
- 어디에서 읽히는가: 프로젝트 관리, 공개 페이지, Scout 탐색, 인수인계.
- 공개 가능 여부: 내부 Timeline과 공개 Timeline이 구분된다.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 고급 시각화, 그래프 모드는 후순위.
- 추가 검토 필요: Timeline 정렬과 요약 규칙은 구현 전 추가 검토가 필요하다.

## Decision Diff

- 정의: 초기 판단과 현재 판단, 전환 전과 후를 비교하는 판단 차이 표현이다.
- 왜 필요한가: 코드 diff가 아니라 문제 정의, 가설, 우선순위의 변화를 보여주기 위해 필요하다.
- 누가 생성하는가: Builder가 비교 기준을 선택하거나 시스템이 후보를 제안할 수 있다.
- 언제 생성되는가: 특정 비교가 필요할 때.
- 어디에서 읽히는가: Decision Timeline, Public Project Page, 인수인계 흐름.
- 공개 가능 여부: 선택 공개.
- 초기 구현 포함 여부: 1차 선택.
- 후순위 여부: 자동 diff 요약, 복수 시점 비교는 후순위.
- 추가 검토 필요: 어떤 변화가 Diff에 포함될 만큼 중요한지 기준이 필요하다.

## Feedback Request

- 정의: Builder가 특정 판단, 문제, 가설, 변화 카드에 대해 피드백을 요청하는 객체다.
- 왜 필요한가: 피드백을 막연한 댓글이 아니라 판단과 연결하기 위해 필요하다.
- 누가 생성하는가: Builder.
- 언제 생성되는가: 공개 페이지 또는 피드백 요청 화면에서 생성된다.
- 어디에서 읽히는가: Public Project Page, Scout 탐색, Feedback Write 화면.
- 공개 가능 여부: 공개 또는 제한 공개 가능.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 세부 모집 조건, 테스트 일정 관리는 후순위.
- 추가 검토 필요: 요청 만료 시점을 둘지 검토한다.

## Feedback

- 정의: 로그인 사용자 또는 Scout가 특정 질문이나 판단에 남기는 피드백이다.
- 왜 필요한가: Builder의 다음 판단과 Change Card로 이어질 수 있는 근거를 만들기 위해 필요하다.
- 누가 생성하는가: 로그인 사용자, Scout.
- 언제 생성되는가: 피드백 작성 시점.
- 어디에서 읽히는가: Builder 대시보드, 피드백 관리, 관련 Change Card.
- 공개 가능 여부: 공개/비공개 또는 Builder 검토 후 공개 후보.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 비로그인 피드백, 투표, 고급 스레드는 후순위.
- 추가 검토 필요: 댓글화되지 않도록 대상과 질문을 명확히 해야 한다.

## Public Project Page

- 정의: Project의 공개 가능한 일부 데이터를 보여주는 공개 뷰다.
- 왜 필요한가: 외부 사용자가 제품 소개보다 판단 흐름과 현재 필요한 도움을 이해하기 위해 필요하다.
- 누가 생성하는가: Builder가 공개 전환을 선택할 때 구성된다.
- 언제 생성되는가: 프로젝트 공개 상태가 링크 공개 또는 전체 공개가 될 때.
- 어디에서 읽히는가: 외부 사용자, Scout, 협업자, 테스터.
- 공개 가능 여부: 그 자체가 공개 뷰다.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 커스텀 도메인, 고급 레이아웃은 후순위.
- 추가 검토 필요: 공개 페이지 요약을 별도 저장할지 파생할지 검토한다.

## Project Link

- 정의: 프로젝트와 관련된 외부 링크다.
- 왜 필요한가: 데모, GitHub, Notion, 설문, 배포 링크 등 맥락 접근을 돕는다.
- 누가 생성하는가: Builder.
- 언제 생성되는가: 프로젝트 생성 또는 공개 페이지 편집 시점.
- 어디에서 읽히는가: Public Project Page, Builder 관리 화면.
- 공개 가능 여부: 링크별 공개 제어 필요.
- 초기 구현 포함 여부: 1차 선택.
- 후순위 여부: 링크 미리보기, 외부 연동은 후순위.
- 추가 검토 필요: GitHub/Notion 링크는 외부 연동과 구분해야 한다.

## Project Follow 또는 Save

- 정의: 사용자가 관심 프로젝트를 저장하거나 팔로우하는 행위다.
- 왜 필요한가: Scout가 진행 중인 프로젝트를 다시 확인할 수 있게 한다.
- 누가 생성하는가: 로그인 사용자, Scout.
- 언제 생성되는가: 저장/팔로우 버튼 클릭 시.
- 어디에서 읽히는가: Scout 저장 목록, 탐색 화면, 알림 후보.
- 공개 가능 여부: 기본 비공개.
- 초기 구현 포함 여부: 후보.
- 후순위 여부: 세부 알림 설정은 후순위.
- 추가 검토 필요: Follow와 Save를 1차에서 분리할지 하나로 둘지 검토한다.

## Visibility State

- 정의: Project, Change Card, Feedback Request 등 객체의 공개/비공개 상태다.
- 왜 필요한가: 내부 기록 후 공개 전환 원칙을 지키기 위해 필요하다.
- 누가 생성하는가: 시스템 기본값과 Builder의 명시 선택.
- 언제 생성되는가: 각 객체 생성 시 기본 상태가 부여된다.
- 어디에서 읽히는가: 모든 관리/공개 화면.
- 공개 가능 여부: 상태 자체는 내부 제어 정보다.
- 초기 구현 포함 여부: 포함.
- 후순위 여부: 세부 권한 정책은 후순위.
- 추가 검토 필요: 민감 정보 포함 상태의 UX 경고 기준이 필요하다.

## Activity Signal

- 정의: 탐색과 카드 그리드 정렬에 사용할 수 있는 내부 활동 신호다.
- 왜 필요한가: 지금 살아 움직이는 프로젝트를 발견하게 하기 위해 필요하다.
- 누가 생성하는가: 시스템이 Change Card, 공개 업데이트, 피드백 등에서 파생한다.
- 언제 생성되는가: 활동 발생 시점.
- 어디에서 읽히는가: Project Card Grid, 추후 히트맵.
- 공개 가능 여부: 집계 또는 배지 형태로 공개 가능.
- 초기 구현 포함 여부: 1차 선택.
- 후순위 여부: 고급 산식, 히트맵 점수는 후순위.
- 추가 검토 필요: 인기순위로 오해되지 않도록 표현을 조정해야 한다.

## Notification Candidate

- 정의: 알림으로 보낼 수 있는 후보 이벤트다.
- 왜 필요한가: 새 변화 카드, 피드백 요청, 테스터 모집 등 업데이트를 전달하기 위해 필요하다.
- 누가 생성하는가: 시스템.
- 언제 생성되는가: 관련 이벤트 발생 시점.
- 어디에서 읽히는가: 후순위 알림 화면.
- 공개 가능 여부: 비공개 내부 이벤트.
- 초기 구현 포함 여부: 제외 또는 후보.
- 후순위 여부: 후순위.
- 추가 검토 필요: 알림 피로도를 줄이기 위한 기준이 필요하다.
