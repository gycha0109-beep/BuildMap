# Feedback Request와 Feedback 모델

## 1. 기본 원칙

BuildMap의 피드백은 일반 댓글이 아니다. 피드백은 특정 판단, 질문, 문제 정의, 가설, 변화 카드에 연결되어야 한다.

피드백의 목적은 Builder를 평가하는 것이 아니라, Builder가 다음 판단을 내리는 데 필요한 근거를 제공하는 것이다.

1차에서는 비로그인 피드백을 보류한다. 최소 계정 또는 로그인 기반 피드백을 기본으로 가정한다.

## 2. Feedback Request

### 정의

Feedback Request는 Builder가 특정 판단에 대해 외부 의견을 요청하는 객체다.

예시:

- “첫 화면에서 현재 필요한 것이 명확하게 보이나요?”
- “이 문제 정의에 공감하시나요?”
- “이번 기능 제거 판단이 납득되나요?”
- “테스터로 참여할 수 있나요?”

### 연결 대상

Feedback Request는 다음 대상에 연결될 수 있다.

- Project
- Problem Definition
- Hypothesis
- Change Card
- Public Project Page

1차에서는 Project 또는 Change Card 연결을 우선하고, Problem/Hypothesis 직접 연결은 선택으로 둘 수 있다.

### 피드백 질문

Feedback Request에는 명확한 질문이 필요하다.

나쁜 예:

- “피드백 주세요.”

좋은 예:

- “현재 문제 정의가 실제로 겪는 문제처럼 느껴지나요?”
- “장기 목표보다 이번 주 증거를 먼저 보여주는 전환이 이해되나요?”

### 원하는 피드백 유형

후보:

- 공감 여부
- 이해도
- 기능 필요성
- 사용 경험
- 개선 제안
- 테스터 참여

유형은 응답자가 무엇을 해야 하는지 명확히 하는 장치다.

### 공개 상태

Feedback Request는 공개 프로젝트 페이지에 노출될 수 있다. 다만 내부 검토용 요청도 있을 수 있으므로 공개 상태를 가진다.

후보:

- 내부 요청
- 공개 요청
- 종료된 요청

### 요청 상태

상태 후보:

- 작성 중
- 공개 중
- 응답 수집 중
- 종료됨
- 보류됨

상태값은 제품 설계용 후보이며 DB enum으로 확정하지 않는다.

### 초기 구현 필수 정보

- 연결된 Project
- 연결 대상 후보
- 피드백 질문
- 원하는 피드백 유형
- 공개 상태
- 요청 상태
- 작성 Builder

### 후순위 정보

- 요청 만료일
- 응답 목표 수
- 대상 Scout 세그먼트
- 테스트 일정
- 보상 조건
- 고급 피드백 양식

## 3. Feedback

### 정의

Feedback은 Scout 또는 로그인 사용자가 Feedback Request에 대해 남기는 응답이다.

Feedback은 막연한 감상 댓글이 아니라 Builder의 다음 Change Card로 이어질 수 있는 근거다.

### 비로그인 피드백 보류

1차에서는 비로그인 피드백을 보류한다.

이유:

- 스팸 위험
- 악성 피드백 위험
- 신뢰도 문제
- Builder가 피드백을 판단 근거로 쓰기 어려움

후순위에서 제한적 비로그인 피드백을 검토할 수 있다.

### 연결 대상

Feedback은 기본적으로 Feedback Request에 연결된다. 간접적으로 Project, Change Card, Problem Definition, Hypothesis와 연결될 수 있다.

권장 방향:

```text
Feedback Request
→ Feedback
→ Builder 검토
→ 반영 또는 미반영 판단
→ 새 Change Card 후보
```

### 피드백 유형

응답자는 Builder가 요청한 유형에 맞춰 피드백을 남길 수 있다.

후보:

- 공감됨
- 이해 안 됨
- 기능 필요해 보임
- 기능 필요성이 약해 보임
- 사용 경험 공유
- 개선 제안
- 테스터 참여 의사 있음

### 피드백 내용

피드백 내용은 자유 텍스트로 시작하는 것이 적합하다. 다만 질문과 연결되어야 하며, 일반 댓글처럼 흘러가지 않도록 요청 문맥을 화면에 계속 보여줘야 한다.

### 테스터 참여 의사

1차에서는 별도 Tester Application 객체를 만들지 않고, Feedback 안에 테스터 참여 의사를 포함하는 방식이 단순하다.

2차 확장에서 테스터 신청 객체를 분리할 수 있다.

### Builder 검토 상태

상태 후보:

| 상태 | 의미 |
|---|---|
| 새 피드백 | Builder가 아직 확인하지 않음 |
| 검토 중 | Builder가 판단 중 |
| 반영됨 | 피드백이 Change Card 또는 수정으로 이어짐 |
| 반영하지 않음 | Builder가 반영하지 않기로 판단함 |

`반영하지 않음`도 중요한 판단이 될 수 있다. 이 경우 `판단 유지` 또는 `기능 보류` Change Card로 이어질 수 있다.

### 공개 가능 여부

Feedback은 작성자 정보와 내용이 포함되므로 공개 정책이 필요하다.

1차 권장 방향:

- Builder는 피드백을 내부에서 검토한다.
- 공개 페이지에 피드백 내용을 직접 노출하는 것은 후순위로 둔다.
- 피드백이 반영된 Change Card는 공개 가능할 수 있다.

### 초기 구현 필수 정보

- 연결된 Feedback Request
- 작성 User 또는 Scout 후보
- 피드백 유형
- 피드백 내용
- 테스터 참여 의사 후보
- Builder 검토 상태
- 작성 시점

### 후순위 정보

- 공개 표시 이름 세부 제어
- 피드백 공개 동의
- 댓글 스레드
- 피드백 품질 신고
- 비로그인 피드백
- 테스터 신청 객체 분리

## 4. 피드백이 Change Card로 이어지는 방식

피드백은 그 자체로 Timeline 원천이 아니다. Builder가 피드백을 근거로 판단을 내렸을 때 Change Card가 된다.

예시:

```text
Feedback: “첫 화면에서 장기 목표 입력이 부담스럽다.”
→ Builder 검토
→ Change Card: “첫 화면에서 장기 목표 입력 제거”
→ 유형: 사용자 피드백 / 판단 수정
→ 근거: 테스트 참여자 3명의 반응
```

## 5. 데이터 모델상 주의점

피드백이 일반 댓글처럼 흘러가면 BuildMap은 커뮤니티 게시판처럼 보일 수 있다. 이를 막기 위해 다음 원칙을 둔다.

- 피드백 요청은 특정 질문에 연결한다.
- 피드백은 Feedback Request에 연결한다.
- Builder가 반영 여부를 검토한다.
- 반영된 피드백은 Change Card의 근거가 될 수 있다.
- 평가 점수나 별점 중심으로 만들지 않는다.
