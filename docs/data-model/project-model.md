# Project 모델

## 1. Project의 역할

Project는 BuildMap에서 판단 흐름이 쌓이는 중심 단위다. Project는 완성작 등록이 아니며, 제품 소개글도 아니다. Project는 문제 정의, 가설, Rough Note, AI Draft, Change Card, Feedback Request, Public Project Page를 묶는 운영 단위다.

Project는 다음 질문에 답하기 위한 컨테이너다.

- 이 프로젝트는 어떤 문제에서 시작했는가?
- 지금 어떤 가설을 검증하고 있는가?
- 어떤 판단과 전환을 거쳤는가?
- 지금 어떤 도움이 필요한가?
- 외부에는 어떤 판단 흐름을 공개할 것인가?

## 2. Project가 포함할 제품상 정보

### 프로젝트명

- 목적: Project를 구분하는 가장 기본적인 이름.
- 초기 구현: 필수.
- 공개 여부: 공개 프로젝트 페이지와 카드 그리드에 노출 가능.
- 주의: 브랜드명 확정이 아니어도 입력 가능해야 한다.

### 한 줄 정의

- 목적: 프로젝트의 현재 방향을 짧게 설명한다.
- 초기 구현: 필수.
- 공개 여부: 공개 가능.
- 주의: 완성된 홍보 문구가 아니라 현재 판단 기준의 설명이어야 한다.

### 현재 상태

- 목적: 프로젝트가 어느 단계에 있는지 보여준다.
- 초기 구현: 필수 후보.
- 공개 여부: 공개 가능.
- 주의: 진행 상태와 공개 상태를 혼동하지 않는다.

### 소유 Builder

- 목적: Project를 책임지는 Builder를 나타낸다.
- 초기 구현: 필수.
- 공개 여부: 공개 프로필 범위에서 노출 가능.
- 주의: 공동 Builder는 후순위로 둔다.

### 참여 Builder 후보

- 목적: 협업 프로젝트에서 다른 Builder를 표현하기 위한 후보 정보.
- 초기 구현: 제외 또는 후순위.
- 공개 여부: 참여자 동의 후 공개.
- 추가 검토 필요: 팀 권한과 소유권 설계가 필요하다.

### 해결하려는 문제 요약

- 목적: Project의 중심 문제를 요약한다.
- 초기 구현: 필수.
- 공개 여부: 공개 가능하나 민감한 문제는 비공개 가능.
- 주의: Problem Definition과 중복 저장될 위험이 있다. 원천은 Problem Definition으로 두고 Project에는 요약 표시만 둔다.

### 현재 가설 요약

- 목적: 현재 Project가 믿고 검증하는 핵심 가설을 보여준다.
- 초기 구현: 필수 또는 강력 권장.
- 공개 여부: 선택 공개.
- 주의: Hypothesis와 중복 저장 위험이 있다. 원천은 Hypothesis로 둔다.

### 현재 필요한 것

- 목적: 피드백, 테스터, 협업자, 멘토링 등 외부 도움을 명확히 한다.
- 초기 구현: 필수.
- 공개 여부: 공개 페이지와 카드 그리드에 노출 가능.
- 주의: 막연한 “도움 필요”가 아니라 구체 요청이어야 한다.

### 공개 상태

- 목적: Project가 내부 전용인지, 링크 공개인지, 전체 공개인지 제어한다.
- 초기 구현: 필수.
- 공개 여부: 제어 정보.
- 주의: Project 공개 상태와 Change Card 공개 상태를 분리해야 한다.

### 최근 변화 요약

- 목적: 프로젝트가 최근 어떤 판단 전환을 거쳤는지 보여준다.
- 초기 구현: 파생 정보로 후보.
- 공개 여부: 공개 가능한 Change Card에서 파생된 경우 공개 가능.
- 주의: 별도 원천으로 중복 저장하지 않는다.

### 관련 링크

- 목적: 데모, 배포, 설문, 문서, 저장소 등 맥락 접근을 돕는다.
- 초기 구현: 선택.
- 공개 여부: 링크별 공개 제어 필요.
- 후순위: 링크 미리보기, 외부 연동.

### 태그

- 목적: 탐색 필터와 분류를 돕는다.
- 초기 구현: 선택.
- 공개 여부: 공개 가능.
- 추가 검토 필요: 자유 태그와 고정 카테고리의 혼합 여부.

### 생성 시점 / 최근 업데이트 시점

- 목적: 프로젝트 활동성을 표시한다.
- 초기 구현: 필수 후보.
- 공개 여부: 공개 가능.
- 주의: 최근 업데이트는 단순 수정 시간이 아니라 공개 가능한 판단 변화 기준으로 보여주는 것이 적합하다.

## 3. Project 상태 후보

상태값은 제품 설계용 후보이며 DB enum으로 확정하지 않는다.

| 상태 | 의미 |
|---|---|
| 아이디어 | 문제의식과 초기 가설을 정리하는 단계 |
| 제작 중 | 실제 산출물을 만들고 있는 단계 |
| 테스트 중 | 사용자 반응, 피드백, 사용성 검증을 진행하는 단계 |
| 베타 | 제한된 사용자에게 공개해 검증하는 단계 |
| 공개됨 | 외부에 공개 가능한 상태 |
| 일시 중단 | 현재 활동이 멈췄지만 폐기하지 않은 상태 |
| 종료됨 | 더 이상 진행하지 않는 상태. 종료 이유도 Change Card가 될 수 있음 |

주의: `공개됨` 상태와 Project 공개 상태의 `전체 공개`는 다르다. 하나는 프로젝트 진행 상태이고, 하나는 외부 노출 정책이다.

## 8. 중복 저장 위험

Project에 문제 정의 요약, 가설 요약, 최근 변화 요약을 둘 때 원천 객체와 중복될 수 있다. 권장 방향은 다음이다.

- Problem Definition이 원천이다.
- Hypothesis가 원천이다.
- Change Card가 최근 변화의 원천이다.
- Project는 필요한 요약을 보여주는 컨테이너 역할을 한다.

성능상 요약 캐시가 필요할 수 있으나, 이 단계에서는 캐시 구조를 확정하지 않는다.
