# 객체 간 관계

이 문서는 BuildMap의 제품 데이터 객체 간 관계를 정리한다. 실제 외래키, 테이블 구조, 인덱스, API는 아직 확정하지 않는다.

## 관계 요약

| 관계 | 필수/선택 | 권장 방향 |
|---|---|---|
| User ↔ Builder/Scout Profile | 선택적 역할 확장 | 하나의 User가 복수 역할 가능 |
| Builder → Project | 필수 | Project의 책임 주체를 명확히 한다 |
| Project → Problem Definition | 권장 필수 | 단, 초기 생성 시 짧게 시작 가능 |
| Project → Hypothesis | 선택 시작, 강력 권장 | 복잡한 가설 관리로 만들지 않는다 |
| Project → Rough Note | 필수 흐름 | 내부 기록으로 시작한다 |
| Rough Note → AI Structured Draft | 선택 생성 | AI 실패 시 원문 보존 |
| AI Structured Draft → Change Card | Builder 승인 시 전환 | 승인 전 공식 기록 아님 |
| Change Card → Decision Timeline | 필수 | Timeline은 파생 표현 |
| Change Card ↔ Problem/Hypothesis | 선택 연결, 승인 시 유도 | 단독 생성 가능 |
| Change Card ↔ Feedback | 선택 연결 | 피드백이 판단 근거가 될 수 있음 |
| Feedback Request → Feedback | 필수 | 피드백은 질문에 연결 |
| Project → Public Project Page | 공개 시 파생 | 별도 제품 소개글 아님 |
| Project → Project Link | 선택 | 관련 링크 공개 제어 필요 |
| Scout → Project Save | 선택 | 재방문과 탐색 보조 |
| Activity Signal → Project Discovery | 파생 | 인기순위화 주의 |

## User와 Builder Profile / Scout Profile

- 관계 설명: User는 BuildMap 계정이고, Builder Profile과 Scout Profile은 역할별 제품 프로필이다.
- 필수 연결인지 선택 연결인지: User는 필수, 역할 프로필은 사용 흐름에 따라 선택 생성될 수 있다.
- 1차 구현에서 필요한지: Builder Profile은 필요하다. Scout Profile은 1차 선택으로 둘 수 있다.
- 후순위인지: 조직 계정, 채용 담당자 인증, 투자자 인증은 후순위다.
- 연결이 너무 강하면 생길 문제: 가입 즉시 역할 선택 부담이 커질 수 있다.
- 연결이 너무 느슨하면 생길 문제: 누가 Project를 책임지는지 불명확해진다.
- 권장 방향: 하나의 User가 Builder와 Scout 역할을 모두 가질 수 있게 둔다.

## Builder와 Project

- 관계 설명: Builder는 Project를 생성하고 성장시키는 책임 주체다.
- 필수 연결인지 선택 연결인지: 필수.
- 1차 구현에서 필요한지: 필요.
- 후순위인지: 다중 Builder, 팀 소유권, 권한 분리는 후순위.
- 연결이 너무 강하면 생길 문제: 협업자 추가가 어려워질 수 있다.
- 연결이 너무 느슨하면 생길 문제: 프로젝트 책임과 공개 제어가 불명확해진다.
- 권장 방향: 1차에서는 소유 Builder 1명을 중심으로 두고, 참여 Builder 후보는 후순위로 둔다.

## Project와 Problem Definition

- 관계 설명: Project는 현재 문제 정의를 가진다. 문제 정의는 수정될 수 있으며, 수정 이유는 Change Card로 남길 수 있다.
- 필수 연결인지 선택 연결인지: 제품상 강력 권장. 처음 생성 시 짧은 문제 입력만 요구한다.
- 1차 구현에서 필요한지: 필요.
- 후순위인지: 과거 문제 정의 이력의 고급 비교는 후순위.
- 연결이 너무 강하면 생길 문제: Builder가 프로젝트 생성을 시작하기 어렵다.
- 연결이 너무 느슨하면 생길 문제: 프로젝트가 단순 소개글이나 아이디어 목록처럼 보일 수 있다.
- 권장 방향: Project 생성 시 문제 정의를 간단히 받되, 이후 수정 시 Change Card 생성을 유도한다.

## Project와 Hypothesis

- 관계 설명: Hypothesis는 Project가 성립하려면 맞아야 하는 판단이다.
- 필수 연결인지 선택 연결인지: 선택으로 시작하되, 핵심 흐름에서는 권장한다.
- 1차 구현에서 필요한지: 필요.
- 후순위인지: 다중 가설 트리, 실험 설계 관리 기능은 후순위.
- 연결이 너무 강하면 생길 문제: 비개발자와 초보 Builder에게 입력 부담이 커진다.
- 연결이 너무 느슨하면 생길 문제: Change Card의 판단 근거가 약해진다.
- 권장 방향: “아직 확정되지 않은 가설”을 짧게 받으며 상태 관리는 단순하게 둔다.

## Project와 Rough Note

- 관계 설명: Rough Note는 특정 Project 안에서 생성되는 내부 원문 기록이다.
- 필수 연결인지 선택 연결인지: Rough Note는 Project에 필수 연결된다.
- 1차 구현에서 필요한지: 필요.
- 후순위인지: 프로젝트 밖 개인 메모함은 후순위.
- 연결이 너무 강하면 생길 문제: 여러 프로젝트에 걸친 메모를 다루기 어렵다.
- 연결이 너무 느슨하면 생길 문제: 어떤 프로젝트 판단인지 알 수 없다.
- 권장 방향: 1차에서는 Rough Note를 반드시 하나의 Project에 속하게 한다.

## Rough Note와 AI Structured Draft

- 관계 설명: AI Structured Draft는 Rough Note를 구조화한 Change Card 초안이다.
- 필수 연결인지 선택 연결인지: 선택. Builder가 AI 구조화를 실행할 때만 생성된다.
- 1차 구현에서 필요한지: 필요.
- 후순위인지: 다중 Rough Note 병합, 음성/이미지 입력 구조화는 후순위.
- 연결이 너무 강하면 생길 문제: AI 없이 직접 Change Card를 만들기 어렵다.
- 연결이 너무 느슨하면 생길 문제: AI 초안의 근거 원문을 확인하기 어렵다.
- 권장 방향: AI Draft는 Rough Note를 참조하되, Builder가 직접 Change Card를 작성하는 경로도 열어둔다.

## AI Structured Draft와 Change Card

- 관계 설명: AI Draft는 승인될 때 Change Card가 된다.
- 필수 연결인지 선택 연결인지: AI 경로에서는 필수. 직접 작성 경로에서는 AI Draft 없이 Change Card 생성 가능.
- 1차 구현에서 필요한지: 필요.
- 후순위인지: 다단계 승인, 팀 승인 워크플로우는 후순위.
- 연결이 너무 강하면 생길 문제: AI Draft가 곧 공식 기록처럼 오해될 수 있다.
- 연결이 너무 느슨하면 생길 문제: 어떤 초안이 승인되어 공식 기록이 되었는지 추적하기 어렵다.
- 권장 방향: AI Draft는 승인 전 공식 기록이 아니며, Builder 승인 시 Change Card로 전환한다.

## Change Card와 Decision Timeline

- 관계 설명: Decision Timeline은 승인된 Change Card를 시간·중요도·관계 기준으로 보여준다.
- 필수 연결인지 선택 연결인지: 필수.
- 1차 구현에서 필요한지: 필요.
- 후순위인지: 고급 시각화, 그래프 표현은 후순위.
- 연결이 너무 강하면 생길 문제: 모든 카드가 동일한 노출 강도를 가져 Timeline이 무거워진다.
- 연결이 너무 느슨하면 생길 문제: Timeline이 원천 기록과 분리된 별도 게시판이 된다.
- 권장 방향: 원천 기록은 Change Card, Timeline은 표현으로 고정한다.

## Change Card와 Problem Definition

- 관계 설명: Change Card는 문제 정의와 연결될 수 있다. 특히 문제 발견, 문제 정의 수정 유형에서 중요하다.
- 필수 연결인지 선택 연결인지: 선택 연결. 승인 시 연결을 유도한다.
- 1차 구현에서 필요한지: 필요하되 강제하지 않는다.
- 후순위인지: 세부 문제 정의 버전 관리와 Diff는 후순위.
- 연결이 너무 강하면 생길 문제: 모든 카드를 문제에 연결해야 해 기록 부담이 커진다.
- 연결이 너무 느슨하면 생길 문제: 문제 정의가 왜 바뀌었는지 추적하기 어렵다.
- 권장 방향: 단독 생성 가능, 승인 시 “연결할 문제 정의가 있나요?”를 유도한다.

## Change Card와 Hypothesis

- 관계 설명: Change Card는 가설 생성, 검증, 반박, 보류 등과 연결될 수 있다.
- 필수 연결인지 선택 연결인지: 선택 연결. 가설 관련 카드에서는 강력 권장.
- 1차 구현에서 필요한지: 필요.
- 후순위인지: 실험 설계 도구 수준의 엄격한 연결은 후순위.
- 연결이 너무 강하면 생길 문제: 가설이 없는 초기 프로젝트에서 카드 생성이 막힌다.
- 연결이 너무 느슨하면 생길 문제: 가설이 어떻게 바뀌었는지 보이지 않는다.
- 권장 방향: 승인 시 연결 후보를 제안하되 필수로 막지 않는다.

## Change Card와 Feedback

- 관계 설명: Feedback은 Change Card의 근거가 되거나, Change Card 이후의 반응이 될 수 있다.
- 필수 연결인지 선택 연결인지: 선택.
- 1차 구현에서 필요한지: 필요.
- 후순위인지: 피드백 스레드, 투표, 신뢰도 관리는 후순위.
- 연결이 너무 강하면 생길 문제: 모든 판단이 외부 피드백을 필요로 하는 것처럼 보인다.
- 연결이 너무 느슨하면 생길 문제: 피드백이 변화로 이어졌는지 알기 어렵다.
- 권장 방향: Builder가 피드백을 반영할 때 관련 Change Card로 연결할 수 있게 한다.

## Feedback Request와 Feedback

- 관계 설명: Feedback은 특정 Feedback Request에 대한 응답이다.
- 필수 연결인지 선택 연결인지: 1차에서는 필수 연결을 권장한다.
- 1차 구현에서 필요한지: 필요.
- 후순위인지: 일반 댓글, 자유 토론은 후순위 또는 제외에 가깝다.
- 연결이 너무 강하면 생길 문제: 사용자가 예상 밖의 의견을 남기기 어렵다.
- 연결이 너무 느슨하면 생길 문제: 피드백이 일반 댓글처럼 흘러간다.
- 권장 방향: 피드백은 특정 질문 또는 판단에 연결한다.

## Project와 Public Project Page

- 관계 설명: Public Project Page는 Project의 공개 가능한 데이터 뷰다.
- 필수 연결인지 선택 연결인지: Project가 공개될 때 필수.
- 1차 구현에서 필요한지: 필요.
- 후순위인지: 고급 커스텀 페이지는 후순위.
- 연결이 너무 강하면 생길 문제: 내부 기록까지 공개 페이지에 노출될 위험이 있다.
- 연결이 너무 느슨하면 생길 문제: 공개 페이지가 별도 소개글로 분리되어 판단 흐름과 멀어진다.
- 권장 방향: 공개 페이지는 Project와 공개 Change Card를 기반으로 파생한다.

## Project와 Project Link

- 관계 설명: Project는 데모, 저장소, 설문, 배포 링크 등 관련 링크를 가질 수 있다.
- 필수 연결인지 선택 연결인지: 선택.
- 1차 구현에서 필요한지: 선택.
- 후순위인지: 외부 링크 미리보기, 자동 동기화는 후순위.
- 연결이 너무 강하면 생길 문제: 링크가 없는 초기 프로젝트가 시작하기 어렵다.
- 연결이 너무 느슨하면 생길 문제: 외부 맥락 접근성이 떨어진다.
- 권장 방향: 관련 링크는 선택 정보로 둔다.

## Scout와 Project Save

- 관계 설명: Scout 또는 로그인 사용자는 관심 프로젝트를 저장할 수 있다.
- 필수 연결인지 선택 연결인지: 선택.
- 1차 구현에서 필요한지: 후보.
- 후순위인지: 저장 목록 고도화와 알림은 후순위.
- 연결이 너무 강하면 생길 문제: 저장 기능이 초기 핵심보다 커질 수 있다.
- 연결이 너무 느슨하면 생길 문제: 재방문 흐름이 약해진다.
- 권장 방향: 1차에서는 Save 또는 Follow를 단순하게 둔다.

## Activity Signal과 Project Discovery

- 관계 설명: Activity Signal은 프로젝트 카드 그리드와 향후 히트맵 탐색에 쓰이는 내부 활동 신호다.
- 필수 연결인지 선택 연결인지: 선택.
- 1차 구현에서 필요한지: 간단한 수준으로만 후보.
- 후순위인지: 고급 히트맵 산식은 후순위.
- 연결이 너무 강하면 생길 문제: 탐색이 지표 중심으로 변질된다.
- 연결이 너무 느슨하면 생길 문제: 살아 움직이는 프로젝트를 발견하기 어렵다.
- 권장 방향: 최근 업데이트, 피드백 요청, 테스터 모집 같은 명확한 신호부터 사용한다.
