# Decision Timeline 모델

## 1. 정의

Decision Timeline은 승인된 Change Card들이 시간, 중요도, 연결 관계에 따라 배열된 판단 흐름 표현이다.

Decision Timeline은 별도의 수동 작성 게시판이 아니다. 원천 기록은 Change Card이고, Timeline은 그 원천 기록을 Builder, Scout, 외부 사용자, 새 팀원/협업자가 이해할 수 있게 보여주는 표현 구조다.

## 2. Change Card 기반 표현

Decision Timeline에 표시되는 핵심 단위는 Change Card다.

```text
승인된 Change Card
→ 시간순 또는 중요도 기반 배열
→ 핵심 전환 강조
→ 일반 카드 접힘 처리
→ 문제/가설/피드백 연결 표시
→ 내부 Timeline 또는 공개 Timeline으로 표현
```

Timeline 자체가 별도 내용을 새로 저장하기 시작하면 Change Card와 중복 원천 기록이 될 위험이 있다.

## 3. Timeline이 직접 생성하는 데이터와 파생 데이터

### 직접 생성하지 않는 것

- 판단 내용
- 근거
- 변경 내용
- 다음 확인 사항
- 문제 정의 수정 내역
- 가설 반박 내용

위 정보의 원천은 Change Card다.

### Timeline 표현을 위해 필요할 수 있는 파생 정보

- 카드 정렬 기준
- 핵심 전환 강조 여부
- 접힘/펼침 상태 후보
- 필터 조건
- 공개 Timeline에서 보이는 카드 목록
- 상단 요약에 노출할 최근 전환 후보

추가 검토 필요: 접힘/펼침 같은 UI 상태를 사용자별로 저장할지, 화면 상태로만 둘지 결정해야 한다.

## 4. Timeline에 표시되는 정보

Timeline 상단 요약 후보:

- 현재 문제 정의
- 현재 가설
- 최근 전환
- 현재 필요한 것

Timeline 본문 카드 후보:

- Change Card 제목
- 유형
- 구조화 요약
- 근거 요약
- 판단
- 변경 내용 후보
- 다음 확인 사항
- 연결된 문제/가설 표시
- 공개 상태 표시
- 중요도 표시
- 작성/승인 시점

## 5. 핵심 전환점 표시 방식

핵심 전환점은 중요도가 `핵심 전환`인 Change Card를 중심으로 표현한다.

예시:

- 문제 정의가 바뀐 카드
- 가설이 반박된 카드
- 기능을 제거한 카드
- 타깃 사용자가 바뀐 카드
- 프로젝트 방향이 전환된 카드
- 중요한 릴리즈 카드

핵심 전환은 Public Project Page 상단의 최근 전환 영역과 프로젝트 진화도에서 강조될 수 있다.

## 6. 일반 카드 접힘 처리

모든 카드를 동일한 크기로 보여주면 Timeline이 활동 로그처럼 보일 수 있다.

권장 방향:

- 핵심 전환 카드는 펼쳐서 보여준다.
- 일반 카드는 제목, 유형, 요약 중심으로 접어 보여준다.
- 사용자가 필요할 때 펼쳐 근거와 판단을 확인한다.

## 7. 필터 기준 후보

필터는 제품 설계용 후보이며 구현 구조는 확정하지 않는다.

- 유형
- 공개 여부
- 핵심 전환
- 연결된 가설
- 기간
- 연결된 문제 정의 후보
- 피드백 반영 여부 후보

1차에서는 유형, 공개 여부, 핵심 전환, 기간 정도면 충분하다.

## 8. Builder 내부 Timeline

Builder 내부 Timeline은 승인된 내부 Change Card를 모두 볼 수 있는 흐름이다. 공개되지 않은 카드도 포함될 수 있다.

목적:

- 프로젝트 판단 흐름 관리
- 미공개 카드 검토
- 공개 전환 후보 확인
- 문제/가설 연결 확인
- 인수인계 준비

## 9. 공개 Timeline

공개 Timeline은 공개된 Change Card만 보여준다.

목적:

- 외부 사용자가 프로젝트가 왜 지금 모습이 되었는지 이해
- Scout가 피드백, 테스트, 협업 여부 판단
- 공개 프로젝트 페이지의 하단 판단 흐름 제공

주의: 공개 Timeline은 내부 Timeline의 일부 뷰다. 별도 내용 원천을 만들지 않는다.

## 10. Scout가 보는 Timeline

Scout는 Timeline에서 점수가 아니라 근거를 본다.

예시 근거:

- 사용자 피드백 반영 기록
- 기능 제거 판단
- 가설 반박 후 방향 전환
- 최근 업데이트
- 현재 필요한 도움

채용/헤드헌팅 관점의 점수화는 장기 확장으로 보류한다.

## 11. 인수인계 목적으로 보는 Timeline

새 팀원이나 협업자는 Timeline을 통해 다음을 빠르게 이해한다.

- 처음 문제 정의
- 주요 가설
- 핵심 전환
- 기능 제거 이유
- 기술/방향 선택 이유
- 현재 필요한 것

인수인계 전용 모드는 후순위지만, 1차 Timeline 구조가 인수인계에 도움이 되도록 설계한다.

## 12. 초기 구현 필수 정보

- Project와 연결된 승인 Change Card 목록
- 카드 승인 시점
- 카드 유형
- 카드 중요도
- 카드 공개 상태
- 카드 요약
- 현재 문제 정의/가설/필요한 것 상단 표시 후보

## 13. 후순위 정보

- 그래프형 프로젝트 진화도
- 고급 필터
- 자동 Timeline 요약
- 인수인계 모드
- 특정 사용자별 읽음 상태
- 고급 검색

## 14. 데이터 모델상 주의점

Decision Timeline이 별도 게시글처럼 설계되면 Change Card와 중복된다. 반드시 다음 원칙을 유지한다.

> 원천 기록은 Change Card이고, Decision Timeline은 그 기록의 표현이다.
