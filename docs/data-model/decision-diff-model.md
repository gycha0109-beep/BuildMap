# Decision Diff 모델

## 1. 정의

Decision Diff는 코드 diff가 아니라 판단 diff다. 특정 시점 사이에 프로젝트의 문제 정의, 가설, 타깃 사용자, 핵심 기능, 우선순위, 성공 기준이 어떻게 달라졌는지 보여주는 표현이다.

Decision Diff는 별도 수동 작성 문서가 아니라 Change Card와 Decision Timeline을 기반으로 파생되어야 한다.

## 2. 비교 대상

비교 대상 후보:

- 초기 판단 vs 현재 판단
- 전환 전 vs 전환 후
- 특정 시점 A vs 특정 시점 B
- 특정 핵심 Change Card 전후

1차에서는 Builder가 비교 기준을 직접 선택하는 단순 좌우 비교가 적합하다.

## 3. 비교 항목

### 문제 정의

- 기존 문제 정의와 현재 문제 정의의 차이.
- 원천: Problem Definition과 관련 Change Card.

### 핵심 가설

- 기존에 믿었던 가설과 현재 검증 중인 가설의 차이.
- 원천: Hypothesis와 관련 Change Card.

### 타깃 사용자

- 프로젝트가 누구를 중심으로 바뀌었는지.
- 원천: Project 설명, Change Card, 공개 요약 후보.
- 추가 검토 필요: 타깃 사용자 객체를 별도로 둘지 여부.

### 핵심 기능

- 유지, 추가, 제거, 보류된 기능의 차이.
- 원천: 기능 추가/제거/보류 Change Card.

### 우선순위

- 무엇을 먼저 검증하기로 했는지.
- 원천: 판단 수정, 방향 전환, 다음 확인 사항.

### 성공 기준

- 어떤 상태를 성공으로 보게 되었는지.
- 원천: Hypothesis, Change Card, Feedback Request.
- 추가 검토 필요: 1차에서 성공 기준을 별도 객체로 둘지는 보류한다.

## 4. 표시 방식

1차 표현은 좌우 비교가 적합하다.

```text
초기 판단        현재 판단
문제 정의        문제 정의
핵심 가설        핵심 가설
타깃 사용자      타깃 사용자
핵심 기능        핵심 기능
우선순위         우선순위
성공 기준        성공 기준
```

각 행에는 변경 이유와 연결된 Change Card를 함께 보여준다.

## 5. 연결된 Change Card

Decision Diff는 반드시 관련 Change Card로 돌아갈 수 있어야 한다.

예시:

- “10년 목표 우선” → “이번 주 증거 우선”
- 연결 Change Card: 사용자 피드백으로 첫 화면 전환
- 근거: 테스트 참여자 3명이 장기 목표 입력을 부담스럽게 느낌

## 6. 공개 페이지 포함 여부

Decision Diff는 공개 프로젝트 페이지에 포함될 수 있다. 다만 공개 가능한 Change Card와 공개 가능한 문제/가설만 사용해야 한다.

공개 포함 후보:

- 큰 방향 전환 요약
- 초기 판단 vs 현재 판단
- 왜 기능을 제거했는지

내부 전용 후보:

- 민감한 전략 판단
- 내부 팀 논의
- 경쟁사 관련 내용

## 7. 초기 구현

초기 구현에서는 다음으로 제한한다.

- 단순 좌우 비교
- Builder가 비교 기준 선택
- 문제 정의, 핵심 가설, 핵심 기능 중심
- 연결 Change Card 링크
- 공개 페이지 포함 여부 선택

## 8. 후순위 확장

- 자동 Diff 생성
- 여러 시점 비교
- 시각적 변화 그래프
- 인수인계 요약과 연결
- Scout용 공개 Diff 카드
- AI 요약

## 9. 데이터 모델상 주의점

Decision Diff가 별도 수동 문서가 되면 원천 기록이 중복된다. Diff는 Change Card와 Timeline을 기반으로 한 파생 표현이어야 한다.

추가 검토 필요: 타깃 사용자, 성공 기준, 우선순위를 별도 객체로 관리할지, Change Card와 Project 요약에서 파생할지 DB 설계 전 확인해야 한다.
