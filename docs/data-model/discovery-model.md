# Discovery 모델

## 1. 기본 원칙

BuildMap의 1차 탐색 구조는 Project Card Grid다. 히트맵은 후순위 실험 화면이다.

탐색의 목적은 인기 프로젝트 순위를 보여주는 것이 아니다. 목적은 지금 살아 움직이고, 피드백이나 테스터, 협업이 필요한 프로젝트를 발견하게 하는 것이다.

## 2. Project Card Grid

Project Card Grid는 1차 탐색 화면의 기본 구조다.

히트맵보다 카드 그리드를 먼저 두는 이유:

- 지표 산식이 아직 확정되지 않았다.
- 비개발자가 프로젝트 맥락을 이해하기 쉽다.
- 현재 필요한 것, 최근 전환, 피드백 요청을 직접 보여줄 수 있다.
- 초기 구현 부담이 낮다.

## 3. 탐색 카드에 필요한 정보

### 프로젝트명

- 원천: Project
- 초기 구현: 필수

### 한 줄 정의

- 원천: Project
- 초기 구현: 필수

### 현재 상태

- 원천: Project 상태
- 초기 구현: 필수 후보

### 현재 필요한 것

- 원천: Project 또는 Feedback Request
- 초기 구현: 필수
- 예시: 피드백 필요, 테스터 모집, 협업자 필요

### 최근 전환

- 원천: 공개된 핵심 전환 Change Card
- 초기 구현: 후보
- 주의: 별도 수동 문구로 중복 저장하지 않는다.

### 최근 활동

- 원천: Change Card 승인, 공개 업데이트, Feedback Request 등 내부 활동 신호
- 초기 구현: 후보

### 배지

- 원천: Project 상태, Feedback Request, Activity Signal 후보
- 초기 구현: 후보

### Builder

- 원천: Builder Profile 공개 정보
- 초기 구현: 필수 후보

## 4. 배지 후보

- 테스터 모집
- 피드백 요청
- 첫 공개
- 최근 전환
- 베타
- 협업자 필요

배지는 평가 점수가 아니라 맥락 안내다.

## 5. 정렬 기준 후보

1차 정렬 기준 후보:

- 최근 업데이트
- 피드백 요청 중
- 테스터 모집 중
- 최근 전환 있음

기본 정렬은 최근 업데이트가 적합하다. 피드백 요청 중, 테스터 모집 중은 필터 또는 배지로 처리하는 것이 무난하다.

## 6. 내부 활동 신호 후보

Activity Signal 후보:

- Change Card 생성 또는 승인
- 공개 업데이트
- Feedback Request 생성
- Feedback 수
- 테스터 참여 의사
- Project Save 또는 Follow
- 공개 Change Card 증가

초기에는 외부 트래픽 데이터를 사용하지 않는다.

## 7. 히트맵은 후순위 실험

히트맵은 프로젝트 타일 크기와 색상으로 활동성을 보여줄 수 있다. 그러나 산식이 불안정하면 인기순위처럼 오해될 수 있다.

따라서 1차에서는 히트맵을 만들지 않고 카드 그리드로 시작한다.

히트맵 확장 시 검토할 수 있는 요소:

- 타일 크기: 최근 공개 변화 카드 수, 피드백 요청 수, 저장 수 후보
- 타일 색상: 최근 전환, 테스터 모집, 피드백 요청 등 활동 상태 후보
- 배지: 베타, 첫 공개, 협업자 필요 등

히트맵 산식은 이번 단계에서 확정하지 않는다.

## 8. 초기 구현 필수 정보

- 공개 Project 목록
- 프로젝트명
- 한 줄 정의
- 현재 상태
- 현재 필요한 것
- Builder 공개 정보
- 최근 업데이트 후보
- 피드백 요청/테스터 모집 배지 후보

## 9. 후순위 정보

- 고급 필터
- 히트맵
- Activity Signal 집계
- 추천 알고리즘
- 개인화 탐색
- 저장 목록 기반 추천
- 외부 트래픽 기반 지표

## 10. 데이터 모델상 주의점

Discovery 모델이 인기순위나 채용 후보 랭킹처럼 보이면 BuildMap의 철학이 흔들린다. 탐색 데이터는 프로젝트의 판단 흐름과 현재 필요한 도움을 발견하게 하는 데 집중해야 한다.
