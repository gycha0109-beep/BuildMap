# Change Card 모델

## 1. 정의

Change Card는 프로젝트에서 발생한 하나의 의사결정 단위다. Change Card는 단순 로그가 아니다. BuildMap에서 프로젝트가 왜 현재 모습이 되었는지를 설명하는 핵심 원천 기록이다.

Decision Timeline, Public Project Page, Decision Diff, 인수인계 흐름은 모두 승인된 Change Card를 기반으로 한다.

## 2. 왜 Change Card가 중심 객체인가

BuildMap이 GitHub, Notion, 블로그, Product Hunt, LinkedIn과 달라지는 지점은 Change Card가 실제 판단 단위로 작동하는지에 달려 있다.

- GitHub는 코드 변경을 남긴다.
- Notion은 문서를 남긴다.
- 블로그는 정리된 회고를 남긴다.
- Product Hunt는 출시 결과를 보여준다.
- LinkedIn은 사람의 이력을 보여준다.
- BuildMap은 프로젝트가 어떤 판단을 거쳐 변화했는지를 Change Card로 남긴다.

## 3. Change Card 유형 후보

유형은 제품 설계용 후보이며 DB enum으로 확정하지 않는다.

| 유형 | 의미 |
|---|---|
| 문제 발견 | 새로운 문제를 발견한 기록 |
| 문제 정의 수정 | 기존 문제 정의가 바뀐 기록 |
| 가설 생성 | 새 가설을 세운 기록 |
| 가설 반박 | 기존 가설이 약해지거나 틀렸다고 본 기록 |
| 실험 | 특정 판단을 확인하기 위한 실험 기록 |
| 사용자 피드백 | 사용자 반응을 판단 근거로 남긴 기록 |
| 기능 추가 | 기능을 추가한 판단 기록 |
| 기능 제거 | 기능을 제거하거나 축소한 판단 기록 |
| 판단 유지 | 제안을 받았지만 기존 판단을 유지한 기록 |
| 판단 수정 | 기존 판단을 일부 수정한 기록 |
| 방향 전환 | 타깃, 문제, 해결 방식이 크게 바뀐 기록 |
| 릴리즈 | 배포 또는 공개의 목적과 판단 기록 |
| 인수인계 메모 | 새 팀원/협업자가 알아야 할 판단 맥락 |

## 4. 제품상 최소 정보

1차 구현에서는 Builder의 입력 부담을 줄이기 위해 필수 정보를 최소화한다.

### 유형

- 왜 필요한가: 카드가 어떤 종류의 판단인지 구분한다.
- Builder 직접 입력: 가능.
- AI 초안 제안: 가능.
- Builder 승인 필요: 필요.
- 공개 페이지 노출: 가능.
- 초기 구현 필수: 필수.
- 후순위: 유형 자동 추천 고도화.

### 제목

- 왜 필요한가: Timeline에서 한눈에 판단을 파악하게 한다.
- Builder 직접 입력: 가능.
- AI 초안 제안: 가능.
- Builder 승인 필요: 필요.
- 공개 페이지 노출: 공개 카드인 경우 가능.
- 초기 구현 필수: 필수.
- 후순위: 제목 템플릿 추천.

### 원문 메모

- 왜 필요한가: 구조화 결과의 근거가 되는 Builder의 원래 기록이다.
- Builder 직접 입력: 예.
- AI 초안 제안: 아니오. AI는 원문을 생성하면 안 된다.
- Builder 승인 필요: 원문 자체는 내부 기록으로 보존.
- 공개 페이지 노출: 기본 비공개.
- 초기 구현 필수: AI 경로에서는 필수.
- 후순위: 음성/파일 원문.

### 구조화 요약

- 왜 필요한가: 비개발자도 변화의 핵심을 빠르게 이해하게 한다.
- Builder 직접 입력: 가능.
- AI 초안 제안: 가능.
- Builder 승인 필요: 필요.
- 공개 페이지 노출: 공개 카드인 경우 가능.
- 초기 구현 필수: 필수.
- 후순위: 요약 길이별 버전.

### 근거

- 왜 필요한가: 판단이 감이 아니라 어떤 관찰, 피드백, 실험에서 나왔는지 보여준다.
- Builder 직접 입력: 가능.
- AI 초안 제안: 가능.
- Builder 승인 필요: 필요.
- 공개 페이지 노출: 공개 카드인 경우 가능.
- 초기 구현 필수: 필수.
- 후순위: 근거별 출처 링크.

### 판단

- 왜 필요한가: Change Card의 핵심이다. 무엇을 하기로 했는지 또는 하지 않기로 했는지 남긴다.
- Builder 직접 입력: 가능.
- AI 초안 제안: 가능.
- Builder 승인 필요: 필요.
- 공개 페이지 노출: 공개 카드인 경우 가능.
- 초기 구현 필수: 필수.
- 후순위: 판단 품질 체크.

### 다음 확인 사항

- 왜 필요한가: Change Card가 회고로 끝나지 않고 다음 실험이나 검증으로 이어지게 한다.
- Builder 직접 입력: 가능.
- AI 초안 제안: 가능.
- Builder 승인 필요: 필요.
- 공개 페이지 노출: 선택.
- 초기 구현 필수: 필수.
- 후순위: 작업 관리 연동.

### 공개 여부

- 왜 필요한가: 내부 기록 후 공개 전환 원칙을 지키기 위해 필요하다.
- Builder 직접 입력: 예.
- AI 초안 제안: 공개 주의 후보 정도 가능.
- Builder 승인 필요: 필요.
- 공개 페이지 노출: 공개됨 상태일 때만.
- 초기 구현 필수: 필수.
- 후순위: 세부 공개 권한.

### 상태

- 왜 필요한가: 초안, 승인, 보류, 공개 상태를 구분한다.
- Builder 직접 입력: 일부 가능.
- AI 초안 제안: 아니오.
- Builder 승인 필요: 필요.
- 공개 페이지 노출: 공개된 카드만.
- 초기 구현 필수: 필수.
- 후순위: 팀 승인 상태.

## 5. 강력 권장 정보

강력 권장 정보는 1차에서 모두 필수로 막지 않는다. Builder 기록 부담을 줄이기 위해 선택 입력 또는 AI 추천으로 둔다.

### 변경 내용

- 왜 필요한가: 판단 결과로 실제 무엇이 바뀌었는지 설명한다.
- 초기 구현: 강력 권장.
- 공개 노출: 공개 카드인 경우 가능.

### 연결된 문제 정의

- 왜 필요한가: 문제가 어떻게 바뀌었는지 추적한다.
- 초기 구현: 선택 연결, 승인 시 유도.
- 공개 노출: 연결된 문제가 공개 가능할 때.

### 연결된 가설

- 왜 필요한가: 어떤 가설이 확인, 반박, 보류되었는지 추적한다.
- 초기 구현: 선택 연결, 가설 관련 유형에서는 강력 권장.
- 공개 노출: 선택.

### 연결된 피드백

- 왜 필요한가: 외부 반응이 판단으로 이어졌는지 보여준다.
- 초기 구현: 선택.
- 공개 노출: 피드백 공개 정책에 따라 제한.

### 중요도

- 왜 필요한가: 핵심 전환과 일반 기록을 구분한다.
- 초기 구현: 강력 권장.
- 후보: 일반, 핵심 전환.
- 공개 노출: 핵심 전환 강조에 활용.

### 관련 링크

- 왜 필요한가: 실험 결과, 데모, 설문, 문서 등을 연결한다.
- 초기 구현: 선택.
- 후순위: 링크 미리보기.

### 관련 프로젝트 상태 변화

- 왜 필요한가: Project 상태가 아이디어에서 제작 중, 테스트 중, 베타 등으로 바뀐 이유를 연결한다.
- 초기 구현: 선택.
- 후순위: 상태 변경 자동 기록.

## 6. 상태 후보

| 상태 | 의미 |
|---|---|
| 초안 | 아직 공식 기록이 아닌 상태 |
| 수정 중 | Builder가 내용을 수정 중인 상태 |
| 승인됨 | 공식 Change Card로 확정된 상태 |
| 보류됨 | 공식 기록으로 반영하지 않고 보류한 상태 |
| 공개됨 | 공개 페이지에 노출 중인 상태 |
| 비공개 유지 | 승인되었지만 내부 기록으로만 남기는 상태 |

주의: `승인됨`과 `공개됨`은 다르다. 승인된 카드는 내부 Timeline에 반영될 수 있지만, 공개 페이지에는 노출되지 않을 수 있다.

## 7. 중요도 후보

| 중요도 | 의미 |
|---|---|
| 일반 | 일반적인 판단 기록 |
| 핵심 전환 | 프로젝트 방향, 문제 정의, 가설, 핵심 기능에 큰 영향을 준 기록 |

핵심 전환은 프로젝트 진화도와 공개 페이지 상단의 최근 전환 영역에 노출될 수 있다.

## 8. 공개 상태 후보

| 공개 상태 | 의미 |
|---|---|
| 내부 전용 | Builder 내부에서만 볼 수 있음 |
| 공개 가능 | 공개 후보지만 아직 공개되지 않음 |
| 공개됨 | Public Project Page에 노출됨 |
| 민감 정보 포함 | 공개 전 주의가 필요한 상태 |

공개 상태는 데이터 보호와 BuildMap의 신뢰를 위해 중요하다. AI 초안은 자동 공개되지 않는다.

## 9. 생성 경로

Change Card는 두 경로로 생성될 수 있다.

### AI 구조화 경로

```text
Rough Note 입력
→ AI Structured Draft 생성
→ Builder 수정
→ Builder 승인
→ Change Card 생성
→ Decision Timeline 반영
```

### 직접 작성 경로

```text
Builder가 Change Card 직접 작성
→ 필수 정보 입력
→ 승인
→ Decision Timeline 반영
```

직접 작성 경로를 막으면 AI 사용이 어려운 상황에서 기록이 끊길 수 있다.

## 10. 읽히는 화면

Change Card는 다음 화면에서 읽힌다.

- Builder 대시보드
- Change Card Review 화면
- Decision Timeline
- Public Project Page
- Project Card Grid의 최근 전환 요약
- Feedback Request 연결 화면
- Decision Diff
- 인수인계 흐름

## 11. 1차 구현 권장 입력 부담

1차에서 Builder에게 직접 요구할 최소 입력은 다음으로 제한하는 것이 적합하다.

- 유형
- 제목
- 원문 메모 또는 요약
- 근거
- 판단
- 다음 확인 사항
- 공개 여부

AI 구조화가 가능한 경우 구조화 요약, 변경 내용, 연결 후보, 중요도는 AI가 제안하고 Builder가 수정하는 방식이 적합하다.

## 12. 데이터 모델상 위험

Change Card가 너무 복잡하면 Builder가 기록하지 않는다. 반대로 너무 단순하면 Decision Timeline이 일반 활동 로그가 된다.

권장 대응:

- 필수 입력은 최소화한다.
- AI가 구조화 초안을 제안한다.
- 연결 정보는 강제보다 유도한다.
- 공개 상태는 명확히 분리한다.
- 승인 전/후 상태를 분명히 한다.
- 핵심 전환만 강조하여 Timeline이 길어져도 읽히게 한다.
