# Problem Definition과 Hypothesis 모델

## 1. 기본 원칙

Problem Definition과 Hypothesis는 BuildMap에서 프로젝트의 판단 중심축이다. 그러나 처음부터 복잡한 전략 문서나 실험 관리 도구처럼 만들지 않는다.

- Problem Definition은 프로젝트가 붙잡고 있는 문제다.
- Hypothesis는 프로젝트가 맞다고 믿고 검증하는 판단이다.
- Change Card는 문제와 가설이 어떻게 바뀌었는지를 기록한다.

연결은 느슨해야 한다. 변화 카드는 단독 생성 가능해야 하며, 승인 시 문제 또는 가설과 연결하도록 유도하는 것이 적합하다.

## 2. Problem Definition

### 정의

Problem Definition은 프로젝트가 해결하려는 문제의 현재 정의다. 문제 정의는 한 번 쓰고 끝나는 문장이 아니다. 사용자 피드백, 실험, 방향 전환에 따라 수정될 수 있다.

### 프로젝트에서의 역할

Problem Definition은 Project의 방향을 잡는 기준이다. 공개 프로젝트 페이지에서는 제품 소개보다 문제 정의가 먼저 이해되어야 한다.

### 현재 문제 정의

현재 문제 정의는 Public Project Page, Decision Timeline 상단 요약, Builder 대시보드에서 사용된다.

제품상 포함 정보 후보:

- 현재 문제 문장
- 문제를 겪는 대상
- 문제를 중요하게 보는 이유
- 마지막 수정 시점
- 관련 Change Card 후보
- 공개 여부

초기 구현에서는 긴 설명보다 짧은 문제 문장을 우선한다.

### 과거 문제 정의 이력

문제 정의가 바뀌면 이전 정의가 사라지면 안 된다. 다만 과거 이력을 별도 복잡한 버전 관리 도구처럼 만들지는 않는다.

권장 방향:

- 문제 정의 수정은 Change Card로 기록한다.
- Change Card 안에 기존 문제, 새 문제, 변경 이유를 남긴다.
- 현재 문제 정의 화면에서는 최근 수정 이력을 간단히 보여준다.

### 문제 정의 수정과 Change Card 연결

문제 정의가 바뀌는 순간은 중요한 Change Card 후보다.

예시:

- 기존 문제: 사용자는 장기 목표를 세우지 못한다.
- 새 문제: 사용자는 장기 목표와 이번 주 실행 단위를 연결하지 못한다.
- 변경 이유: 인터뷰에서 장기 목표보다 이번 주 실행 단위의 어려움이 반복되었다.

이 정보는 Change Card의 `문제 정의 수정` 유형으로 남길 수 있다.

### 공개 가능 여부

Problem Definition은 공개 프로젝트 페이지에서 핵심 정보가 된다. 그러나 민감한 시장, 고객군, 내부 전략이 포함될 수 있으므로 공개 여부를 제어할 수 있어야 한다.

### 초기 구현 필수 정보

- 현재 문제 정의 문장
- Project와의 연결
- 공개 여부 후보
- 마지막 수정 시점 후보

### 후순위 정보

- 문제 정의 이력 목록
- 문제 대상 세그먼트
- 관련 인터뷰/피드백 묶음
- Decision Diff 자동 비교
- 문제 정의 품질 진단

## 3. Hypothesis

### 정의

Hypothesis는 프로젝트가 맞다고 믿고 검증하는 판단이다. 기능 목록과 다르다.

기능 목록 예시:

- 캘린더 연동을 만든다.

가설 예시:

- 사용자는 이번 주에 증명할 행동 단위가 보이면 장기 목표를 더 쉽게 실행한다.

### 기능 목록과 다른 점

Hypothesis는 “무엇을 만들 것인가”가 아니라 “무엇이 사실이어야 프로젝트가 성립하는가”에 가깝다. 기능은 가설을 검증하기 위한 수단일 수 있지만, 가설 자체는 아니다.

### 가설 문장

제품상 포함 정보 후보:

- 가설 문장
- 가설 상태
- 관련 Project
- 연결된 Change Card
- 연결된 Feedback
- 공개 여부
- 마지막 업데이트 시점

초기 구현에서는 가설 문장과 상태만으로도 충분하다.

### 가설 상태 후보

상태값은 제품 설계용 후보이며 DB enum으로 확정하지 않는다.

| 상태 | 의미 |
|---|---|
| 가정 | 아직 검증 전이며 현재 믿고 있는 판단 |
| 검증 중 | 실험, 테스트, 피드백 수집 중인 상태 |
| 일부 확인 | 일부 근거는 있으나 완전히 확정하지 않은 상태 |
| 확인됨 | 충분한 근거로 현재는 맞다고 보는 상태 |
| 반박됨 | 피드백이나 실험으로 약해지거나 틀렸다고 본 상태 |
| 보류 | 지금 당장 검증하거나 판단하지 않는 상태 |

### 연결된 Change Card

가설 생성, 가설 반박, 판단 수정, 기능 제거, 방향 전환은 Hypothesis와 연결될 수 있다.

다만 모든 Change Card에 가설 연결을 강제하면 기록 부담이 커진다. 승인 시 연결 후보를 제안하되 필수로 막지 않는 방향이 적합하다.

### 연결된 Feedback

Feedback은 가설의 근거 또는 반례가 될 수 있다. 예를 들어 “첫 화면이 복잡하다”는 피드백이 “장기 목표를 먼저 보여주면 동기부여가 된다”는 가설을 약화시킬 수 있다.

1차에서는 Feedback을 직접 Hypothesis에 연결하기보다, Builder가 Feedback을 반영한 Change Card를 만들고 그 카드가 Hypothesis와 연결되는 흐름이 더 단순하다.

### 공개 가능 여부

Hypothesis는 공개 가능하지만, 내부 전략이나 경쟁상 민감한 내용은 비공개일 수 있다. Public Project Page에는 Builder가 공개한 현재 가설만 보여준다.

### 초기 구현 필수 정보

- 가설 문장
- 가설 상태
- Project와의 연결
- 공개 여부 후보

### 후순위 정보

- 관련 실험 목록
- 관련 피드백 묶음
- 상태 변경 이력
- 자동 가설 추출
- 가설별 성공 기준

## 4. 데이터 모델상 주의점

Problem Definition과 Hypothesis를 너무 엄격하게 설계하면 BuildMap이 복잡한 PM 도구처럼 보일 수 있다. 반대로 너무 느슨하면 Change Card가 왜 만들어졌는지 설명하기 어렵다.

권장 방향은 다음이다.

- Project 생성 시 짧은 문제 정의를 받는다.
- 가설은 짧은 문장으로 시작한다.
- 상태 관리는 단순하게 둔다.
- 문제·가설 변경은 Change Card로 남기도록 유도한다.
- 연결은 강제보다 추천에 가깝게 둔다.
