# 제품 데이터 모델 위험과 대응

## 1. 변화 카드가 너무 복잡해져서 Builder가 기록하지 않는 위험

### 위험 설명

Change Card에 너무 많은 필수 항목을 요구하면 Builder가 기록을 시작하지 못한다.

### 왜 문제가 되는가

BuildMap의 핵심은 판단 흐름을 계속 쌓는 것이다. 기록이 끊기면 Decision Timeline도 형성되지 않는다.

### 예방 원칙

- 필수 입력은 최소화한다.
- AI가 구조화 초안을 제안한다.
- 연결 정보는 강제하지 않고 유도한다.

### 1차 구현 대응

필수 정보는 유형, 제목, 원문 메모/요약, 근거, 판단, 다음 확인 사항, 공개 여부 정도로 제한한다.

### 후순위 검토

고급 연결, 관련 링크, 중요도 자동 추천, 세부 근거 출처는 후순위로 둔다.

## 2. 변화 카드가 너무 단순해져서 Decision Timeline이 의미 없어지는 위험

### 위험 설명

Change Card가 제목과 날짜만 남는 활동 로그가 되면 BuildMap의 차별점이 사라진다.

### 왜 문제가 되는가

Decision Timeline은 “무엇을 했다”가 아니라 “왜 그렇게 판단했다”를 보여줘야 한다.

### 예방 원칙

근거와 판단을 최소 구조로 유지한다.

### 1차 구현 대응

Change Card에 근거, 판단, 다음 확인 사항을 포함한다.

### 후순위 검토

가설 연결, 피드백 연결, Decision Diff와의 연결을 강화한다.

## 3. Decision Timeline과 Change Card가 중복 원천 기록이 되는 위험

### 위험 설명

Timeline을 별도 게시글처럼 만들면 Change Card와 같은 내용이 중복 저장된다.

### 왜 문제가 되는가

수정 불일치, 공개 상태 혼선, 원천 기록 불명확 문제가 생긴다.

### 예방 원칙

Change Card를 원천 기록으로 고정한다. Timeline은 표현이다.

### 1차 구현 대응

Timeline에는 승인된 Change Card만 표시한다.

### 후순위 검토

성능상 요약 캐시가 필요할 경우 원천과 캐시의 관계를 명확히 한다.

## 4. 문제 정의·가설·변화 카드 연결을 너무 강제해 입력 부담이 커지는 위험

### 위험 설명

모든 Change Card에 문제 또는 가설 연결을 필수로 하면 기록 흐름이 막힌다.

### 왜 문제가 되는가

초기 프로젝트는 문제와 가설이 아직 불분명할 수 있다.

### 예방 원칙

단독 Change Card 생성을 허용하고, 승인 시 연결을 유도한다.

### 1차 구현 대응

“연결할 문제 또는 가설이 있나요?” 정도의 유도 문구를 둔다.

### 후순위 검토

AI가 연결 후보를 추천할 수 있다.

## 5. 공개/비공개 상태가 복잡해져 민감 정보가 노출되는 위험

### 위험 설명

Project 공개와 Change Card 공개를 혼동하면 내부 기록이 공개될 수 있다.

### 왜 문제가 되는가

Builder 신뢰를 잃고, 내부 대화나 사용자 정보가 노출될 수 있다.

### 예방 원칙

기본값은 내부 비공개다. 공개는 명시적 행동이어야 한다.

### 1차 구현 대응

Project 공개 상태와 Change Card 공개 상태를 분리한다. AI Draft와 Rough Note는 자동 공개하지 않는다.

### 후순위 검토

민감 정보 자동 탐지, 공개 전 체크리스트 고도화.

## 6. Scout 데이터가 채용 플랫폼처럼 흘러가는 위험

### 위험 설명

Scout 모델이 채용 담당자, 지원자, 평가 점수 중심으로 확장되면 BuildMap이 채용 플랫폼처럼 보인다.

### 왜 문제가 되는가

BuildMap의 초기 가치는 프로젝트 발견, 피드백, 테스터, 협업 연결이다.

### 예방 원칙

Scout가 보는 정보는 점수가 아니라 프로젝트 기록에서 드러나는 근거다.

### 1차 구현 대응

Scout Profile은 선택으로 두고, 탐색·피드백·저장 중심으로 제한한다.

### 후순위 검토

채용/헤드헌팅은 장기 확장 문서에서 별도 검토한다.

## 7. AI 구조화가 과장된 포트폴리오 문장을 만드는 위험

### 위험 설명

AI가 Builder의 거친 기록을 멋진 성과 문장으로 꾸미면 BuildMap의 신뢰가 깨진다.

### 왜 문제가 되는가

BuildMap은 결과 포장이 아니라 판단 근거를 남기는 플랫폼이다.

### 예방 원칙

AI는 생성보다 구조화에 집중한다.

### 1차 구현 대응

AI Draft에는 원문, 근거, 판단, 다음 확인 사항을 유지하고 Builder가 승인하게 한다.

### 후순위 검토

AI 출력 검증 체크리스트, 원문 대비 변경 표시.

## 8. 피드백이 일반 댓글처럼 흘러가는 위험

### 위험 설명

피드백이 자유 댓글이 되면 판단 흐름과 연결되지 않는다.

### 왜 문제가 되는가

Builder가 피드백을 다음 Change Card로 전환하기 어렵다.

### 예방 원칙

Feedback Request를 특정 질문에 연결한다.

### 1차 구현 대응

Feedback은 로그인 기반으로, Feedback Request에 연결한다.

### 후순위 검토

공개 댓글, 스레드, 비로그인 피드백은 신중히 검토한다.

## 9. 히트맵이 인기순위로 오해되는 위험

### 위험 설명

히트맵 타일 크기와 색상이 인기 지표처럼 보일 수 있다.

### 왜 문제가 되는가

BuildMap의 탐색 목적은 살아 움직이는 프로젝트 발견이지 인기 경쟁이 아니다.

### 예방 원칙

1차 탐색은 카드 그리드로 시작한다.

### 1차 구현 대응

최근 업데이트, 피드백 요청, 테스터 모집 같은 맥락 신호를 우선 사용한다.

### 후순위 검토

히트맵 산식과 시각 표현은 별도 실험으로 검토한다.

## 10. 데이터 모델이 Notion/블로그/GitHub 복제처럼 보이는 위험

### 위험 설명

Project가 문서, Change Card가 게시글, Timeline이 활동 로그처럼 변하면 기존 도구의 조합처럼 보인다.

### 왜 문제가 되는가

BuildMap의 독립적 위치가 사라진다.

### 예방 원칙

모든 데이터 객체는 판단 흐름을 중심으로 정의한다.

### 1차 구현 대응

Change Card를 원천 기록으로 두고, Project는 판단 흐름 컨테이너, Timeline은 표현 구조, Public Page는 공개 뷰로 정리한다.

### 후순위 검토

사용자 리서치에서 “그냥 Notion 아닌가?”라는 반응이 나오는지 확인한다.
